# Node.js 测试入门：一个最小 spec 文件长什么样

## 这篇笔记想回答什么

对于新手来说，第一次看到 spec/test 文件时，最容易困惑的不是某个命令，而是：

- 它为什么长这样？
- describe、it、expect 到底分别在干什么？
- 为什么测试代码看起来像一句一句英语？

所以这篇笔记不先讲命令，而是先建立一个直觉：

一个最小 spec 文件，本质上是在用结构化语言写出：
“某个对象，在某个场景下，应该产生某个结果。”

## 一句话先记住

一个最小 spec 文件，通常至少要回答三件事：

1. 你在描述谁？
2. 你期待它怎么表现？
3. 你如何验证这个期待是真的？

把这三件事翻译成测试代码里的常见结构，就是：

1. describe：我们现在在讨论哪一块功能
2. it：这一块功能应该有什么具体表现
3. expect：我们怎么检查这个表现是否真的成立

## 最小骨架长什么样

下面这段代码，就是一个典型的最小 spec 文件骨架：

```js
describe('add', () => {
  it('should return 3 when adding 1 and 2', () => {
    expect(add(1, 2)).toBe(3)
  })
})
```

先不要急着把它当成某个框架的语法去背。
更重要的是把它读成人话：

“对于 add 这个东西来说，
当我给它 1 和 2，
它应该返回 3。”

也就是说，这段代码不是在实现功能，
而是在描述并验证一个行为承诺。

## 1. describe 在表达什么

```js
describe('add', () => {
  ...
})
```

可以把 describe 理解成：
“给一组相关测试起一个主题”。

它像一本书里的章节标题，回答的问题是：
“我们现在在讨论谁？”

例如：
- describe('add')
- describe('User login')
- describe('formatPrice')

它的重点不是执行逻辑，而是提供语境和组织结构。

所以 describe 的作用更像：
- 一个讨论范围
- 一个章节标题
- 一个行为主题

## 2. it 在表达什么

```js
it('should return 3 when adding 1 and 2', () => {
  ...
})
```

可以把 it 理解成：
“在某个场景下，它应该怎么表现”。

把它读成人话就是：
“It should return 3 when adding 1 and 2.”

这里最重要的不是英语本身，而是这种句子结构在表达：

不是“我来试试看”，
而是“它应该如此表现”。

所以 it 的本质是：
- 描述一个具体行为场景
- 写出一个明确的预期
- 把功能拆成一条一条可阅读的说明

一个功能通常不会只有一个 it，因为真实功能往往有多个行为场景。

例如：

```js
describe('add', () => {
  it('should return 3 when adding 1 and 2', () => {
    expect(add(1, 2)).toBe(3)
  })

  it('should return 0 when adding -1 and 1', () => {
    expect(add(-1, 1)).toBe(0)
  })
})
```

这就像在说：
“关于 add，我们有两条行为说明。”

## 3. expect 在表达什么

```js
expect(add(1, 2)).toBe(3)
```

expect 是测试里最关键的验证动作。

它的意思是：
“我期待某个结果成立。”

也就是：
- 如果 add(1, 2) 真的是 3，测试通过
- 如果不是 3，测试失败

所以 expect 回答的问题是：
“你怎么证明刚才那条行为说明是真的？”

如果没有 expect，前面的 describe 和 it 就只是在说，
还没有真正进入“验证”。

## 4. 为什么这三层结构这么自然

因为它刚好对应人的思维顺序：

1. 我们在讨论谁？
2. 它在某个场景下应该怎么做？
3. 我怎么确认它真的这么做了？

所以可以把它们记成：

- describe：上下文
- it：行为场景
- expect：验证标准

这也是为什么很多测试代码读起来像自然语言。
它不是偶然设计成这样的，而是为了让测试既能执行，也能阅读。

## 5. 再看一个更小的例子

假设有一个函数：

```js
function isEven(n) {
  return n % 2 === 0
}
```

那一个最小 spec 可以是：

```js
describe('isEven', () => {
  it('should return true for 2', () => {
    expect(isEven(2)).toBe(true)
  })
})
```

翻译成人话就是：

“对于 isEven 这个函数，
当输入 2 时，
它应该返回 true。”

这已经是一个完整的最小 spec 了。

## 6. 它和普通实现代码有什么不同

普通实现代码在做的是：
“让功能发生”。

例如：

```js
function add(a, b) {
  return a + b
}
```

而 spec 文件在做的是：
“说明这个功能应该如何发生，并验证它是否真的这样发生。”

例如：

```js
describe('add', () => {
  it('should return 3 when adding 1 and 2', () => {
    expect(add(1, 2)).toBe(3)
  })
})
```

所以可以把两者区分成：

- 实现代码：生产行为
- spec 文件：描述并验证行为

这个分界很重要。
因为很多新手看到测试代码时，会误以为它也在“实现某种功能”。
实际上它主要在表达和校验承诺。

## 7. 为什么新手会觉得 spec 文件奇怪

因为初学者通常先习惯的是：
“写能跑的代码”。

而 spec 文件属于另一种代码：
“写描述代码行为的代码”。

所以第一次看到：

```js
describe(...)
it(...)
expect(...)
```

很容易觉得：
- 为什么要写得像英语？
- 为什么不直接写 if 判断？
- 为什么看起来不像普通函数？

原因是测试框架在提供一种统一的“行为描述语言”。

这种语言的价值在于：
- 更接近自然语言
- 测试失败时报告更清晰
- 方便组织大量测试
- 更适合团队阅读和协作

所以它不是把事情变复杂，
而是在给“行为说明”提供统一语法。

## 8. 从思想上看，最小 spec 文件真正不可缺少的是什么

如果不从具体框架语法看，而只看思想，
一个最小 spec 文件不可缺少的核心只有三样：

1. 被测试对象
   - 比如 add、isEven、login

2. 一个具体场景
   - 比如输入 1 和 2、输入错误密码、空数组输入

3. 一个明确预期
   - 比如返回 3、抛出错误、得到空数组

所以你以后看任何测试文件，都可以先不怕语法，
只先问三件事：

- 它在测谁？
- 它测什么场景？
- 它期待什么结果？

只要抓住这三点，测试文件就不会显得神秘。

## 9. 一个适合记忆的人话模板

以后你看到 spec 文件，可以先在脑中翻译成这个模板：

“关于【某个功能】，
当处于【某个场景】时，
它应该【产生某个结果】。”

例如：

“关于 add，
当输入 1 和 2 时，
它应该返回 3。”

映射成代码就是：

```js
describe('add', () => {
  it('should return 3 when input is 1 and 2', () => {
    expect(add(1, 2)).toBe(3)
  })
})
```

## 10. 当前阶段最重要的收获

如果把这篇笔记压缩成最核心的认识，可以记成：

1. 一个最小 spec 文件不是在实现功能，而是在描述并验证行为
2. describe、it、expect 分别对应：上下文、行为场景、验证标准
3. 测试代码之所以像自然语言，是因为它既要给机器执行，也要给人阅读
4. 不必一开始死记所有 API，先抓住“测谁、测什么场景、期待什么结果”这三个核心

## 一句话总结

一个最小 spec 文件，就是把“某个对象在某个场景下应该产生某个结果”写成结构化、可执行、可阅读的行为说明。
