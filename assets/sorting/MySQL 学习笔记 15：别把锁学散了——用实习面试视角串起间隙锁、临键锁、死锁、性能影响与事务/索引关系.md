- 锁等待

## 二、先给你整条主线：锁到底在解决什么

你可以先把锁理解成一句话：

“当多个事务同时碰同一批数据时，数据库要想办法让真正冲突的操作有顺序，不要把结果弄乱。”

从这句话往下展开，很多概念就能串起来：

- 共享锁 / 排他锁：在区分‘别人还能做什么’
- 行锁 / 表锁：在区分‘影响范围多大’
- 间隙锁 / 临键锁：在补上‘不只是已有记录，范围里的变化也可能要防’
- 死锁：是多个事务彼此等待形成环
- 性能影响：是因为锁让一部分操作不能同时推进
- 和事务的关系：因为锁主要服务于并发事务的正确性
- 和索引的关系：因为 InnoDB 往往沿着索引访问路径去加锁

所以锁不是一个孤立知识点，
它其实是“事务并发控制”这条主线上非常核心的一环。

## 三、间隙锁：它不是在锁已有记录，而是在防‘这个空档里插进新东西’

这是很多人第一次看会懵的点。

你会想：

“锁不是应该锁记录吗？为什么还要锁间隙？”

因为数据库有时候要保护的，不只是“某一行别被改”，
还包括：

“在我这次事务处理期间，这个范围里的结果别突然多出一条新记录。”

你可以把间隙锁先理解成：

“不是锁住某条已经存在的记录，而是锁住索引范围中的一个空档，防止别人往这个空档插入新记录。”

### 最小体感例子

假设有索引值：

```text
10, 20, 30
```

现在事务 A 在处理：

```sql
select * from t where score > 10 and score < 20 for update;
```

虽然这个范围里现在可能一条记录都没有，
但数据库可能仍要防止：

- 事务 B 突然插入 score = 15

为什么？
因为如果不防，
事务 A 前后两次看这个范围，结果就可能变了。

所以你现阶段先把间隙锁记成：

“间隙锁主要是在防某个索引范围里突然插入新记录。”

这就够用了。

## 四、临键锁：你可以先把它理解成‘记录 + 周边间隙一起保护’

如果只讲间隙锁，很多人会问：

“那数据库到底是只防插入，还是也防当前已有记录被并发影响？”

这就会碰到临键锁。

你现阶段不用死背细定义，
先抓这个直觉：

“临键锁可以先理解成：既照顾当前记录，也顺手把相邻间隙一起保护起来。”

它出现的背景是：
- 数据库有时候不只是要保护‘这一条记录’
- 还要保护‘围绕这条记录的索引范围变化’

所以它很适合用来理解成一种更稳妥的范围保护。

### 实习面试够用的说法

- 间隙锁：更偏向防止范围内插入
- 临键锁：可以先理解为记录锁 + 间隙锁的组合式保护

你不必在当前阶段深抠每种边界细节，
只要知道它们出现的原因是：

“数据库要保护的有时不是单个点，而是一个查询或更新范围。”

## 五、死锁为什么会发生

死锁的本质其实不复杂。

你可以先记一个最小定义：

“事务 A 等事务 B，事务 B 又等事务 A，或者形成更长的等待环，大家都不肯先让，于是就卡死了。”

### 最小案例

事务 A：

```sql
begin;
update account set balance = balance - 100 where id = 1;
update account set balance = balance + 100 where id = 2;
```

事务 B：

```sql
begin;
update account set balance = balance - 50 where id = 2;
update account set balance = balance + 50 where id = 1;
```

如果执行顺序是：
- A 先锁住 id = 1
- B 先锁住 id = 2
- 然后 A 想拿 id = 2
- B 又想拿 id = 1

这时就形成互相等待。

这就是死锁最经典的模式。

### 你要建立的直觉

死锁不是说数据库“坏了”，
而是并发控制里非常典型的代价之一。

因为只要允许多事务并发、又允许它们按不同顺序拿资源，
就可能出现等待环。

所以面试里更好的表达不是：
- 死锁是 bug

而是：
- 死锁是并发事务争抢资源时可能出现的结果，数据库通常会检测并主动回滚其中一个事务来打破僵局

这个表达会成熟很多。

## 六、为什么锁会影响性能

这个问题特别常见，而且很适合项目表达。

你不要把“锁影响性能”理解成锁本身计算很慢。
更核心的点是：

“锁会让原本能并发推进的操作，变成必须等待。”

这就会带来几个直接后果：

### 1）等待时间变长

一个事务拿着锁不提交，
别的事务就可能一直等。

### 2）吞吐量下降

本来大家可以一起做，
现在一部分请求得排队，
单位时间内能完成的事务数就下降。

### 3）响应时间抖动更大

有些请求很快，
有些请求刚好撞上锁等待，就会突然变慢。

### 4）锁范围越大，影响越重

比如：
- 精准命中主键更新，影响相对小
- 范围更新，影响更大
- 没走好索引，锁范围变粗，影响更大

所以你可以记住：

“锁影响性能，本质上不是因为加锁这个动作本身有多贵，而是因为它限制了并发。”

## 七、锁和事务是什么关系

这部分你一定要串起来。

锁不是单独存在给单条 SQL 玩的，
它主要是在事务并发这个背景下才显得重要。

为什么？
因为：
- 如果系统里永远只有一个操作
- 或每次操作都完全串行执行

那锁的重要性就会大幅下降。

真正需要锁，是因为：
- 多个事务可能同时读写同一批数据
- 数据库既想要正确性，又不想所有操作都完全串行
- 所以只能让“真正冲突的部分”排队

于是你会发现：
- 事务提供的是一组操作的边界
- 锁提供的是并发下的顺序控制

一句话记：

“事务负责把业务动作包起来，锁负责让并发事务在关键资源上别打架。”

## 八、锁和索引是什么关系

这一部分你刚学完，
这里帮你收束成面试版表达。

你可以记住：

“InnoDB 的锁，尤其是行锁效果，通常和索引访问路径强相关。”

原因不是因为索引和锁是两个硬绑死的独立功能，
而是因为数据库得先找到记录，才能处理记录。

而“找到”的过程，往往依赖索引。
所以：
- 主键 / 唯一索引等值命中时，锁更容易收敛
- 范围查询时，锁影响更容易扩大
- 没有合适索引时，锁控制会变粗，冲突更明显

现阶段你就记这一层够了。

## 九、这些锁在实际开发里是怎么体现出来的，是不是 InnoDB 自动使用的

这个问题非常重要。
很多人学锁时会有一种困惑：

“我知道有共享锁、排他锁、间隙锁、临键锁这些名字，但业务代码里好像没有天天手写这些词，那它们到底怎么体现？”

你可以先抓一个最重要的结论：

“锁的存在感，通常不是体现在你写了某个锁名，而是体现在多个事务并发时，SQL 会不会等待、阻塞、死锁，以及影响范围有多大。”

也就是说，锁最常见的体现方式是：
- 事务 A 执行后，事务 B 为什么卡住了
- 为什么同样是 update，有的只影响一行，有的却把很多请求都拖慢了
- 为什么有的范围查询会把插入挡住
- 为什么两个事务会互相等待，最后触发死锁

### 1）哪些锁是开发者“间接使用”的

从业务开发视角看，你平时更常写的是这些 SQL：
- `select ...`
- `select ... for update`
- `select ... for share` / `lock in share mode`
- `update ...`
- `delete ...`
- `insert ...`

而共享锁、排他锁、间隙锁、临键锁，更多是数据库在这些 SQL 背后采用的并发控制手段。

所以更准确的说法不是：
- 开发者天天手动选“我要用哪种锁”

而是：
- 开发者通过 SQL 的读写意图、事务边界、where 条件、索引命中方式，间接决定锁会不会出现、锁的范围会不会扩大；
- InnoDB 再根据这些信息自动决定具体怎么加锁。

### 2）哪些情况更像是 InnoDB 自动加锁

你可以先这么理解：

#### 普通 `select`

很多时候主要是快照读，更多依赖 MVCC。
它不等于“自动加一个共享锁”。

也就是说，普通查询最常见的情况不是“我一读就把别人锁住”。

#### `select ... for update`

这是显式锁定读。
你在 SQL 层已经明确告诉数据库：

“我这次不是只看一眼，我后面大概率还要处理它，所以请把相关记录纳入更强的并发控制。”

这种场景下，InnoDB 会自动对命中的记录加更强的锁。

#### `select ... for share` / `lock in share mode`

这也是显式锁定读。
意思更接近：

“我要稳定地读这份数据，别人也许还能读，但别在我处理期间把它改了。”

所以这类语句更接近共享锁语义。

#### `update` / `delete`

这是最典型的自动加锁场景。
你不会手写“请给我一个排他锁”，
但 InnoDB 为了保证并发修改不冲突，会自动对相关记录加锁。

#### `insert`

`insert` 也会涉及锁，只是它的体感往往不是“锁住已有记录”，
而是和索引范围、唯一键检查、是否允许插入这个位置等问题相关。

这也是你后面理解间隙锁会用到的地方。

### 3）间隙锁、临键锁是不是我手动点名使用的

通常不是。

你平时不会在 SQL 里直接写：
- “给我来一个间隙锁”
- “这次我要临键锁”

更常见的真实情况是：
- 你写了范围查询 / 范围更新
- 你用了锁定读
- 当前隔离级别需要保护范围稳定性
- InnoDB 发现只锁已有记录还不够

于是它自动把“相邻间隙”也一起保护起来，防止别人往这个范围里插入新记录。

所以你可以把它理解成：

“间隙锁、临键锁更多是 InnoDB 为了保护范围稳定性而自动使用的机制，不是业务开发里天天手动点名选择的工具。”

## 十、用最小 SQL 场景把这些锁怎么体现直观看懂

下面我用最小例子，帮你把“普通读、锁定读、写、范围保护”区分开。

### 场景 1：普通 `select` —— 更像 MVCC 快照读

事务 A：

```sql
begin;
select * from account where id = 1;
```

这时候你不要先脑补成：
- “数据库已经自动给这一行加了共享锁”

更常见的理解是：
- 这是普通读
- 主要依赖 MVCC 处理可见性
- 不等于我显式把这一行锁住了

所以普通 `select` 的重点通常是“能看到哪个版本”，
而不是“我靠锁把别人挡住了”。

### 场景 2：`select ... for update` —— 我先锁住，后面准备改

事务 A：

```sql
begin;
select * from account where id = 1 for update;
```

事务 B：

```sql
begin;
update account set balance = balance + 100 where id = 1;
```

这时事务 B 往往需要等事务 A 提交或回滚。

你可以把它理解成：
- A 在说：这条记录我先控制住，后面要基于它继续处理
- B 想改同一条记录，就会被挡住

这就是“锁定读 + 排他语义”最容易感受到的场景。

### 场景 3：`select ... for share` —— 我要稳定读，但不希望你改

事务 A：

```sql
begin;
select * from product where id = 1 for share;
```

事务 B：

```sql
begin;
update product set price = 99 where id = 1;
```

这时事务 B 可能需要等待。

这能帮助你体会共享锁语义：
- A 不是要立刻修改
- 但 A 想稳定地读
- 所以不希望 B 在它处理期间把这行改掉

### 场景 4：`update` —— 排他锁往往是自动出现的

事务 A：

```sql
begin;
update account set balance = balance - 100 where id = 1;
```

事务 B：

```sql
begin;
update account set balance = balance + 50 where id = 1;
```

事务 B 很可能会等待。

这个场景很适合你建立一个直觉：

“写操作不需要你手动写‘加排他锁’，InnoDB 通常会自动把并发修改控制起来。”

### 场景 5：范围锁定读 —— 间隙锁开始有体感

假设索引值里当前有：

```text
10, 20, 30
```

事务 A：

```sql
begin;
select * from t where score > 10 and score < 20 for update;
```

事务 B：

```sql
begin;
insert into t(score) values(15);
```

如果事务 B 被卡住，你就可以体会到：
- A 不只是想保护“已有记录”
- A 还想保护“这个范围的结果别突然变化”
- 所以数据库可能会把这个间隙一起保护起来

这就是间隙锁最容易理解的最小场景。

### 场景 6：范围更新 —— 临键锁 / 范围保护更容易出现

事务 A：

```sql
begin;
update t set status = 'done' where score >= 10 and score < 30;
```

这种时候，数据库要保护的通常就不再只是某一个点，
而更像是一段索引范围。

所以你现阶段可以先形成这个感觉：
- 单点命中，更容易理解成“锁一行”
- 范围命中，更容易出现“锁范围扩大”
- 间隙锁、临键锁就是这种范围保护逻辑的一部分

## 十一、把这几个点串成一张脑图式理解

你可以把锁模块脑子里压成这样一条链：

1. 事务并发访问同一批数据，会产生冲突风险
2. 锁是为了解决这些冲突，让关键操作有顺序
3. 锁有语义区别：共享锁、排他锁
4. 锁有粒度区别：行锁、表锁
5. InnoDB 行锁通常和索引访问路径强相关
6. 当要保护一个范围而不只是一个点时，就会出现间隙锁、临键锁这类机制
7. 多事务拿锁顺序不一致时，可能发生死锁
8. 锁会影响性能，因为它会让一部分并发操作等待

这条链一旦顺了，锁这一模块就不会散。

## 十二、项目里怎么表达这部分知识

你在面试里不要散点背术语。
更好的说法是：

“MySQL 里的锁本质上是事务并发控制手段。共享锁、排他锁解决的是不同事务还能不能同时读写；行锁、表锁解决的是锁影响范围大小；InnoDB 的行锁通常又和索引访问路径强相关。对于范围查询或范围更新，数据库有时还会通过间隙锁、临键锁去保护索引范围，避免并发插入影响结果稳定性。多个事务拿锁顺序不一致时可能发生死锁，而锁会影响性能，本质上是因为它限制了并发。”

这段如果你能顺下来，已经很像“懂主线”的回答了。

## 十三、这一篇你先记住的最小骨架

如果你现在只想记最小骨架，就记这 7 句：

1. 锁的本质是并发下控制冲突访问顺序
2. 间隙锁主要是在防索引范围里插入新记录
3. 临键锁可以先理解成记录 + 相邻间隙的组合式保护
4. 死锁本质是事务之间形成等待环
5. 死锁不是 bug，而是并发控制可能付出的代价
6. 锁影响性能，本质是因为它让并发操作等待
7. 锁和事务、索引不是孤立知识点，而是一整套并发控制逻辑的一部分

## 十四、面试版简答模板

### 1）什么是间隙锁？

“间隙锁可以理解成对索引范围内空档的保护，它主要不是锁已有记录，而是防止其他事务在这个范围里插入新记录，从而影响当前事务看到的结果。”

### 2）什么是临键锁？

“临键锁可以先理解成记录锁和间隙锁的组合式保护。它不只是保护已有记录，也会顺带保护相邻索引间隙，常见于范围查询或范围更新场景。”

### 3）死锁为什么会发生？

“死锁通常是因为多个事务拿锁顺序不一致，彼此等待对方释放资源，最终形成等待环。数据库一般会检测死锁并回滚其中一个事务。”

### 4）为什么锁会影响性能？

“因为锁会让原本可以并发推进的操作发生等待，导致吞吐下降、响应时间变长或抖动更大。锁范围越大、持锁时间越长，性能影响通常越明显。”
