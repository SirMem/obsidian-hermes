## 一、这份规划的定位

这不是一份“系统学完 Redis 全家桶”的长期路线，而是一份面向面试冲刺的 Redis 复习规划。

你的当前状态是：
- 以前学过 Redis
- 对很多概念有印象
- 但容易遗忘、答不成结构
- 当前目标不是讲源码级底层，而是做到“能围绕简历和项目，把 Redis 讲顺”

所以这份规划的核心目标是：

“用尽量短的时间，把 Redis 重新整理成一套可口述、可用于项目表达、能覆盖高频面试问题的回答框架。”

这也和 [[3 天面试冲刺计划：目标是能回答简历 70% 的内容]] 的节奏一致：Redis 属于 A 级重点，但学习方式要围绕‘能说清楚’，而不是围绕‘背最深原理’。

## 二、先明确：这次 Redis 要学到什么程度

你这轮复习不需要达到：
- 能把 Redis 源码实现讲透
- 能把跳表、压缩列表、渐进式 rehash、对象编码细节全讲明白
- 能系统比较 Redis Cluster、Codis、代理分片、哨兵所有差异

你需要达到的是：
- 能讲清 Redis 在系统里是干什么的
- 能讲清 Redis 为什么快
- 能讲清常见数据结构适合什么场景
- 能讲清 Redis 和 MySQL 的分工
- 能讲清持久化为什么存在
- 能讲清缓存带来的典型问题与应对思路
- 能结合自己的项目，说出 Redis 在业务中解决了什么问题

一句话说，就是：

“基础原理讲主线，项目使用讲真实，遇到追问能继续展开 1~2 层。”

## 三、Redis 这轮复习的总策略

### 1. 先从“为什么需要 Redis”入手，不要一上来背八股
按你现在的学习目标，最重要的不是先记命令，而是先建立这条主线：

业务系统为什么除了 MySQL，还需要 Redis？

因为只靠 MySQL，很多场景会遇到：
- 读压力大
- 响应慢
- 临时状态不适合落库
- 高频访问数据没必要每次都查磁盘
- 某些计数、排行榜、会话状态更适合内存结构

所以 Redis 的本质定位你要先想清：
- 高性能缓存
- 临时状态存储
- 轻量级高频数据结构服务
- 某些分布式协作能力的载体（如分布式锁、计数、限流等）

### 2. 复习顺序按“面试表达链路”来，不按文档章节来
建议按下面顺序复习：
1. Redis 在系统中的角色
2. Redis 为什么快
3. 常见数据结构与典型业务场景
4. 持久化与数据可靠性
5. 缓存常见问题
6. 和项目绑定的表达
7. 高频追问模拟

这个顺序的好处是：你学完每一块，都能立刻回答一批面试题。

### 3. 只抓高频，不平均用力
这轮 Redis 学习要分层：

必须讲清：
- 为什么快
- String / Hash / List / Set / ZSet 的典型用途
- 缓存和数据库的分工
- 持久化存在的意义
- 缓存穿透、击穿、雪崩
- 项目里怎么用 Redis

知道大概即可：
- 发布订阅
- Stream
- Lua 脚本
- 集群分片
- 复杂淘汰策略实现细节
- 底层源码级对象编码

## 四、建议的学习模块

## 模块 1：先把 Redis 的“系统定位”讲顺

### 学习目标
你要能先回答一个最基础但最关键的问题：

“为什么项目里要用 Redis，而不是只用 MySQL？”

### 这一模块你要学会的内容
- Redis 是基于内存的 key-value 数据库
- 它主要解决的是高频读写性能问题和临时状态存储问题
- Redis 不完全替代 MySQL，而是和 MySQL 分工协作
- MySQL 更适合持久、结构化、强一致业务数据
- Redis 更适合缓存、会话、计数、热点数据、短期状态

### 你最终要能说出的标准回答
可以先练成下面这种口语化表达：

“Redis 在系统里一般不是拿来替代 MySQL 的，而是和 MySQL 配合使用。MySQL 负责存核心业务数据，Redis 更适合放热点数据、缓存结果和一些临时状态。这样可以减少数据库压力，也能提高接口响应速度。像用户信息缓存、验证码、排行榜、计数器这类场景，用 Redis 会更合适。”

### 这一模块的输出要求
你至少能独立回答：
1. Redis 是干什么的？
2. 为什么系统里既有 MySQL 又有 Redis？
3. Redis 更适合存什么数据？

## 模块 2：Redis 为什么快

### 学习目标
这一部分是高频中的高频，你要能先讲主线，再视情况补细节。

### 这一模块你要掌握的主线
Redis 快，核心可以从这几个方向讲：
- 基于内存，访问速度远快于磁盘
- 数据结构设计适合高频读写
- 单线程模型避免了很多线程切换和锁竞争开销（面试中这样回答通常够用）
- IO 多路复用提高网络处理效率

### 你需要注意的表达边界
这里不要把自己逼到源码级。

你当前阶段能讲到：
- 内存
- 单线程避免锁竞争
- IO 多路复用
- 数据结构高效

就已经足够覆盖大多数实习 / 初中级面试问题。

如果被追问“为什么单线程还快”，你可以继续说：
- Redis 的瓶颈很多时候不在 CPU，而在内存和网络
- 它把核心命令执行做得很轻量
- 单线程反而减少了并发控制成本

### 这一模块的输出要求
你至少能独立回答：
1. Redis 为什么快？
2. Redis 单线程为什么性能还高？
3. Redis 快和 MySQL 快的侧重点有什么不同？

## 模块 3：常见数据结构与业务场景

### 学习目标
这一部分的目标不是背 API，而是建立“结构 -> 特点 -> 场景”的映射。

### 这轮必须掌握的 5 个核心结构

#### 1. String
最常用。
适合：
- 缓存对象
- 计数器
- token
- 验证码
- 分布式锁中的简单 key

你要会说：
- Redis 的很多场景本质上都离不开 String
- 它可以直接缓存序列化后的对象或简单值

#### 2. Hash
适合：
- 存对象的多个字段
- 用户信息、配置项等结构化数据

你要会说：
- 相比把整个对象序列化成一个 String，Hash 更适合按字段读写

#### 3. List
适合：
- 消息队列的简单实现
- 按顺序存储的数据
- 最新记录列表

你要会说：
- List 本质上适合有顺序、支持头尾操作的场景
- 但真正复杂 MQ 还是通常交给 RabbitMQ / RocketMQ / Kafka

#### 4. Set
适合：
- 去重
- 共同关注 / 标签集合
- 某类对象是否存在

你要会说：
- Set 最大特点是元素唯一，适合做判重

#### 5. ZSet
适合：
- 排行榜
- 按分数排序
- 延迟队列的一些简单实现思路

你要会说：
- ZSet 在面试里最经典的业务场景就是排行榜

### 这一模块的输出要求
你至少能独立回答：
1. Redis 常见数据结构有哪些？
2. String / Hash / Set / ZSet 分别适合什么场景？
3. 如果做排行榜，为什么会想到 ZSet？

## 模块 4：Redis 和 MySQL 的分工

### 学习目标
这是面试里非常实用的一块，因为它天然能和项目表达结合。

### 你要掌握的核心认知
MySQL 和 Redis 不是谁替代谁，而是：
- MySQL 负责核心持久化业务数据
- Redis 负责高频访问和临时状态
- Redis 常常作为 MySQL 前面的加速层

### 你要会讲的典型分工
MySQL 适合：
- 订单
- 用户核心信息
- 交易记录
- 需要长期保存和事务保障的数据

Redis 适合：
- 热点数据缓存
- 用户会话
- 短期验证码
- 点赞数、浏览数等高频计数
- 排行榜
- 分布式锁辅助控制

### 这一模块的输出要求
你至少能独立回答：
1. Redis 和 MySQL 怎么分工？
2. 什么数据更适合放 Redis？
3. 为什么核心业务数据不能只放 Redis？

## 模块 5：持久化——不是为了替代数据库，而是为了减少数据丢失风险

### 学习目标
你需要知道 Redis 为什么也要持久化，但不用把它讲成和 MySQL 一样的核心数据存储。

### 你至少要掌握这条主线
Redis 虽然主要基于内存，但如果完全不持久化，服务重启后数据就会丢。
所以 Redis 提供持久化机制，用来：
- 尽量减少数据丢失
- 在重启后恢复数据
- 提升可用性

### 你至少要会区分
#### RDB
- 某个时间点的数据快照
- 恢复速度快
- 适合做整体备份思路
- 缺点是两次快照之间可能丢数据

#### AOF
- 记录写操作命令
- 数据完整性通常更好
- 恢复时通过重放命令来还原
- 一般比 RDB 更实时，但文件可能更大

### 你当前阶段最合适的回答深度
能讲清：
- Redis 为什么需要持久化
- RDB 和 AOF 的核心区别
- 二者的优缺点大方向

就够了。

### 这一模块的输出要求
你至少能独立回答：
1. Redis 为什么需要持久化？
2. RDB 和 AOF 有什么区别？
3. Redis 有了持久化，为什么核心数据通常还是放 MySQL？

## 模块 6：缓存常见问题与高频坑

### 学习目标
这是最容易被问的一组追问，必须会基础版。

### 这轮必须掌握的 3 个问题

#### 1. 缓存穿透
查询一个根本不存在的数据，请求每次都落到数据库。

你要会讲：
- 典型原因是请求的数据本身就不存在
- 常见思路有缓存空值、布隆过滤器、参数校验

#### 2. 缓存击穿
某个热点 key 失效瞬间，大量请求直接打到数据库。

你要会讲：
- 击穿的重点是“热点 key”
- 常见思路有互斥锁、热点永不过期、逻辑过期

#### 3. 缓存雪崩
大量 key 在同一时间失效，导致请求集中压向数据库。

你要会讲：
- 雪崩的重点是“大量 key 同时失效”
- 常见思路有过期时间加随机值、服务降级、多级缓存、限流

### 可顺带补一层
如果还有精力，可以补：
- 缓存和数据库一致性
- 先更新数据库还是先删缓存
- 延时双删是怎么回事

但这里先不用追到特别复杂。

### 这一模块的输出要求
你至少能独立回答：
1. 缓存穿透、击穿、雪崩分别是什么？
2. 这些问题大概怎么应对？
3. 使用缓存会带来什么副作用？

## 模块 7：把 Redis 和你的项目经历绑定起来

### 学习目标
Redis 最终不是一门独立知识，而是你项目表达里的一个支撑点。

你必须准备这样的表达：
- 我项目里为什么用了 Redis
- Redis 存了什么数据
- 它解决了什么问题
- 如果不用 Redis 会怎么样
- 使用 Redis 后可能带来的问题是什么

### 推荐你用的项目表达框架
你可以按下面这套模板组织：

1. 业务场景是什么
2. 为什么不能直接全查 MySQL
3. Redis 存了什么
4. 这样做带来了什么收益
5. 可能的风险和补充措施是什么

### 一种通用回答模板
“在这个场景里我们引入 Redis，主要是因为这部分数据访问频率比较高，而且对响应速度要求更敏感。如果每次都直接查 MySQL，会增加数据库压力，接口响应也会更慢。所以我们把热点数据或者临时状态放到 Redis 中，作为缓存层来承接高频请求。这样做的好处是降低数据库压力、提升响应速度，但同时也要考虑缓存一致性、失效策略和异常情况下的兜底方案。”

### 这一模块的输出要求
你至少要为自己的项目准备 2~3 段 Redis 相关表达：
- 一个偏缓存
- 一个偏临时状态 / 计数 / 分布式协作
- 一个偏“如果不用 Redis 会怎样”的解释
