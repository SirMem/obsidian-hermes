## 一、这一节的目标：把 MVCC 从“模糊感觉”拆成三个具体角色

你前一节已经有了一个很重要的直觉：

- 一条记录不只是一个当前值
- 并发事务下，不同事务可能看到不同版本
- MVCC 的核心，是让数据库按事务视角读到合适版本

但如果只停在这里，MVCC 还是容易显得有点玄。

因为你会隐约知道：

“数据库好像能让我看到适合我的版本。”

可新的问题立刻就来了：

- 它到底什么时候看历史版本？
- 又什么时候看当前最新值？
- 谁来决定“这次我能看到哪一版”？

这时候，就要把 MVCC 从一个大概念，拆成三个更具体的角色：

1. 快照读
2. 当前读
3. Read View

你可以把这三个东西先想成：

- 快照读：偏向“读一个适合事务视角的版本”
- 当前读：偏向“读当前真正最新、并准备处理它”
- Read View：负责定义“对这个事务来说，哪些版本可见”

这三个点一打通，MVCC 就不会再像魔法。

## 二、先别急着背名词，先问：为什么‘读’还要分两种

很多人第一次学这里会懵：

“读数据不就是读数据吗？为什么还要分快照读和当前读？”

这是非常正常的疑问。

因为在日常语言里，“读”好像就一个动作。
但在事务隔离和并发控制里，不同的“读”其实目标不一样。

### 1. 有些读只是想看一个一致的结果

比如：
- 普通查询订单
- 查看用户资料
- 读取一批商品信息
- 在事务里多次读取做判断

这类读很多时候重点是：

“给我一个对当前事务来说合理且一致的版本。”

它不一定非要抓住最新值，也不一定要立即和别人争抢锁。

### 2. 有些读不是单纯看，而是准备‘处理当前对象’

比如：
- 我先 select ... for update 把这行锁住
- 然后我准备更新它
- 或者我要确保我看到的是当前最新状态，再决定后续写入

这类读就不能只看历史快照了。
因为你接下来不是做“历史观察”，而是要对“现在这条真实记录”采取行动。

所以数据库必须区分：

- 这是一个偏“观察型”的读
- 还是一个偏“处理型”的读

这就是读会分成快照读和当前读的根本原因。

## 三、先理解快照读：它是在读“对当前事务可见的版本”

### 1. 什么叫快照读

你可以先把快照读理解成一句话：

“不是一定读当前最新值，而是读一个对当前事务来说可见、合理、稳定的版本。”

这个定义很重要。
因为它说明快照读的重点不是“新不新”，而是“对当前事务是否可见”。

### 2. 为什么叫‘快照’

因为它很像：

数据库不是每次都把你推到最前线看正在变化的现场，
而是给你一个“符合你当前事务视角的画面”。

这个画面未必是整个数据库此刻最最新的状态，
但它是一个对你来说成立的、可用来稳定阅读的状态。

### 3. 快照读主要想解决什么

主要是为了支持：

- 普通读取不总被写操作阻塞
- 同一事务里读取结果尽量稳定
- 事务隔离不必总靠强锁来实现

所以快照读常常和 MVCC 紧密绑定。

### 4. 你可以把它理解成什么角色

快照读更像是：

“我现在主要是想看，我不一定要立刻控制这条数据；请给我一个对我这个事务而言正确的可见版本。”

## 四、再理解当前读：它是在读“此刻真正要处理的那条记录”

### 1. 什么叫当前读

当前读可以先理解成：

“读取数据的当前最新状态，并且通常会配合锁，确保你接下来对它的处理是面向当前真实记录的。”

### 2. 它和快照读最本质的区别是什么

快照读关注的是：
- 对当前事务可见的版本

当前读关注的是：
- 当前真实最新的记录状态
- 并且往往要防止别人同时乱改

所以当前读不是为了“安静观察”，而更像是：

“我现在要碰这个东西了，先把当前真实情况确认下来，并按需要控制住。”

### 3. 哪些操作更像当前读

入门阶段你可以先记住几个典型：

- select ... for update
- select ... lock in share mode（或共享锁读的同类语义）
- update
- delete

它们本质上都不是“随便看看”。
它们是在处理当前记录，所以更接近当前读。

### 4. 为什么更新前不能只靠快照读

假设事务 T1 很早开始，看到库存是 10。
后来事务 T2 已经把库存改成 8 并提交。

这时如果 T1 还只抱着自己旧快照去更新，就很容易把事情做错。

因为更新这种操作面对的是“当前真实世界”，不是单纯回顾历史。

所以你要理解：

- 快照读适合“读一个对事务可见的版本”
- 当前读适合“处理当前真实数据”

## 五、现在你就能先建立一个粗分工

先不要追求百分之百严丝合缝，先记住这个最实用的分工：

### 快照读：
- 典型场景是普通 select
- 不强调锁住当前行
- 更强调读取一个事务可见版本
- 核心价值是读得稳定、减少冲突

### 当前读：
- 典型场景是 update / delete / select ... for update
- 面向当前最新记录
- 往往和加锁有关
- 核心价值是确保你处理的是现在这条真实数据

这个粗分工一旦立住，后面就不容易混。

## 六、那 Read View 到底是干什么的

到这里，问题又自然出现了：

“如果快照读不是直接读当前最新值，那数据库怎么知道我这次能看到哪个版本？”

这就是 Read View 出场的地方。

你可以先把 Read View 理解成：

“数据库给当前事务生成的一套‘可见性判断规则’，用来决定某个版本对这个事务来说能不能看见。”

注意，它不是“数据本身”，也不是“版本本身”。
它更像一套判断尺子。

也就是说：

- 数据记录可能存在多个版本
- 但不是每个版本你都能看见
- Read View 就是在判断：当前这个事务，哪些版本算可见，哪些版本不算可见

这就是它的核心职责。

## 七、你可以把 Read View 理解成‘事务的可见性滤镜’

这是我最建议你现在记住的比喻。

同一条记录可能有：
- 旧版本
- 新版本
- 别的事务还没提交的版本

而 Read View 就像一个滤镜。
它会说：

- 这个版本，对你可见
- 那个版本，对你不可见
- 太新的不行
- 别人未提交的不行
- 符合你事务视角的那个可以

所以 Read View 不是在“创造版本”，而是在“筛版本”。

这点特别重要。

很多人会误以为：
“Read View 是不是保存了一份数据副本？”

不是。

你应该先理解成：

“它是一套用来判断版本可见性的视角规则。”

## 八、把三者放在一起看，就不容易乱了

现在把它们摆在一起：

### 1. 快照读是什么
是读取“对当前事务可见的那个版本”。

### 2. 当前读是什么
是读取“当前真实最新、并准备处理的那条记录”。

### 3. Read View 是什么
是帮助快照读判断“哪些版本对当前事务可见”的规则集。

所以它们之间的关系可以先理解成：

- MVCC 是大思路
- 快照读是 MVCC 最常见的使用场景之一
- Read View 是快照读判断版本可见性的关键机制
- 当前读则是在另一类场景下，不走“只看历史快照”这条路，而是面对当前记录

这四句话你记住，框架就很稳。

## 九、用一个例子把三者串起来

假设有一条商品记录：

- price = 100

然后发生这些事：

### 时刻 1
事务 T1 开始。

### 时刻 2
T1 执行普通 select，读取商品价格。
看到 100。

这时它更像是在做快照读。
数据库会根据 T1 的事务视角，用 Read View 判断：

“哪个版本对 T1 可见？”

结果是：100 这个版本可见。

### 时刻 3
事务 T2 开始，把价格改成 120，并提交。

### 时刻 4
T1 再次执行普通 select。

如果当前隔离语义要求 T1 在事务内读取保持稳定，那它此时仍可能看到 100。

为什么？
不是因为数据库没更新，
而是因为 T1 此时做的仍是快照读，
而 Read View 会让它继续看到对自己可见的那个版本。

### 时刻 5
如果 T1 这时执行的是：
select ... for update

那性质就变了。
这更像当前读。

它不是在问：
“从我的历史视角看，哪个版本可见？”

而是在问：
“现在这条记录真正是什么状态？我要处理它，并且必要时锁住它。”

这时它就更会面向当前真实最新版本去工作。

这个例子里，你就能看到三者的分工。

## 十、为什么说 MVCC 不能脱离“读的类型”单独理解

这点非常关键。

很多人学 MVCC 时，心里会有一种模糊印象：

“反正 MVCC 就是一直看历史版本。”

这其实是不准确的。

更准确地说：

- 对很多普通 select，MVCC 会通过快照读发挥作用
- 但对 update、delete、for update 这类操作，数据库面对的是当前读逻辑

所以你不能把 MVCC 理解成：

“所有场景一律读旧版本。”

真正正确的理解是：

“数据库会根据操作类型，决定是按事务视角做快照读取，还是面向当前记录做当前读取。”

这也是为什么你必须先把‘读分两种’搞清楚。

## 十一、你现在还不用死记 Read View 的内部字段，但要知道它回答的是哪类问题

Read View 以后你肯定还会看到一些更底层的内容，比如：

- 活跃事务列表
- 最小活跃事务 ID
- 最大事务 ID
- 某个版本的创建事务 ID 是否可见

这些东西后面可以再学。

但你现在先别急。
你此刻最应该先抓住的是：

Read View 不是让你背字段，
而是让你理解：

“快照读时，数据库必须有一套规则来判断某个版本对当前事务是否可见。”

只要这点立住，后面那些字段才有意义。

## 十二、为什么这套机制很重要：它帮助数据库做到‘尽量读写不互相卡死’

如果没有这套思路，数据库想实现隔离，很容易走向一种粗暴方案：

- 读的时候把别人拦住
- 写的时候把别人拦住
- 所有人尽量排队

这样虽然容易理解，但并发性能差。

MVCC + 快照读 + Read View 的价值就在于：

“很多时候，我不必因为有人正在改数据，就让普通读取全部卡住；我也不必因为有人在读，就把写全部堵死。”

数据库可以换一种思路：

“我给读操作一个合适版本看，你继续读；写操作去处理当前记录，该加锁的再加锁。”

这就是为什么 MVCC 在工程上这么重要。

## 十三、先给你一个现阶段最实用的总结公式

如果你现在觉得内容有点多，可以先只记这个公式：

普通稳定读取 -> 更像快照读 -> 靠 Read View 判断看哪个版本

需要处理当前记录、准备修改/加锁 -> 更像当前读 -> 面向当前最新状态

这个公式不覆盖全部细节，但在你当前阶段已经非常够用。

## 十四、把前面知识链到这里，你会发现逻辑已经闭环了

我们一路走过来，其实已经形成了很完整的链路：

### 第 01 节：
MySQL 是核心业务事实存储中心。

### 第 02 节：
事务保证业务动作整体成立。

### 第 03 节：
并发事务会制造各种异常。

### 第 04 节：
隔离级别是在正确性和性能之间做取舍。

### 第 05 节：
MVCC 的关键问题是：当前事务到底该读哪一版数据。

### 第 06 节：
MVCC 之所以能落地，是因为数据库区分了快照读和当前读，并用 Read View 帮快照读判断版本可见性。

到这里，MVCC 就已经不是玄学名词，而是一套开始能说清角色分工的机制了。

## 十五、这一节最核心的认知总结

请先牢牢记住下面这些话：

- MVCC 不是“所有读都看旧版本”的魔法
- 读要分两类：快照读和当前读
- 快照读的重点，是读取一个对当前事务可见且相对稳定的版本
- 当前读的重点，是面对当前真实最新记录，并通常配合锁来处理它
- Read View 不是数据副本，而是判断“哪个版本对当前事务可见”的规则集
- 快照读 + Read View，是 MVCC 能支持事务隔离、又减少读写冲突的重要方式

## 十六、你现在应该能回答的 6 个问题

学完这一节，你至少要能用自己的话回答：

1. 为什么数据库里的“读”还要分快照读和当前读？
		因为有些读只是想稳定观察一个对事务可见的版本，而有些读是为了处理当前真实记录，目标不同，所以数据库必须区分
2. 快照读最朴素的理解是什么？
		就是读取一个对当前事务可见、合理、相对稳定的数据版本，而不一定是当前最新值
3. 当前读最朴素的理解是什么？
		就是面向当前真实最新记录的读取，通常用于准备更新、删除或加锁处理
4. Read View 到底在干什么？
		它提供一套可见性判断规则，用来决定某个数据版本对当前事务是否可见
5. 为什么说 Read View 不是数据副本？
		因为它不是在存一份记录内容，而是在定义一个事务看版本时的过滤规则
6. 为什么 MVCC 能减少读写冲突？
		因为很多普通读取可以通过快照读去看合适版本，而不必总和写操作直接互相阻塞

如果这六个问题你能说顺，下一步再学 undo log、版本链、Read View 细节，你就会轻松很多。
