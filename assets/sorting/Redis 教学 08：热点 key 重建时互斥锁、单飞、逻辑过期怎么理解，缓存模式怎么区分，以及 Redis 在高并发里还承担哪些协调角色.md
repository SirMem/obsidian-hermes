## 一、这节课为什么要学

前面几节你已经逐步把 Redis 作为缓存的主线理顺了：
- Redis 为什么快
- 常见数据结构和业务场景怎么对应
- 为什么核心数据通常还是放 MySQL
- 缓存穿透、击穿、雪崩分别是什么
- 缓存和数据库一致性为什么天然会有时序问题
- 删除缓存失败时，重试、MQ、binlog 各自怎么参与补偿

走到这里，面试官或者项目复盘里通常会继续追问三类问题：
- 热点 key 失效以后，为什么数据库会突然被打爆？互斥锁、单飞、逻辑过期各自在解决什么？
- Cache Aside、Read Through、Write Through、Write Back 这些缓存模式到底怎么区分？
- Redis 在项目里除了“存缓存”，为什么还经常承担一些高并发协调角色？

这三组问题看似分散，其实背后是一条连续主线：

“Redis 不只是把数据放快一点的地方，它还经常站在数据库前面做读写调度、回源控制、并发协调和系统削峰。”

如果你能把这条主线讲顺，面试时你对 Redis 的理解就不会停留在“背定义”，而会更像真正做过系统的人。

## 二、先把背景补齐：什么叫热点 key 重建，为什么它会变成数据库问题

### 1. 热点 key 是什么

热点 key 可以先粗理解为：
- 被访问频率特别高的 key
- 一旦失效，会有大量请求同时打到它

最常见例子：
- 热门商品详情
- 秒杀商品库存信息
- 首页热点榜单
- 热门直播间信息

比如：
- `product:1001`
- 平时每秒可能有几千次读取

### 2. 什么叫“重建缓存”

假设 Redis 里这个 key 过期了。
这时新的请求进来，发现缓存 miss，于是去 MySQL 查，再把结果写回 Redis。

这个“从 DB 查出数据，再回填 Redis”的过程，就叫缓存重建。

### 3. 为什么它会突然打爆数据库

关键点在于：

如果这是个普通 key，miss 了查一次库也没什么。
但如果这是热点 key，同一时刻来了很多请求，那么它们会一起发现缓存失效，然后一起回源数据库。

最小例子：
- 某商品详情缓存刚过期
- 1 秒内来了 5000 个请求
- 如果都直接查库，就可能不是 1 次 DB 查询，而是 5000 次

于是问题就从“缓存 miss”升级成：

“热点 key 失效后，怎么避免并发请求一起冲击数据库。”

这就是为什么缓存击穿本质上最后会落到数据库扛压问题上。

## 三、热点 key 重建时，互斥锁、单飞、逻辑过期分别怎么理解

这一部分你不要把它们当成孤立技巧，而要把它们看成三种不同的权衡方案：
- 谁负责回源
- 其他请求要不要等
- 系统更优先保一致性，还是更优先保性能和稳定性

### 1. 互斥锁：一个人去重建，其他人先别一起冲进去

互斥锁的核心思想是：

“同一个热点 key 在重建缓存时，只允许一个线程或实例去查数据库并回填缓存，其他请求等待、重试，或者短暂失败。”

你可以把它想成：
- 一个房间里只有一个人可以进去维修
- 其他人排队等修好

#### 最小流程
1. 请求读 Redis，发现 `product:1001` 不存在
2. 尝试抢一个锁，比如 `lock:product:1001`
3. 抢到锁的人去查 DB，并把数据回填 Redis
4. 没抢到锁的人：
   - sleep 一小会儿再重试
   - 或者快速返回“稍后再试”

#### 最小伪代码
```java
Object data = redis.get(key);
if (data == null) {
    if (tryLock(lockKey)) {
        data = db.query(id);
        redis.set(key, data, ttl);
        unlock(lockKey);
        return data;
    } else {
        Thread.sleep(50);
        return retry();
    }
}
return data;
```

#### 它在解决什么
它解决的是：
“缓存 miss 后，大量线程同时回源数据库。”

它把原本并发的数据库访问，尽量收缩成一次真正的重建动作。

#### 优点
- 思路直观
- 能显著降低数据库瞬时压力
- 对热点 key 很有效

#### 缺点
- 没拿到锁的请求要等待
- 如果锁实现不好，可能有死锁、误删锁等问题
- 高并发下等待和重试本身也有成本

#### 一句话记忆
互斥锁 = “只让一个人回源重建，其余人等结果”

---

### 2. 单飞（singleflight）：不是大家都查一次，而是同一批请求共用一次查询结果

单飞和互斥锁很像，但关注点不同。

互斥锁更像在说：
“谁能进去做事。”

单飞更像在说：
“同一个 key 的并发请求不要重复做同一件事，大家共享一次执行结果。”

你可以把它理解成“拼车”：
- 不是每个人都各自打车去数据库
- 而是同一批人拼一辆车

#### 最小流程
1. A 请求发现 `product:1001` miss，于是发起 DB 查询
2. B、C、D 请求也来了，也发现 miss
3. 系统发现“这个 key 已经有人在查了”
4. B、C、D 不再重复查 DB，而是等待 A 的结果
5. A 查完后，把结果共享给 B、C、D，并写入缓存

#### 它和互斥锁的差别
互斥锁强调的是“只有一个执行者”；
单飞强调的是“同 key 请求合并，共享一次结果”。

在工程实现上，很多单飞方案内部也会借助锁或 promise/future 机制，但它的表达重心更偏向请求合并，而不是资源互斥。

#### 它适合什么场景
- 单机服务内部有大量同 key 并发查询
- 希望减少重复回源
- 希望同一批请求直接复用同一次结果

#### 它的边界
- 很多 singleflight 实现主要适用于单进程内
- 分布式多实例下，往往还需要结合分布式锁或别的协同手段

#### 一句话记忆
单飞 = “同一个 key 的并发 miss 合并成一次回源”

---

### 3. 逻辑过期：先把旧值顶上去，再后台慢慢更新

逻辑过期是和前两种差异最大的一种思路。

它的核心不是“大家等一个新值回来”，而是：

“即使这个数据逻辑上已经过期，也先返回旧值，随后由后台异步重建缓存。”

也就是说，它更偏向保性能和稳定性，而不是让每次过期都强制等待新值。

#### 它怎么存数据
缓存里存的通常不只是业务数据本身，还会带一个逻辑过期时间：

```json
{
  "data": {"id": 1001, "price": 120},
  "expireTime": "2026-04-15 16:00:00"
}
```

#### 读取流程
1. 读到 Redis 里有值
2. 检查里面的 `expireTime`
3. 如果还没过期，直接返回
4. 如果逻辑过期了：
   - 先返回旧值
   - 同时尝试抢锁
   - 由后台线程异步查 DB 并刷新缓存

#### 它在解决什么
它解决的是：
“热点 key 过期时，不要让大量请求阻塞在缓存重建上。”

#### 优点
- 用户几乎不会因为缓存重建而等待太久
- 高并发下响应更稳定
- 非常适合热点数据

#### 缺点
- 用户可能短时间读到旧值
- 业务要接受最终一致，而不是严格实时一致
- 实现比单纯 TTL 稍复杂

#### 适合什么场景
- 热点数据非常热
- 对极短时间旧值可容忍
- 更在意吞吐、稳定性、低延迟

#### 一句话记忆
逻辑过期 = “先返回旧值，再异步更新”

---

### 4. 三种方案怎么放在一起理解

你可以直接记这个对比：

1）互斥锁
- 重点：控制谁能回源
- 其他请求：等待 / 重试
- 倾向：相对更保一致性

2）单飞
- 重点：同 key 请求合并
- 其他请求：共享同一次结果
- 倾向：减少重复工作

3）逻辑过期
- 重点：先服务请求，再后台修正
- 其他请求：直接拿旧值
- 倾向：更保性能和稳定性

你在面试里可以这样概括：

“热点 key 重建时，互斥锁是让一个线程回源、其他线程等待；singleflight 是把同 key 的并发 miss 合并成一次回源；逻辑过期则是先返回旧值、后台异步更新。前两者更偏控制回源并发，逻辑过期更偏高可用和低延迟。”

---

## 四、缓存 Aside、读写穿透、写回这些缓存模式应该怎么区分

这里最容易混乱的原因是：
大家经常听到一堆英文名词，但没有抓住区分维度。

其实最有效的区分方法就两个：
- 读请求 miss 以后，谁负责去 DB 查
- 写请求发生时，缓存和 DB 谁先写、谁后写、是同步还是异步

### 1. Cache Aside：应用自己维护缓存和数据库

这是业务项目里最常见的一种模式，也叫旁路缓存模式。

#### 读流程
1. 应用先查 Redis
2. 命中就直接返回
3. miss 了再查 MySQL
4. 再把结果写回 Redis

#### 写流程
1. 先更新 MySQL
2. 再删除 Redis 缓存

你会发现，这种模式下：
- 应用自己感知 miss
- 应用自己查 DB
- 应用自己决定删缓存或回填缓存

也就是说，应用代码在显式地协调缓存和数据库。

#### 它为什么常见
- 简单直观
- Redis 本身通常不是完整的 DB 代理层
- 业务最容易自己控制流程
- 出问题时也更容易排查

#### 一句话记忆
Cache Aside = “应用自己管缓存和 DB，读时 miss 再查库，写时常见是先更库再删缓存”

---

### 2. Read Through：应用问缓存层，缓存层自己去 DB 加载

Read Through 的关键是：
应用不自己处理 miss 后查库，而是由缓存层代劳。

#### 读流程
1. 应用只向缓存层要数据
2. 缓存层发现 miss
3. 缓存层自己去 DB 查
4. 查到后回填缓存，再把结果返回给应用

#### 它和 Cache Aside 的核心区别
- Cache Aside：应用自己补
- Read Through：缓存层代理补

所以它更像一种封装方式。

#### 一句话记忆
Read Through = “缓存 miss 后，由缓存层代替应用查库并回填”

---

### 3. Write Through：写缓存时同步写数据库

Write Through 的关键是：
应用把写请求发给缓存层，缓存层同步把数据写到 DB。

#### 写流程
1. 应用写缓存层
2. 缓存层同步写 DB
3. DB 写成功后，整个写请求才算成功

#### 它的特点
- 应用视角上像是在“写缓存”
- 但背后缓存层会同步维护 DB
- 一致性逻辑更集中
- 但写延迟一般会更高，因为它要同步落 DB

#### 一句话记忆
Write Through = “先写缓存层，但缓存层会同步把数据写到 DB”

---

### 4. Write Back / Write Behind：先写缓存，数据库以后再慢慢落

Write Back 又叫 Write Behind。

它和 Write Through 最大的差别就是：
是否同步落数据库。

#### 写流程
1. 应用更新缓存层
2. 缓存层先返回成功
3. 后台异步批量把变更刷回 DB

#### 它的优点
- 写入快
- 容易合并多次更新
- 适合高频写场景

#### 它的代价
- 缓存和 DB 会暂时不一致
- 如果回写链路出问题，可能丢数据
- 对可靠性、持久化、补偿机制要求更高

它有点像操作系统里的脏页回写：
先把修改留在快的地方，稍后再刷到底层介质。

#### 一句话记忆
Write Back = “先写缓存，异步回写数据库”

---

### 5. 这几种模式怎么一眼区分

可以直接记下面这个最小表述：

- Cache Aside：应用自己协调缓存和 DB
- Read Through：读 miss 后，缓存层帮你查 DB
- Write Through：写缓存时，同步写 DB
- Write Back：写缓存后，异步写 DB

在实际互联网项目里，最常见仍然是 Cache Aside。
原因不是它理论上最优，而是它最现实、最容易控制、最容易和现有业务代码结合。

## 五、Redis 在项目里除了缓存，还常承担哪些“高并发协调角色”

这一部分很重要，因为它决定你是不是把 Redis 仅仅看成“存热点数据的地方”。

更贴近真实项目的理解是：
Redis 经常被用作高并发系统里的轻量协调器。

它不只是存数据，还在控顺序、控频率、控资源、做共享状态。

### 1. 分布式锁：控制同一资源别被多个实例同时处理

#### 典型场景
- 热点 key 重建时，只允许一个实例回源
- 同一个订单不能并发重复处理
- 同一个优惠券发放任务只允许执行一次

#### 它在扮演什么角色
Redis 在这里不是缓存，而是在做并发控制。

#### 最小例子
```text
SET lock:order:1001 uuid NX EX 10
```

含义可以粗理解为：
- key 不存在时才能设置成功
- 成功的人拿到锁
- 并且这个锁 10 秒后自动过期，避免死锁

---

### 2. 限流器：控制访问速率，保护下游系统

#### 典型场景
- 接口每秒最多 1000 次请求
- 单用户一分钟最多发 5 次验证码
- 秒杀活动入口防刷

#### Redis 为什么适合
- 有原子自增
- 有过期时间
- 单 key 高并发性能好

#### 最小例子
```text
INCR sms:uid:1001
EXPIRE sms:uid:1001 60
```

粗理解：
一分钟内这个计数不断累加，超过阈值就拒绝。

---

### 3. 分布式计数器 / 库存预扣：在高并发下先把数量控住

#### 典型场景
- 秒杀库存扣减
- 活动名额控制
- 今日领取次数限制
- 某类资源的全局配额控制

#### 为什么它重要
Redis 的原子加减很适合在流量高峰时快速做并发控制，先把数量扣住，再交给后续链路慢慢落库或补偿。

但你也要知道：
Redis 更适合做“高并发前置控制”，最终正确账本通常还是要回到 DB 或消息链路保证。

---

### 4. 排队与削峰：把瞬时洪峰变成可控处理速度

#### 典型场景
- 秒杀请求先入队，再异步创建订单
- 批量通知任务排队消费
- 某类重任务先进入待处理列表

#### Redis 在扮演什么角色
它在这里更像一个轻量队列或削峰器。

常见数据结构：
- List
- Stream
- ZSet（延迟任务）

它不一定替代专业 MQ，但在中小场景里很常见。

---

### 5. 幂等控制：防止同一请求重复执行

#### 典型场景
- 用户重复点击支付
- MQ 消息重复投递
- 任务重试导致同一业务重复执行

#### 做法
把业务唯一标识写成一个 Redis key：
- 第一次 setnx 成功，说明可以执行
- 后续再来发现已存在，说明已经处理过或正在处理

所以 Redis 在这里是在做“防重”。

---

### 6. 会话与共享状态：多节点之间共享临时状态

#### 典型场景
- 登录 session
- 用户在线状态
- WebSocket 连接路由信息
- 某个短期业务状态的共享

这类场景里，Redis 更像一个“分布式共享内存”。

---

### 7. 排行榜与实时排序：按分值动态维护结果

#### 典型场景
- 游戏积分榜
- 热门帖子榜
- 销量榜
- 活跃用户榜

#### 为什么常用 Redis
因为 ZSet 天然支持按 score 排序，读 topN 也很方便。

这里它承担的是“实时排序引擎”的角色。

---

### 8. 轻量通知与事件协调：Pub/Sub、Stream 做服务间通知

#### 典型场景
- 某配置更新后通知多个服务刷新本地缓存
- 某个任务状态变化做广播
- 轻量级事件分发

这里 Redis 更像一个轻量事件总线。

当然，强可靠消息场景通常还是更偏专业 MQ。

---

## 六、把这三组内容串成一套面试表达

如果面试官连续追问，你可以把前面内容整合成下面这种回答：

“Redis 在项目里不只是缓存，它经常还承担一些高并发协调角色。比如做缓存时，热点 key 失效后如果所有请求一起回源，会把数据库打爆，所以常见会用互斥锁、singleflight 或逻辑过期来控制缓存重建。互斥锁是一个线程回源、其他线程等待；singleflight 是把同 key 的并发 miss 合并成一次查询；逻辑过期则是先返回旧值，再由后台异步更新。

从缓存和数据库配合模式看，业务里最常见的是 Cache Aside，也就是应用自己负责查缓存、回源数据库和删缓存；Read Through/Write Through 更像由缓存层代理数据库读写；Write Back 则是先写缓存、再异步回写数据库，吞吐更高但一致性更弱。

另外 Redis 在项目里还经常承担分布式锁、限流、库存预扣、幂等控制、轻量队列、排行榜、会话共享这些角色，本质上它经常站在系统前面做高并发下的协调与节流。”

## 七、这一节最容易混淆的点

### 1. “读写穿透”不是“缓存穿透”

两者不是一个概念：
- Read Through / Write Through：是在讲缓存架构模式
- 缓存穿透：是在讲请求查不存在的数据，导致大量请求落到 DB

不要混着说。

### 2. 单飞不等于分布式锁

- 单飞更偏请求合并
- 分布式锁更偏资源互斥

有些实现里两者会一起用，但概念层面不是同一个东西。

### 3. 逻辑过期不是 TTL 真过期

- TTL 真过期：key 直接不存在了
- 逻辑过期：key 还在，只是值内部标记“该刷新了”

这个区别在面试里很容易被追问。

### 4. Write Back 吞吐高，不代表它适合所有业务

它的代价是：
- 更复杂的可靠性保障
- 更长的一致性窗口
- 一旦回写失败，问题比单纯缓存 miss 更麻烦

所以不要把它说成“更高级就一定更好”。

---

## 八、你现在最应该记住的最小版本

### 1. 热点 key 重建三件套
- 互斥锁：一个人回源，其他人等
- 单飞：同 key 并发请求合并成一次回源
- 逻辑过期：先返回旧值，再后台更新

### 2. 缓存模式最小区分
- Cache Aside：应用自己协调缓存和 DB
- Read Through：缓存层在 miss 后帮你查 DB
- Write Through：写缓存时同步写 DB
- Write Back：写缓存后异步写 DB

### 3. Redis 的高并发协调角色
- 分布式锁
- 限流
- 分布式计数器 / 库存预扣
- 排队削峰
- 幂等控制
- 会话共享
- 排行榜
- 轻量通知
