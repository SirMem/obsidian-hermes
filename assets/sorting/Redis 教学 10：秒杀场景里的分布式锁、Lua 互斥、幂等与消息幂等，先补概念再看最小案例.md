## 二、先补最基础的：什么叫分布式锁

### 1. 先不要急着背定义

你先想一个最小问题：

系统现在有 3 台服务实例：
- 机器 A
- 机器 B
- 机器 C

某个业务动作同一时刻只能允许一个执行者进去，比如：
- 同一个用户只能创建一次订单
- 同一个任务只能有一台机器执行
- 同一个热点 key 只能有一个实例去回源数据库

这时如果只用 Java 本地的 `synchronized` 或 `ReentrantLock`，只能锁住单机进程内的线程，锁不住别的机器。

所以就需要一个“所有机器都能看见的公共锁”。
这就是分布式锁最朴素的理解。

### 2. 一句话定义

分布式锁可以先这样理解：

“分布式锁就是把‘同一时刻只能一个执行者进入关键区’这件事，扩展到多机器、多进程环境里的锁。”

### 3. 它到底在保护什么

这里特别关键。
分布式锁通常保护的不是“数据库某一行数据本身”，
而是：

“某个业务动作的执行资格。”

比如：
- 用户 1001 的下单流程
- 商品 3001 的缓存重建流程
- 每日结算任务的执行资格

所以它更偏业务层互斥，不是数据库事务锁的替代品。

### 4. 一个最小例子

假设同一个用户提交订单时，你不希望他在极短时间内并发创建两次订单。
你可以设计一个锁 key：

`lock:order:user:1001`

谁先抢到这把锁，谁进入下单逻辑；
其他请求拿不到锁，就直接失败或稍后再试。

这就是分布式锁最小的业务画面。

## 三、Redis 为什么常被用来做分布式锁

因为 Redis 有几个天然优势：
- 所有实例都能访问它
- 读写快
- 支持原子命令
- 支持给 key 设置过期时间

所以它很适合承载这种：
- 锁状态比较轻
- 生命周期比较短
- 高频判断很多

的业务互斥需求。

但你要注意：

“Redis 适合做业务互斥控制，不代表只要写个 setnx 就万事大吉。”

后面我们要讲的很多细节，恰恰就是因为锁很容易“看起来会了，实际上有坑”。

## 四、最小分布式锁方案先长什么样

### 1. 最朴素的想法

很多人第一次实现锁，会写成这样：

```text
SETNX lock:order:user:1001 1
```

如果返回成功，说明拿到锁；
如果返回失败，说明别人已经拿到了。

这一步的思路没错，
但它还不够。

### 2. 为什么不够

因为它至少缺两个关键东西：

#### 第一，没有过期时间
如果服务拿到锁后宕机，
这把锁可能永远留在 Redis 里，后续请求再也拿不到。

#### 第二，没有唯一 value
如果 value 只是固定写成 `1`，
后面释放锁时你根本无法确认：

“现在要删掉的这把锁，到底是不是我自己当初加的那把？”

所以更合理的加锁应该是：
- 用 `SET key value NX PX ttl`
- value 用唯一 requestId

比如：

```text
SET lock:order:user:1001 req-abc123 NX PX 3000
```

这表示：
- 只有 key 不存在时才能设置成功
- 锁值是 `req-abc123`
- 3 秒后自动过期

### 3. 这时最小方案已经比刚才靠谱很多了

因为它至少解决了：
- 锁不会永久不释放
- 不同请求能区分“谁是锁的持有者”

但还有一个关键问题没解决：

“释放锁时，为什么要用 Lua？”

## 五、为什么释放锁常常要用 Lua，而不是 get 后再 del

这是一个高频重点。

### 1. 很多人会这样写释放逻辑

伪代码：

```java
if (requestId.equals(redis.get(lockKey))) {
    redis.del(lockKey);
}
```

看起来很合理：
- 先看锁是不是自己的
- 如果是，再删除

但这里有一个隐藏问题：

“get 和 del 不是原子动作。”

### 2. 什么叫这里的风险

看一个最小时序：

1. 线程 A 拿到锁，value = `req-A`
2. A 业务执行太久，锁过期了
3. 线程 B 这时拿到了一把新锁，value = `req-B`
4. 线程 A 终于执行完，开始释放锁
5. A 先 `get(lockKey)`，读到的可能已经不是自己的锁
6. 如果实现不严谨，A 可能把 B 的锁删掉

这就出大问题了：
- B 明明已经合法拿到锁
- 却被 A 误删
- 结果两个执行者都可能进入关键区

### 3. Lua 在解决什么

Lua 在这里解决的是：

“校验锁值是不是自己 + 删除锁，这两个动作要么一起完成，要么一起不完成。”

也就是原子性。

### 4. 最小 Lua 解锁脚本

```lua
if redis.call('get', KEYS[1]) == ARGV[1] then
    return redis.call('del', KEYS[1])
else
    return 0
end
```

它的意思是：
- 如果 lockKey 对应的 value 确实等于我的 requestId
- 才执行删除
- 否则什么都不做

### 5. 一句话记忆

加锁常靠：
- `SET key value NX PX ttl`

解锁常靠：
- Lua 做“比较 value + 删除”的原子操作

所以你以后看到“Redis + Lua 分布式锁”，
很多时候说的就是这个释放阶段的原子保障。

## 六、一个完整一点的 Redis + Lua 互斥锁最小案例

假设场景是：

“同一个用户短时间内只能有一个下单流程在执行。”

### 1. 加锁

key：
`lock:order:user:1001`

value：
`req-abc123`

命令：

```text
SET lock:order:user:1001 req-abc123 NX PX 3000
```

如果成功：
- 当前请求进入下单流程

如果失败：
- 说明已经有别的请求在处理
- 直接返回“请勿重复提交”或“稍后再试”

### 2. 执行业务

比如：
- 校验商品状态
- 计算价格
- 生成订单
- 调下游服务

### 3. 释放锁

用 Lua：

```lua
if redis.call('get', KEYS[1]) == ARGV[1] then
    return redis.call('del', KEYS[1])
else
    return 0
end
```

参数：
- `KEYS[1] = lock:order:user:1001`
- `ARGV[1] = req-abc123`

### 4. 这个案例到底说明什么

它说明了 3 件事：
- 分布式锁是让同一时刻只有一个执行者进入关键区
- 锁值必须唯一，不能大家都写死成 1
- 释放锁不能简单 get + del，通常要用 Lua 保证原子性

## 七、但你一定要知道：分布式锁不是幂等

这是非常关键的一步。

很多初学者会把这两个概念混在一起。

### 1. 先分别看它们在管什么

分布式锁管的是：

“同一时刻，谁能执行。”

幂等管的是：

“即使同一个请求重复来了很多次，最终结果也只生效一次。”

你看，这两者保护的维度就不一样。

### 2. 一个最小反例

假设：
- 用户第一次请求下单
- 请求 A 成功拿到锁，创建了一笔订单
- 但客户端因为网络超时，没收到成功响应
- 用户又重试一次，请求 B 稍后再来
- 这时 A 早就执行完了，锁已经释放
- B 这次也成功拿到锁

注意：
这整个过程中，分布式锁并没有失效。
它确实做到了“同一时刻只允许一个执行者”。

但如果系统没有幂等控制，
B 还是可能再创建一笔新订单。

所以：

“锁保证的是并发互斥，不保证重复请求永远只生效一次。”

这就是为什么锁 != 幂等。

---

## 八、什么是幂等

### 1. 先用最简单的话理解

幂等可以先理解成：

“同一个操作，执行一次和执行多次，最终效果应该一样。”

### 2. 在业务里是什么意思

比如“用户提交订单”这件事：
- 第一次请求创建成功
- 第二次相同请求再来，不应该再创建第二笔订单

最终系统里还是只有一笔订单。
这就叫这个操作做到了幂等。

### 3. 为什么系统里会需要幂等

因为重复请求太常见了：
- 用户手抖点两次
- 前端超时自动重试
- 网关重试
- 服务调用重试
- MQ 重复投递
- 消费失败后重新消费

如果没有幂等，你的系统很容易出现：
- 重复下单
- 重复扣库存
- 重复发券
- 重复发通知

### 4. 一个最小业务例子

假设用户购买秒杀商品，业务规则是：
- 一个用户对一个活动商品只能下一单

那么“同一用户 + 同一活动 + 同一商品”就可以看成一份唯一业务身份。

也就是说：
- 第一次处理它，创建订单
- 第二次再处理同一份业务身份，应该识别为重复，不再重复创建

## 九、幂等通常怎么做

幂等不是只有一种做法，
而是你要找到“这次业务的唯一身份”。

常见思路有三类：

### 1. 业务唯一键

比如订单场景，你可以规定：
- `user_id + activity_id + sku_id` 唯一

然后在 MySQL 表上建立唯一索引。

最小效果：
- 第一次 insert 成功
- 第二次再 insert，同样的唯一键直接冲突失败

这是一种非常常见、也非常稳的幂等兜底方式。

### 2. 幂等号 / 请求号

客户端在发起请求时带一个唯一 requestId：
- `requestId = 8f3d...`

服务端处理前先检查：
- 这个 requestId 之前处理过没有

如果处理过，直接返回之前结果或告知重复。

这适合：
- 支付
- 提交订单
- 外部接口回调

### 3. 状态表 / 幂等记录表

系统专门维护一张幂等表，记录：
- 业务唯一号
- 处理状态
- 处理结果

如果后续重复请求进来，先查这张表。

这个思路在 MQ 消费幂等里尤其常见。

## 十、消息幂等是什么意思

### 1. 为什么一引入 MQ，就一定会碰到它

很多人第一次接触 MQ 时会以为：

“生产者发一次，消费者收一次，不就好了？”

但真实系统里，为了可靠性，MQ 很多时候不会承诺“绝对只消费一次”。
更常见的是至少一次投递语义。

翻成大白话就是：

“同一条消息，消费者有可能会收到不止一次。”

为什么？
比如：
- 消费者业务执行成功了，但 ack 丢了
- 消费者执行一半崩了，消息被重新投递
- 网络抖动导致重复投递
- 人工补偿时又重放了一次消息

所以一旦有 MQ，
你就要默认：

“重复消费不是意外，是设计上必须面对的现实。”

### 2. 一句话定义

消息幂等可以这样理解：

“同一条消息即使被消费多次，最终业务结果也只能生效一次。”

---

## 十一、一个最小消息重复消费案例

假设 MQ 里有一条消息：

```json
{
  "messageId": "msg-10001",
  "userId": 1001,
  "activityId": 2001,
  "skuId": 3001
}
```

消费者拿到这条消息后要做：
- 扣减真实库存
- 创建订单

### 正常情况

第一次消费：
- 扣库存成功
- 创建订单成功
- ack 成功

### 问题情况

第一次消费时：
- 扣库存成功
- 创建订单成功
- 但 ack 还没发出去，消费者挂了

于是 MQ 认为这条消息还没处理完，
后面可能把它再投一次。

如果没有消息幂等，第二次消费可能就会：
- 再扣一次库存
- 再建一笔订单

这显然是不能接受的。

所以系统必须能识别：

“这条消息我其实已经处理过了。”

---

## 十二、消息幂等最常见怎么做

### 方案一：消费前先查幂等记录

最常见思路：
- 每条消息都有唯一 messageId
- 消费者先检查“messageId 是否已经处理过”
- 处理过就直接跳过
- 没处理过才继续执行业务

### 最小流程

1. 收到消息 `msg-10001`
2. 查询幂等记录表 / Redis 幂等 key
3. 如果已存在“processed”标记，直接返回
4. 如果不存在，进入业务处理
5. 业务成功后，写入“已处理”标记
6. 再 ack MQ

### 方案二：业务本身带唯一约束

比如消息本质上是在做“一人一单”的下单动作，
那么即使消息重复消费，
你也可以依赖数据库唯一索引兜底：
- `(user_id, activity_id, sku_id)` 唯一

第一次插入成功，第二次插入报唯一键冲突。
这时你可以把它识别成“重复消费但结果已存在”。

### 方案三：幂等记录 + 业务唯一约束双保险

这是工程上更常见的做法。
因为：
- 幂等记录可以在前面快速挡住大部分重复消息
- 数据库唯一约束可以做最终兜底

也就是说：
前面减少无效处理，
后面防止最终结果出错。

## 十三、一个更具体的消息幂等最小案例

假设消费者处理“秒杀下单”消息。

### 1. 消息内容

```json
{
  "messageId": "msg-10001",
  "userId": 1001,
  "activityId": 2001,
  "skuId": 3001
}
```

### 2. Redis 或数据库中做幂等判断

可以有一个 key：

`mq:seckill:processed:msg-10001`

### 3. 处理逻辑伪代码

```java
if (exists(processedKey)) {
    return;
}

try {
    // 1. MySQL 条件扣减真实库存
    // update stock set count = count - 1
    // where sku_id = ? and count > 0

    // 2. 创建订单
    // insert into orders(...)

    // 3. 标记消息已处理
    set(processedKey, "1");

    // 4. ack
} catch (DuplicateKeyException e) {
    // 订单唯一索引冲突，视为重复处理
    set(processedKey, "1");
    // ack
}
```

### 4. 这个例子想说明什么

消息幂等通常不是单靠 Redis，也不是单靠 MySQL，
而是：
- 前面用 Redis / 幂等表识别已处理消息
- 后面用数据库唯一约束兜底结果唯一

---

## 十四、为什么“先查再处理”还不一定完全够

这里你再多想一步。

如果你只是：
1. 先查没有处理过
2. 再开始处理
3. 最后写 processed 标记

中间还是可能有并发窗口。

比如两个消费者几乎同时拿到重复消息：
- 都先查到“还没处理”
- 都继续往下执行

所以更严谨的工程方案里，
常会把“抢到处理资格”也做成原子动作。

比如：
- `SET processed:msgId processing NX PX 60000`
- 抢到的人负责处理
- 处理成功后再把状态改成 done

或者直接依赖数据库唯一约束作为最后裁决。

所以你要建立一个工程认知：

“消息幂等不只是一个 if 判断，而是要考虑并发下谁先抢到处理资格，以及最终结果如何兜底。”

## 十五、把这些概念重新放回秒杀场景，你就不容易乱了

现在我们把前面几个词重新放到秒杀里：

### 1. 分布式锁在秒杀里什么时候用

适合用在：
- 同一个用户下单流程互斥
- 同一个任务只允许一个实例执行
- 热点 key 缓存重建只允许一个实例回源

它解决的是：
“同一时刻谁能执行。”

### 2. Lua 锁方案在秒杀里具体补什么

它补的是：
- 加锁时的原子互斥
- 解锁时避免误删他人锁

它解决的是：
“锁实现不要自己埋坑。”

### 3. 幂等在秒杀里什么时候用

适合用在：
- 用户重复点击下单
- 请求重试
- 回调重复通知

它解决的是：
“重复请求最终只能生效一次。”

### 4. 消息幂等在秒杀里什么时候用

适合用在：
- MQ 消费者处理秒杀下单消息
- 库存扣减消息
- 发券消息

它解决的是：
“同一条消息被消费多次，也不能重复扣库存、重复建单。”

所以你可以这样区分：

- 分布式锁：管同一时刻的互斥
- 幂等：管重复请求的最终唯一结果
- 消息幂等：管重复消息消费的最终唯一结果

## 十六、最后给你一个“一口气说清”的口语版总结

如果面试官追问这些概念，你可以先用这个版本回答：

“分布式锁本质上是在多机器环境里控制同一时刻只能一个执行者进入关键区，比如同一用户下单流程只能有一个请求执行。Redis 常被拿来做分布式锁，因为它快、支持原子命令，也方便设置过期时间。比较常见的做法是用 SET key value NX PX ttl 来加锁，value 要放唯一 requestId，释放锁时不能简单 get 后 del，因为这两个动作不是原子的，可能误删别人的锁，所以通常会用 Lua 脚本做比较 value 再删除。

幂等和锁不是一回事。锁解决的是同一时刻谁能执行，幂等解决的是重复请求来了多次，最终结果也只能生效一次。像一人一单，通常会通过业务唯一键、唯一索引、幂等号这些方式实现。

如果引入 MQ，就还要做消息幂等，因为同一条消息可能被重复消费。常见做法是给消息一个唯一 messageId，消费前先查幂等记录，处理成功后写入已处理标记，同时再用数据库唯一约束做最终兜底，这样即使消息重复投递，也不会重复扣库存、重复建单。”
