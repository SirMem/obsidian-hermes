## 一、这一模块为什么现在学

按照第一阶段路线，事务和索引之后，下一步最自然就是“锁”。

因为你前面已经搭起了两条主线：
- 事务主线：为什么业务动作需要整体正确，为什么并发会制造异常，隔离级别在做什么取舍，MVCC 又是在解决什么问题
- 索引主线：为什么查询会慢，索引如何减少扫描范围，explain 怎么帮助判断 SQL 的访问路径

那接下来就要补上一个非常关键的并发控制问题：

“当多个事务同时碰同一批数据时，数据库怎么避免它们互相把结果搞乱？”

这就是锁要解决的事。

所以这一模块不是单纯背“共享锁、排他锁、行锁、表锁”的名词，
而是要把锁放回项目并发场景里理解：
- 为什么会锁
- 锁住的到底是什么
- 为什么有时候会阻塞
- 为什么有时候会死锁
- 为什么索引是否命中，会影响锁的范围和性能

这块一旦学通，你在项目表达里就会比“只会写事务”更进一步，开始能解释并发更新、抢库存、状态流转、锁等待这些真实问题。

## 二、模块 3 的总目标

这一模块学完后，你要达到的目标不是背全底层实现，而是能比较稳定地讲清楚下面几件事：

1. 为什么数据库需要锁
2. 共享锁、排他锁分别在保护什么
3. 行锁、表锁大概区别是什么
4. 为什么 InnoDB 里“锁”和“索引命中”经常绑在一起出现
5. 间隙锁、临键锁是为了解决什么问题
6. 为什么会出现锁等待、死锁
7. 项目里哪些并发问题，本质上是锁竞争问题
8. 怎么用“正确性 vs 并发性能”的视角去讲锁

一句话总结：

“锁模块的目标，是让你从‘知道事务能保证正确’进一步走到‘知道数据库具体怎么在并发下强行维持正确’。”

## 三、这一模块的核心主线

你这一轮不要把锁学成散点，而要沿着下面这条线走：

并发修改同一批数据
-> 可能互相覆盖 / 读写冲突 / 条件范围不稳定
-> 数据库必须控制访问顺序
-> 锁用来限制并发下的访问冲突
-> 锁保证正确性，但会带来等待、阻塞、死锁、吞吐下降
-> 索引命中情况会影响锁定范围

如果这条线你顺了，后面很多名词就不会飘在空中。

## 四、建议拆成 6 个学习小节

### 第 1 节：为什么数据库需要锁

这一节先不讲类型，先回答根问题：

“如果没有锁，只靠事务，为什么还不够？”

建议重点学：
- 事务解决的是“业务动作整体边界”
- 但并发事务同时改同一份数据时，仍可能互相干扰
- 锁的本质：在关键时刻限制别人碰这份数据
- 锁不是为了让系统变慢，而是为了在并发下守住正确性

这一节学完要能说：
- 为什么事务不等于自动解决所有并发冲突
- 为什么数据库必须让某些操作排队

对应项目表达：
- 扣库存
- 修改订单状态
- 更新账户余额
- 幂等控制下的重复提交处理

### 第 2 节：共享锁、排他锁先分清

这一节先建立最基础的两种锁语义：
- 共享锁（S 锁）：我可以读，但不允许别人随便改
- 排他锁（X 锁）：我正在改，别人不能同时改，也不能做某些会破坏一致性的读取/处理

建议重点学：
- 为什么“读”有时也需要锁
- 为什么“写”天然更强势
- S 锁 / X 锁冲突关系先建立直觉，不用死背矩阵

这一节学完要能说：
- 锁不是只有写才有
- 数据库是在不同访问动作之间制定兼容规则

### 第 3 节：表锁、行锁，重点理解粒度

这一节开始进入“锁的范围”问题。

建议重点学：
- 表锁：一次锁一大片，简单但并发差
- 行锁：只锁住命中的记录，并发更好
- 锁粒度越大，控制越粗；粒度越小，并发越强，但管理更复杂

这一节最重要的不是背定义，而是知道：

“数据库并不是只有‘锁不锁’两个状态，它还在权衡：到底锁多大一片区域更合适。”

这一节学完要能说：
- 为什么行锁更适合高并发 OLTP 场景
- 为什么表锁简单但容易影响吞吐

### 第 4 节：为什么 InnoDB 里的锁经常和索引命中绑在一起

这是项目表达和面试追问里的重点。

建议重点学：
- InnoDB 行锁并不是你想象中“凭空精准锁某一行”
- 它通常依赖索引去定位和加锁
- 索引命中好，锁定范围更精确
- 索引没命中，可能扫描范围更大，锁影响也更大

这一节是连接“索引模块”和“锁模块”的桥。

这一节学完要能说：
- 为什么一个 SQL 是否命中索引，不只影响查询性能，也可能影响锁范围
- 为什么慢 SQL 有时还会放大并发问题

项目里非常适合这样表达：

“某些更新语句如果没有很好命中索引，不只是查得慢，还可能扩大扫描和锁定范围，导致并发下锁等待更明显。”

### 第 5 节：间隙锁、临键锁，先理解它们在防什么

这一节不要一开始背细枝末节，而是先理解：

“为什么只锁住已有行还不够？”

建议重点学：
- 有些问题不是别人改了现有行，而是别人插入了新行
- 范围查询 + 并发插入，会让结果集不稳定
- 间隙锁是在锁“记录之间的空档”
- 临键锁可以粗理解为“记录锁 + 间隙锁”的组合，用来在范围内同时防改和防插

这一节学完要能说：
- 为什么幻读不只是“多了一行”这么简单
- 为什么数据库有时要锁住一个范围，而不只是某一条已有记录

注意：
你这一阶段不用强迫自己细抠所有边界条件，先理解它们是在保护“范围稳定性”。

### 第 6 节：锁等待、死锁、项目怎么表达

这是项目并发表达的落点。

建议重点学：
- 为什么事务会等待别人的锁释放
- 为什么两个事务可能互相等，最后形成死锁
- 死锁不是数据库坏了，而是并发访问顺序冲突了
- 数据库通常会主动回滚其中一个事务来打破死锁
- 如何从业务顺序统一、索引优化、缩短事务、固定加锁顺序等角度降低风险

这一节学完要能说：
- 什么是锁等待
- 什么是死锁
- 项目里出现并发冲突时，你会从哪些方向排查

## 五、建议的学习顺序

这一模块建议按这个顺序推进：

1. 为什么数据库需要锁
2. 共享锁和排他锁
3. 表锁和行锁
4. 锁与索引命中的关系
5. 间隙锁与临键锁
6. 锁等待、死锁与项目表达

这个顺序的好处是：
- 先建立“锁为什么存在”
- 再建立“锁的基本语义”
- 再建立“锁的粒度”
- 再把它和你刚学过的索引连接起来
- 最后落到项目里的并发问题表达

## 六、这一模块建议重点关注的问题

你学习时，建议反复围绕这些问题思考：

- 如果没有锁，并发更新到底会出什么事？
- 锁是在保护“值”，还是保护“访问顺序”？
- 为什么写操作通常要更强的锁？
- 为什么行锁性能通常比表锁好？
- 为什么索引命中不好，会让锁问题更严重？
- 为什么有时候锁的目标不只是已有记录，还有“空档范围”？
- 为什么死锁并不神秘，本质上是互相等待形成环？
- 项目里如何减少锁竞争，而不是只会说“数据库有锁”？

## 七、这一模块学完后的输出目标

### 1. 面试表达层

你应该能比较顺地回答：
- MySQL 为什么需要锁？
- 共享锁和排他锁有什么区别？
- 行锁和表锁有什么区别？
- 间隙锁、临键锁是为了解决什么问题？
- 什么情况下会发生死锁？
- 索引和锁有什么关系？

### 2. 项目表达层

你应该能解释：
- 为什么库存扣减会有锁竞争
- 为什么订单状态更新需要考虑并发安全
- 为什么某条 update / select for update 在高并发下可能阻塞
- 为什么索引设计不合理会放大锁等待
- 为什么缩短事务、统一操作顺序能降低死锁概率

### 3. 知识结构层

你脑子里要形成这条关系：
- 事务定义业务边界
- 隔离级别定义允许互相影响的程度
- 锁在关键访问点上强行限制并发
- MVCC 主要缓解读写冲突
- 索引会影响锁定范围和性能表现

## 十、模块 3 的一句话版总结

模块 3：锁，核心不是背锁名，而是理解：

“数据库为了在并发下守住正确性，会在关键数据访问点限制事务之间的互相影响；而锁带来的等待、阻塞、死锁，以及索引对锁范围的影响，正是项目里并发问题表达的重点。”
