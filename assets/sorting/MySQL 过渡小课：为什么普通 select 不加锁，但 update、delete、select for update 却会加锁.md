# MySQL 过渡小课：为什么普通 select 不加锁，但 update、delete、select for update 却会加锁

## 一、先说结论：不是“读没有锁，写才有锁”这么简单

很多人刚学 MySQL 时，会先形成一个比较粗的印象：
- 读就是读
- 写才会加锁

这个印象有一点方向感，
但如果就停在这里，后面一遇到 `select ... for update` 就会马上混乱。

更准确的理解应该是：

- 普通 `select`，很多时候属于快照读，主要靠 MVCC，通常不加锁
- `select ... for update` / `select ... for share` 属于当前读，会加锁
- `update` / `delete` 属于真正修改数据的操作，也会加锁

所以问题不是“读还是写”四个字本身，
而是：

“你这次读取，到底只是想看一个快照，还是想读当前版本并控制并发顺序？”

## 二、为什么数据库不想让普通 select 也到处加锁

你先从业务体感想一件事。

如果一个系统里：
- 很多人在查商品
- 很多人在查用户列表
- 很多人在看订单详情

而每一次普通查询都要先加锁，
那整个系统会非常容易排队。

数据库当然不希望出现这种局面。

它更想做到的是：
- 查的人尽量能直接查
- 改的人再按规则控制顺序

所以 InnoDB 引入了 MVCC 这套思路。

你可以先把它理解成：

“普通读尽量不要靠锁硬排队，而是尽量去读一个一致性的历史版本。”

这个思路的目标不是让数据“完全没有并发问题”，
而是让“普通读”这件事尽可能轻一点、顺一点。

## 三、普通 select 为什么通常不加锁：因为它常常在做快照读

最常见的查询：

```sql
select * from user where id = 1;
```

这种普通 `select`，在 InnoDB 里通常会被理解成一种快照读。

什么叫快照读？
你可以先把它理解成：

“我这次读，不一定非要把当前正在被别人改的最新版本硬抢过来看；
我可以读一个对当前事务来说一致的版本。”

于是它背后主要靠的是 MVCC：
- 利用数据的多版本信息
- 让读操作读到一个一致性视图
- 尽量减少读写之间互相堵住

所以普通 `select` 的核心感觉是：

“我主要是来看的，不是来抢控制权的。”

也正因为如此，
它通常不会像写操作那样先去加锁把路堵住。

## 四、那为什么 select for update 又会加锁：因为它已经不是“只是看看”

来看这句：

```sql
select * from user where id = 1 for update;
```

这虽然长得还是 `select`，
但它的业务意图已经变了。

它通常不是单纯表示：
- 我想看看 id = 1 这行长什么样

而更像是在说：
- 我先把这行拿到手
- 而且我要锁住它
- 接下来可能马上要改它
- 这段时间别的人不要随便和我并发地改同一行

这时候数据库就不能再把它当成“普通参观式读取”了，
而要把它当成一种“带控制意图的读取”。

这类读，通常就叫当前读。

你可以把当前读理解成：

“我要读当前这个真实会参与并发控制的版本，而且我要让别的事务在某些操作上受到约束。”

所以它会加锁。

## 五、update / delete 为什么会加锁：因为它们不只是读到数据，而是要改动它

比如：

```sql
update user set balance = 1500 where id = 1;
```

或者：

```sql
delete from user where id = 1;
```

这类语句和普通 `select` 最大的不同是：

它们不是“把这一行看看”，
而是“我要真的动这一行”。

一旦你要动数据，数据库就必须处理这些并发问题：
- 两个人同时改同一行怎么办
- 一个事务正在改，另一个事务能不能也改
- 一个事务刚读完准备改，另一个事务是不是已经先改了
- 删除中的行，别人还能不能同时更新

如果没有锁来控制顺序，
这些动作很容易彼此打架。

所以你可以把锁先理解成：

“数据库在写入、当前读、锁定读这些关键访问点上，用来控制先后顺序的一种机制。”

这也和你前面建立的那个理解完全一致：
- MVCC 更像多版本查询 / 读取机制，偏向让读少排队
- 锁更像在写入、当前读、锁定读等关键访问点控制顺序

## 六、用一个最小场景感受：为什么普通 select 不需要锁，但 update 需要

假设用户表里有一条数据：

```text
id = 1, name = Alice, balance = 1000
```

现在有两个事务。

### 场景 1：事务 A 只是普通查询

事务 A：

```sql
select * from user where id = 1;
```

事务 B：

```sql
update user set balance = 1200 where id = 1;
```

如果普通查询也必须先强行加锁，
那事务 B 可能就得排队。

但数据库会觉得：
- A 只是看一眼
- 没必要为了“看一眼”把真正的修改挡住太久

于是普通查询通常通过 MVCC 去读一个合适的版本，
尽量不和写操作硬碰硬。

### 场景 2：事务 A 不是只看，而是想锁住后再改

事务 A：

```sql
select * from user where id = 1 for update;
```

事务 B：

```sql
update user set balance = 1200 where id = 1;
```

这时候 A 的意思已经不是“看一下”，
而是“这行我要先控制住，后续可能继续修改”。

那数据库就必须让这两边排出先后顺序，
否则同一行数据就可能被并发抢改。

所以这里就会涉及锁。

## 七、你可以把“读”拆成两类，就不容易混了

以后不要再笼统说“读加不加锁”，
而是先问：这是哪一种读？

### 1. 快照读

常见形态：

```sql
select * from user where id = 1;
```

特点：
- 普通查询
- 主要靠 MVCC
- 通常不加锁
- 更像“读一个一致性视图”

### 2. 当前读

常见形态：

```sql
select * from user where id = 1 for update;
select * from user where id = 1 for share;
```

特点：
- 读当前版本
- 带并发控制意图
- 会加锁
- 更像“我要读到可参与控制的那一版，并限制别人某些操作”

一旦你把“读”拆成这两类，
整个问题马上就顺了。

## 八、为什么说 update / delete 本质上也会先定位，再进入锁的世界

你之前已经学过：
- `update ... where ...`
- `delete ... where ...`

这里的 `where` 是先圈目标行。

但它们和普通 `select ... where ...` 的区别在于：

- 普通 `select ... where ...`：圈出来，看看
- `update/delete ... where ...`：圈出来，动它

一旦从“看看”变成“动它”，
数据库就必须考虑并发顺序，
所以锁就会介入。

也就是说：

`where` 负责“找哪批数据”，
锁负责“对这批数据的关键访问如何排队”。

这个连接点你如果建立起来，
后面学事务和锁就会特别顺。

## 九、一个你现在就可以记住的面试版表述

如果以后别人问你：

“为什么 MySQL 里普通 select 不加锁，但 update / delete / select for update 却会加锁？”

你可以这样答：

“因为在 InnoDB 里，不能简单理解成‘读没锁、写有锁’。更准确地说，普通 select 大多属于快照读，背后主要通过 MVCC 读取一致性视图，目标是让普通读尽量少阻塞；而 update、delete 以及 select for update 这类当前读 / 写操作，需要控制并发访问顺序，所以会涉及锁。也就是说，MVCC 更偏向解决普通读并发，锁更偏向控制写和当前读的顺序。”

这段话已经足够适合实习面试和项目表达了。

## 十、你现在先不用急着深挖的点

这一小课先不展开这些细节：
- 行锁、间隙锁、临键锁分别是什么
- 不同隔离级别下快照读的细节差异
- 当前读到底会加哪种锁
- 索引命中与锁范围的关系

因为你当前最重要的是先把主线理顺：

不是“MVCC 和锁二选一”，
而是：
- 普通读更偏 MVCC
- 当前读 / 写更偏锁控制

## 十一、这一小课你至少要记住的 5 句话

1. 不能简单说“读没有锁，写才有锁”。
2. 普通 `select` 大多是快照读，通常靠 MVCC，不强调加锁。
3. `select ... for update` / `for share` 属于当前读，会加锁。
4. `update` / `delete` 会真正修改数据，所以会和锁强相关。
5. MVCC 更偏向让普通读少排队，锁更偏向控制写和当前读的顺序。

## 十二、这节小课和后面语句学习怎么连起来

这节小课其实正好能帮你把两条线接起来：

1. 语句线
- `select`
- `where`
- `update`
- `delete`

2. 并发控制线
- 普通读
- 当前读
- MVCC
- 锁

所以你后面再看 `where`，不要只理解成“条件怎么写”，
还可以多加一层意识：

- 在 `select` 里，它是在圈我要看谁
- 在 `update/delete` 里，它不只是在圈我要动谁，还隐含着“数据库要如何控制这批数据的并发访问”

这会让你的 SQL 学习和事务 / 锁学习自然连上。
