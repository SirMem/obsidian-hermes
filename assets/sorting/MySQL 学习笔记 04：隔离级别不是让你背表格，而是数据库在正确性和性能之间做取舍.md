## 一、这一节不要先背四个英文名，先想数据库到底在取舍什么

你上一节已经知道，并发事务会制造很多问题：

- 脏读
- 不可重复读
- 幻读
- 丢失更新

那么一个非常自然的问题就来了：

“既然并发这么容易把数据搞乱，数据库为什么不直接把所有事务彻底隔开？”

答案是：
因为数据库不是只追求正确，还要追求性能、吞吐量、响应速度。

所以隔离级别的本质，不是一张让你背诵的表格，而是数据库在做一个永恒取舍：

“我愿意让事务之间隔离到什么程度，来换取多大的并发能力？”

这就是你这一节最应该先抓住的总逻辑。

## 二、什么叫“隔离”

事务里的隔离，可以先用一句非常朴素的话来理解：

“一个事务在执行时，别的事务对它造成影响的程度，要被控制住。”

也就是说，数据库要回答这样几个问题：

- 别人的未提交修改，你能不能看到？
- 你读过的数据，别人能不能中途改掉？
- 你按条件查出来的一批数据，别人能不能中途插入新行，让结果集变掉？
- 两个人同时更新同一份数据，数据库要不要强行帮你们排队？

所谓隔离级别，其实就是这些问题的不同答案。

所以不要把隔离级别理解成死知识点。
它本质上是一套“允许事务互相影响到什么程度”的规则设计。

## 三、为什么数据库不总是用最强隔离

这是最容易想偏的地方。

很多人会直觉觉得：

“既然隔离越强越安全，那直接都用最强的不就好了？”

理论上最强隔离确实最安全，但工程上会带来很大代价。

### 1. 隔离越强，通常冲突越多

如果数据库非常严格地禁止事务互相影响，那么：

- 更多事务要等待
- 更多事务会被阻塞
- 同时能顺畅执行的事务变少

### 2. 隔离越强，吞吐量往往越低

事务不能自由穿插，意味着数据库更像“排队串行处理”。
这样虽然稳定，但高并发下性能压力会变大。

### 3. 很多业务并不需要最强保护

不是所有业务都像银行转账那么敏感。

比如有些查询场景：
- 报表统计允许轻微时间差
- 后台列表页允许你前后两次看见的数据不完全一样
- 某些分析型查询并不要求绝对实时一致

这时如果为了极致隔离牺牲大量性能，反而不划算。

所以数据库真正做的不是“永远最强”，而是：

“针对不同正确性要求，提供不同强度的隔离方案。”

## 四、四种隔离级别，本质上是四档不同强度的保护

MySQL 常讲的四种事务隔离级别是：

1. Read Uncommitted
2. Read Committed
3. Repeatable Read
4. Serializable

你现在先不要背英文，先把它们想成四档：

- 第一档：隔离很弱
- 第二档：比第一档强一点
- 第三档：再强一点
- 第四档：最强

隔离越往后：
- 正确性保护通常越强
- 并发代价通常越高

所以它们不是四个孤立名词，而是一条“保护强度逐步上升”的刻度尺。

## 五、第一档：Read Uncommitted —— 几乎不怎么隔离

### 1. 它在做什么

Read Uncommitted 的意思可以粗略理解成：

“一个事务，连别的事务还没提交的数据都可能读得到。”

这意味着事务之间几乎没什么读隔离。

### 2. 会带来什么问题

这种级别下，最典型的问题就是：

- 脏读可能发生
- 不可重复读可能发生
- 幻读可能发生

因为连别人未提交的中间结果你都可能看到，那后面的各种不稳定现象当然更容易出现。

### 3. 为什么一般不推荐

因为它让一个事务可能基于“别人还没认账的数据”做判断。
这在多数业务里都太危险。

所以你可以把它理解成：

“并发自由度很高，但正确性保护太弱。”

## 六、第二档：Read Committed —— 至少不读别人的未提交结果

### 1. 它想解决什么

Read Committed 的核心想法是：

“别人没提交的数据，我不看；只有提交后的结果，我才认。”

这就先把最离谱的脏读挡掉了。

### 2. 它能防什么

最关键的一点：

- 能防脏读

因为只有别人提交后，你才会读到它的修改。

### 3. 它还防不住什么

但它仍然可能出现：

- 不可重复读
- 幻读

为什么？
因为虽然你不读未提交数据了，但别人一旦在你两次读取之间提交修改，你第二次读到的结果仍然可能变化。

比如：
- 你第一次查余额是 1000
- 别的事务提交后把它改成 800
- 你第二次再查，就看到 800

这就是不可重复读。

如果别人中途插入一条满足条件的新记录，你再按条件查询时，结果集也会变化，这就是幻读。

### 4. 它的直觉定位

你可以把 Read Committed 理解成：

“至少我只看正式认账后的数据，但我不保证你在同一事务里反复读取时看到的是同一个世界。”

这是一个很重要的理解。

## 七、第三档：Repeatable Read —— 尽量保证同一事务里反复读时世界稳定

### 1. 它想解决什么

Repeatable Read 的核心直觉是：

“同一个事务里，如果我已经读过一条数据，那么我后面再读，应该还能看到同一份结果，而不是老变。”

所以它重点想缓解的是：

- 不可重复读

### 2. 它能防什么

从入门学习角度，你可以先记：

- 能防脏读
- 能防不可重复读

而且在 MySQL 的 InnoDB 里，Repeatable Read 还配合 MVCC、间隙锁等机制，对幻读也做了很强处理。

这一点你现在先有印象就够，不用一下钻太深。

### 3. 它为什么比 Read Committed 更强

因为它不只是要求：

“别人的数据得先提交我才能看。”

还要求：

“我这个事务一旦建立起自己的读取视角，后面重复读取时不要轻易变。”

这就让事务内部的观察更稳定了。

### 4. 为什么它在 MySQL 里很重要

因为很多业务操作需要在一个事务里：

- 先读
- 再判断
- 再继续处理

如果你每次读到的世界都在变，业务逻辑会很难写。
所以 Repeatable Read 在工程上非常实用。

## 八、第四档：Serializable —— 把并发尽量压成接近串行执行

### 1. 它想实现什么

Serializable 可以粗略理解成：

“让事务执行效果尽量像一个一个排队串行执行。”

也就是说，虽然底层还是并发系统，但数据库会用非常强的控制，尽量避免各种并发异常。

### 2. 它的保护最强

从学习角度，你可以先理解为：

- 它对脏读、不重复读、幻读等问题保护最强
- 事务之间互相干扰最少

### 3. 代价也最大

因为它最强调顺序和隔离，所以通常意味着：

- 更多锁竞争
- 更多等待
- 更低并发
- 更差吞吐量

所以它不是“默认最好”，而是“最严格、但也最贵”。

### 4. 它适合什么直觉场景

适合那种你宁可慢，也不能容忍并发异常风险的场景。

但大多数通用业务系统不会默认上来就用它。

## 九、先别死背表格，先用一条主线串起来

你现在可以先这样理解四种隔离级别：

- Read Uncommitted：连未提交都可能看到，保护最弱
- Read Committed：只看已提交结果，但前后读取可能变
- Repeatable Read：同一事务内的读取更稳定
- Serializable：尽量把事务变得像串行执行，保护最强

这一条线比背表格重要得多。

因为它抓住了“级别为什么逐步增强”的因果逻辑。

## 十、为什么说隔离级别是在“正确性”和“性能”之间做取舍

这句话你必须真正理解，不然后面还是会把隔离级别看成定义题。

### 1. 如果更偏性能

数据库会倾向于允许更多事务并发穿插。
优点是：

- 系统更快
- 吞吐量更高
- 等待更少

但缺点是：

- 你更可能读到变化中的世界
- 更可能遇到前后读不一致
- 更可能遇到结果集变化

### 2. 如果更偏正确性

数据库会更严格限制事务互相干扰。
优点是：

- 事务内部观察更稳定
- 异常更少
- 更容易保证业务逻辑一致

但缺点是：

- 等待更多
- 冲突更多
- 高并发性能更差

所以数据库不是在追求一个绝对最优答案，而是在问：

“这个业务，值得付出多大性能代价，来换取多强的一致性保护？”

## 十一、你应该怎么选，不要先问‘哪个最好’，先问‘业务怕什么’

这是工程思维里非常重要的一步。

选隔离级别时，不要先问：

“哪个级别最高级？”

而要先问：

“这个业务最怕哪类错误？”

例如：

### 1. 如果业务最怕读到未确认数据
那至少不能太弱，脏读必须挡住。

### 2. 如果业务最怕同一事务前后读到不同结果
那你会更在意不可重复读问题。

### 3. 如果业务最怕条件查询结果中途变化
那你会更在意幻读和范围控制。

### 4. 如果业务最怕并发更新把结果算错
那你会更关注更新冲突控制、锁、MVCC、悲观锁/乐观锁等策略。

也就是说：

隔离级别只是并发控制大图中的一部分。
真正工程落地时，你还要配合：

- 锁机制
- 索引设计
- SQL 写法
- 事务长度控制
- 乐观锁 / 悲观锁

所以别把隔离级别神化成“一个设置解决一切”。

## 十二、为什么很多资料把 MySQL 默认隔离级别讲得很重要

因为默认值往往反映了数据库在通用业务场景下做出的工程平衡。

对于 MySQL 的 InnoDB 来说，默认隔离级别通常是 Repeatable Read。

这件事值得你先建立一个直觉：

“在大多数通用 OLTP 业务中，MySQL 认为更稳定的事务内读取视角，是一个很划算的默认选择。”

为什么不是更弱？
因为太弱会让很多业务容易踩坑。

为什么不是最强？
因为最强代价太大，不适合大多数高并发系统作为默认。

所以默认值本身，就是一种工程取舍结果。

## 十三、你现在先形成的不是定义记忆，而是一张取舍地图

到了这里，你最该得到的不是一张硬背表，而是一张脑内地图：

- 隔离级别越弱，并发越自由，但越容易出现异常
- 隔离级别越强，结果越稳定，但代价越高
- 不同级别的存在，是为了给不同业务提供不同强度的保护
- MySQL 不只是“有没有事务”，而是在设计“事务之间如何共存”

只要这张地图形成了，后面你再去学：

- 哪个级别防哪些问题
- InnoDB 的 MVCC 怎么帮 Repeatable Read 工作
- 为什么还需要 next-key lock

都会顺很多。

现在知识链条已经更完整了：

### 第 01 节：

因为它是核心业务事实存储中心。

### 第 02 节：
为什么需要事务？
因为业务动作天然是多步骤联动的，需要整体成功或失败。

### 第 03 节：
并发到底会制造哪些问题？
会出现脏读、不可重复读、幻读、丢失更新等异常。

数据库怎么面对这些问题？
通过不同强度的隔离级别，在正确性和性能之间做取舍。

到这里你对事务的理解，已经不是“一个名词”，而是开始形成系统图景了。

## 十五、这一节最核心的认知总结

请先记住这几句话：

- 隔离级别不是死表格，而是数据库设计出来的并发取舍方案
- 本质问题不是“会不会背四个名字”，而是“数据库允许事务互相影响到什么程度”
- 隔离越弱，并发通常越高，但异常风险越大
- 隔离越强，事务观察越稳定，但等待和冲突通常更多
- Read Uncommitted 最弱，Serializable 最强
- Read Committed 强调“不读未提交”
- Repeatable Read 强调“同一事务内读取更稳定”
- MySQL 默认常用 Repeatable Read，本身就是一种工程平衡选择

## 十六、你现在应该能回答的 5 个问题

学完这一节，你至少要能用自己的话回答：

1. 隔离级别本质上是在解决什么问题？
		是在解决多个事务并发执行时，事务之间允许互相影响到什么程度的问题
2. 为什么数据库不直接全部使用最强隔离级别？
		因为最强隔离虽然最安全，但是性能代价高，会带来更多等待、阻塞和更低吞吐量
3. Read Committed 和 Repeatable Read 的直觉差别是什么？
		Read Committed 只保证我不读到别人未提交的数据，而 Repeatable Read 更进一步，强调同一事务里反复读取时结果尽量稳定
4. 为什么说 MySQL 默认的 Repeatable Read 是一种工程取舍？
		因为它既不像更弱隔离那样容易踩并发坑，也不像 Serializable 那样代价很大，所以是一个通用业务里比较平衡的默认选择
5. 学隔离级别时，为什么不能只背“它防哪些问题”？
		因为真正要理解的是数据库在正确性和性能之间怎么做权衡，以及业务到底怕什么错误

如果这五个问题你能回答顺，后面再学 MVCC、快照读、当前读、间隙锁，就不会只是堆术语。
