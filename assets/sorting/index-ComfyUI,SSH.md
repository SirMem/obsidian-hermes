## 模块一：ComfyUI 核心概念解析

### 1. 什么是 ComfyUI？
* **本质**：AI 领域的“可视化编程工具”与“乐高积木”。
* **工作机制**：基于**节点（Node）**和**连线（Workflow）**。用户无需编写代码，通过拖拽输入模块（图片/文本）、处理模块（TTS/SadTalker等大模型）和输出模块（视频/音频），即可构建复杂的自动化生成流水线。
* **最大优势（图片即源码）**：将生成的图片或 JSON 格式的工作流文件拖入浏览器，即可瞬间还原原作者的所有节点配置，极大地降低了“抄作业”的门槛。

### 2. 为什么整合包体积巨大（通常 150GB+）？
开箱即用的整合包（如 69GB 的压缩包）之所以大，是因为它包含了整个“AI 超级工厂”：
1. **AI 大脑（Checkpoints）**：占据 60% 以上空间，如 SDXL、Flux 或各类数字人底模。
2. **运行环境**：内置了完整的 Python 环境、PyTorch 以及庞大的 CUDA 深度学习运算库。
3. **辅助模型与插件**：面部修复（GFPGAN）、各类控制网（ControlNet）及社区开发的数百个自定义节点。
> **硬件铁律**：必须解压在纯英文路径的 SSD 固态硬盘中，否则会导致严重的 I/O 瓶颈。

### 3. 商业化潜力：作为 HTTP 后端 API
市面上大多数 AI 短视频 App 的核心引擎就是 ComfyUI，它原生支持作为后端服务运行：
* **导出模板**：在开发者模式下，可将连线好的工作流导出为 `API Format (JSON)`。
* **动态传参**：通过 Java/Python 等后端代码，修改 JSON 中的文本或图片路径参数。
* **异步调用**：通过 HTTP POST 请求发送至 `8188`（或自定义的 `6006`）端口，并通过 WebSocket 监听视频生成进度。

### 4. 算力策略：为什么商业应用都在“租”显卡？
* **潮汐效应**：App 流量存在波峰波谷。云算力（如 AutoDL、阿里云）支持弹性扩缩容，按秒/小时计费，避免了购买顶级显卡（如 A100/4090）在夜间的闲置浪费。
* **OpEx vs CapEx**：将前期的重资产投入（几千万买机房）转化为轻资产的运营支出（租云服务器），是 AI 创业的标配。

## 模块二：SSH 隧道 (内网穿透/端口映射) 实战

### 1. 核心应用场景
当我们租用 AutoDL 等云算力平台时，为了安全和成本，服务器通常**没有独立的公网 IP**。为了能在本地电脑的浏览器中访问云端服务器内部运行的 Web UI（例如运行在 `6006` 端口的 ComfyUI），我们需要建立一条 **SSH 隧道（SSH Tunnel）**。

### 2. 专属命令解析
**标准操作命令：**
```bash
ssh -CNgv -L 6006:127.0.0.1:6006 -L 6008:127.0.0.1:6008 root@connect.westc.seetacloud.com -p 12536    

```

这部分是云端开发中非常常见的一个“安全陷阱”。为了方便你存入 Obsidian，我将这段解决过程重构为结构化的技术笔记。

## 笔记：解决 SSH 远程主机标识变更 (Remote Host Identification Changed)

### 1. 现象描述

执行 SSH 命令时，终端出现大量 `@` 符号警告，并提示：

`WARNING: REMOTE HOST IDENTIFICATION HAS CHANGED!`

`Host key verification failed.`

### 2. 核心原理

- **SSH 指纹机制**：当你第一次连接某个 IP 和端口时，本地电脑会在 `~/.ssh/known_hosts` 文件中记录该服务器的“指纹”。
    
- **云算力特性**：AutoDL 的公网地址（如 `connect.westc.seetacloud.com`）和端口（如 `12536`）是**循环回收**给不同用户使用的。
    
- **冲突原因**：你的电脑发现当前地址的服务器指纹与记录中的旧指纹不一致，触发安全保护机制，主动切断连接以防止“中间人攻击”。
    

### 3. 解决方案 (SOP)

#### 方法一：命令行一键清除（推荐）

在本地终端执行以下命令，强制删除该地址对应的旧指纹记录：

Bash

```
ssh-keygen -R "[connect.westc.seetacloud.com]:12536"
```

_注：方括号和引号需保留，以匹配特定端口的记录。_

#### 方法二：手动编辑配置文件

若方法一无效，可根据报错信息手动删除：

1. **定位文件**：打开 `C:\Users\用户名\.ssh\known_hosts`。
    
2. **查找行号**：报错信息通常会提示 `Offending ECDSA key in ...:6`（数字即为行号）。
    
3. **删除并保存**：用记事本打开该文件，删除对应行，保存退出。
    

### 4. 重新连接

重新执行 SSH 隧道命令，系统会提示：

`Are you sure you want to continue connecting (yes/no/[fingerprint])?`

- **操作**：输入 **`yes`** 并回车，接受新机器指纹。
    
- **结果**：输入密码后，隧道即可成功打通。

### 1. CLIP 文本编码器 —— 【精通双语的“翻译官”】

- **痛点**：AI 其实是个“外星人”，它根本看不懂人类语言里的“赛博朋克”、“可爱小猫”是什么意思，它只懂数学。
    
- **作用**：CLIP 就是你的翻译官。你把提示词输给它，它会把这些词汇翻译成一长串的**“数学向量（特征数字）”**，这样后面的画师才能听懂你想要什么。
    
- **通俗理解**：你给出的剧本，经过翻译官变成了一份极其详尽的“美术需求文档”。
    

### 2. UNet (扩散模型) —— 【核心的“灵魂画师”】

- **痛点**：这是整个团队里最累、最占显卡算力的打工人。
    
- **作用**：UNet 拿到“需求文档”后，会在一张满是“雪花噪点”的空白画布上，一点一点地把图像的轮廓、光影、细节给“雕刻”出来。
    
- **注意**：UNet 并不是在真正的像素上画画，为了节省算力，它是在一个极其压缩的“潜空间（Latent Space）”里打草稿。这时的画，人类肉眼看就是一堆马赛克，只有机器懂。
    

### 3. KSampler (K采样器) —— 【控制节奏的“导演”】

- **痛点**：主画师 UNet 虽然会画，但它不知道该画多快，也不知道该多听你的话。
    
- **作用**：KSampler 是整个画室的控制台。它规定了画师要画多少步（**Steps**，步数越高细节越多但越慢），也规定了画师要多大程度上听从提示词（**CFG Scale**，数值越高越刻板，越低越放飞自我）。
    
- **通俗理解**：导演拿着大喇叭喊：“准备！一共画 20 笔（20步），每画一笔给我看一眼，继续去噪！”
    

### 4. VAE (变分自编码器) —— 【最终的“冲洗暗房”】

- **痛点**：刚才说了，UNet 画完的图是一堆“压缩草稿（Latent）”，这玩意儿人类的显示器是显示不出来的。
    
- **作用**：VAE 就像是老式胶片摄影里的暗房。它把这些机器能看懂的“压缩包数据”，解压缩、解码成人类肉眼可见的 RGB 像素图片（就是你最后看到的 PNG 或 JPG）。
    
- **通俗理解**：如果最后出图发现画面灰蒙蒙的、颜色非常暗淡，通常就是 VAE 这个环节忘了加，或者用错了。

List, Set, Map的区别
题一
核心方式在于它们组织数据的方式不一样，List用于组装有顺序的数据，Set用于组装去重的数据，Map通过键值对查找数据。List可以通常用于保存用户列表，Set可以用于权限去重，Map可以通过id直接查询数据

题二
ArrayList 和 LinkedList 有什么区别？
底层结构，CRUD, 应用频率不同
ArrayList的底层是动态数组，可以通过index直接定位，LinkedList是双向链表，查询通过遍历链表定位

ArrayList 插入是O(n), 查询是O(1) LinkedList 删除是O(1)，但其查询是O(n) 因此删除插入也是O(n)，大家更常用ArrayList 因为大多数场景是读多写少


 HashMap 为什么查找通常很快
 因为HashMap是先定位桶，然后在桶中查找数据
 Hash把key映射为index，然后将得到的hash转为桶index
 冲突就是不同的key可能会映射到相同的桶，这时候才在桶内查找
 hash查找通常很快是因为hash通常会把大量数据分散的比较开

Hash为什么慢
HashMap 慢，通常是因为分散失败，很多数据挤进少数桶里。
HashMap 变慢的本质原因，是冲突变多，导致同一个桶里元素太多。常见诱因有：hashCode 设计差、容量太小频繁扩容、数据过于密集。

什么叫“多态”？
多态 = 同一套调用方式，因对象不同而表现不同。
它在真实开发中的作用，是让代码解耦、易扩展、少写大量 if-else。

接口和抽象类，你怎么区分使用场景？
 接口：表达能力，强调规范，耦合低
 抽象类：表达归属，强调复用，耦合更强
 
1. 我是在描述“会干什么”，还是“它是什么”？

• 会干什么 → 接口
• 它是什么 → 抽象类

2. 我需不需要共享字段和通用实现？

• 需要 → 抽象类更合适
• 不需要，只要统一规范 → 接口

3. 我更想要灵活扩展，还是固定继承结构？

• 灵活扩展、低耦合 → 接口
• 稳定父子关系、可复用代码 → 抽象类

HashMap为什么不是线程安全
HashMap 不是线程安全的，因为并发 put/get 时没有同步保护。多个线程同时 put 可能造成数据覆盖、丢失；如果碰上扩容，还可能导致内部结构混乱；并发 get 也可能读到不一致的数据。



ArrayList 扩容大概是怎么做的？
它为什么不是每次只扩 1 个位置？
ArrayList 扩容就是新建更大数组，再拷贝旧数据。
不每次只扩 1，是因为那样会频繁复制数组，性能太差，所以通常按比例扩容。

重载（overload）和重写（override）有什么区别？
重载
同名方法，换“入口”
• 改的是参数形式
• 解决的是调用便利性
重写
同名方法，换“做法”
• 改的是方法实现
• 解决的是子类行为差异

你会优先用接口还是抽象类？
为什么？
如果两者结合，你会怎么设计？

接口,因为“邮件发送 / 短信发送 / 站内信发送”这件事，本质上更像是在表达一种 发送能力，而不是一种很强的“归属父类关系”。如果有大量公共逻辑，再加抽象类，形成“接口定规范，抽象类做复用，子类做实现”的结构。

反射 = 运行时不仅能“看”类的信息，还能“动”对象。
它的价值是让程序可以动态创建对象、调用方法、操作字段，常用于框架底层。

Java 是强类型语言，适合在编译期把类型写清楚、校验清楚；

强类型解决“确定性”，反射解决“动态性”。

───

1. 强类型语言已经很好了，为什么还要反射？

因为普通写法有一个前提：

你在写代码时，就已经知道要操作哪个类、哪个方法、哪个字段。

比如：

User user = new User();
user.login();

这很清晰、很安全、很高效。
但它有个限制：

• User 是提前写死的
• login() 也是提前写死的

如果运行时才知道具体要处理哪个类，
这种普通写法就不够灵活了。

───

2. 反射到底解决了什么普通写法不好解决的问题？

最本质的问题就是：

“编译时不知道，运行时才知道。”

这类问题，普通强类型写法不擅长。

───

场景 1：框架不知道你的业务类长什么样

比如 Spring。

Spring 框架在设计时，不可能提前知道你项目里会有：

• UserService
• OrderController
• LoginFilter

但它又要做到：

• 扫描这些类
• 创建对象
• 注入依赖
• 调用方法

问题来了：

框架作者在写 Spring 源码时，
根本不知道未来用户会定义哪些类。

这时候如果不用反射，就很难做到“通用框架支持任意业务类”。

所以反射解决的是：

让框架能操作“事先未知的类”。

场景 2：根据配置或字符串决定加载哪个类

比如你配置文件里写：

3. 强类型和反射不是对立关系

这个特别重要。

很多人会误以为：

• Java 强类型
• 反射很动态
• 所以它们是冲突的

其实不是。

更准确地说：

• 强类型 是 Java 的主流工作方式
• 反射 是 Java 为少数“动态场景”准备的补充能力

平时业务开发，当然优先直接写：

User user = new User();
user.login();

因为它：

• 类型安全
• 性能更好
• 可读性更强

但当你在写：

• 框架
• 通用组件
• 插件系统
•

• 自动映射工具

你就会发现：
普通写法太死了，
这时就必须要有反射。

4. 反射到底填补了什么空白？

可以浓缩成一句：

反射填补了 Java 在“运行时动态决定类型和行为”上的能力空白。

因为强类型语言擅长的是：

• 编译期确定类型
• 编译期检查调用是否合法

但它不擅长：

• 运行时根据字符串决定类
• 运行时扫描并操作未知类
• 运行时自动装配对象
• 运行时通用处理任意对象

而这些，恰恰是框架和底层组件非常需要的。

───

5. 用一句更通俗的话说

普通写法像是：

“我提前知道要找谁，所以我直接叫名字。”

反射像是：

“我现在还不知道对方是谁，但运行时我可以拿到名单，再决定找谁、怎么调用。”

所以反射不是为了取代普通写法，
而是为了处理那些“不能提前写死”的场景。
