## 一、这节课学什么

上一节你已经先把 Redis 的两个总问题理顺了：
1. Redis 在系统里是干什么的
2. Redis 为什么快

这一节我们继续往前走，但依然不是按“背命令”的方式学，而是按“面试能不能说清、项目能不能用上”的方式学。

这节课的核心目标是：

把 Redis 常见数据结构整理成一条稳定主线：
“这个结构有什么特点 -> 为什么适合这个场景 -> 项目里一般怎么用 -> 面试里怎么说”

你现在最需要的，不是把所有 API 都背下来，而是建立映射关系。

也就是你看到一个业务需求时，脑子里能自然冒出：
- 这是缓存对象，可能想到 String / Hash
- 这是去重，可能想到 Set
- 这是排行榜，应该想到 ZSet
- 这是按顺序处理，可能想到 List

如果这一层建立起来，后面讲缓存问题、项目实战时就会顺很多。

---

## 二、先别急着背结构，先想一个问题：Redis 为什么强调“数据结构”

很多人学 Redis 时会觉得：
“Redis 不就是 key-value 吗？”

这句话只对一半。

Redis 确实是 key-value 存储，但它真正特别有价值的地方在于：

它不是只有一种 value 形态，而是给你提供了多种适合业务场景的内存数据结构。

这意味着什么？

意味着很多业务需求，你不用自己从零设计复杂存储逻辑，Redis 已经给了你一个更顺手的结构：
- 简单值 -> String
- 对象字段 -> Hash
- 有顺序列表 -> List
- 唯一集合 -> Set
- 带分数排序 -> ZSet

所以你可以把 Redis 理解成：

它不只是“把数据放进内存”，而是“用合适的内存结构去承接高频业务操作”。

这也是 Redis 和“普通缓存容器”不太一样的地方。

---

## 三、这一节你最该建立的认知：结构不是拿来背的，是拿来选型的

面试官问你 Redis 数据结构，通常不只是想听你背：
- String
- Hash
- List
- Set
- ZSet

他真正想看的是：

你能不能把“结构特点”和“业务场景”连起来。

所以这一节你要学会的不是“有哪些”，而是：

1. 这个结构最核心的特征是什么
2. 它适合什么业务动作
3. 为什么它比其他结构更合适
4. 如果放到项目里，你能举出具体例子吗

下面我们就按这个逻辑讲。

---

## 四、String：最常用，也是你最不能轻视的结构

### 1. String 是什么
String 是 Redis 里最基础、最常见的类型。

你可以把它先粗略理解成：
key 对应一个值，这个值可以是：
- 普通字符串
- 数字
- JSON 序列化后的对象
- token
- 验证码
- 某个状态标记

也就是说，虽然名字叫 String，但它在业务里承载的内容非常广。

### 2. 为什么 String 这么常用
因为大量业务场景，本质上并不需要复杂结构。

很多时候你只是想做这几件事：
- 缓存一个对象
- 存一个验证码
- 存一个登录 token
- 记录一个是否已处理的标记
- 做一个简单计数器

这些需求都可以直接用 String 解决。

所以你要先建立一个认知：

Redis 的很多业务使用，表面上在说“用了 Redis”，本质上很多最终落地都是 String。

### 3. String 最典型的业务场景

#### 场景 1：缓存热点数据
比如：
- 商品详情缓存
- 用户主页信息缓存
- 首页推荐结果缓存
- 配置项缓存

常见做法是：
把查询结果序列化后，直接作为 value 存进去。

你可以这样理解：

String 最适合“整个结果拿出来就用”的场景。

#### 场景 2：验证码 / token / 登录状态
比如：
- sms:code:138xxxx -> 483921
- login:token:xxx -> 用户 ID 或用户会话信息

这类场景特点是：
- 值本身简单
- 生命周期短
- 通常要配过期时间

这和 Redis 的使用方式非常契合。

#### 场景 3：计数器
比如：
- 视频播放量
- 页面访问次数
- 点赞数
- 某接口调用次数

String 很适合做简单数值自增。

这类场景你要先记住一个感觉：

如果业务本质上是“某个值不断 +1 / -1”，那 String 往往就是最直接的选择。

#### 场景 4：分布式锁的简单 key
比如：
- lock:order:create:123
- lock:user:submit:456

本质上是通过一个简单 key 表示“这个资源当前有没有被占住”。

### 4. String 的优点和使用边界

优点：
- 使用最简单
- 适用范围广
- 很多缓存场景直接可用
- 可以存数字，适合计数
- 很适合和过期时间结合使用

边界：
- 如果你要频繁只改对象里的某个字段，用整个 String 覆盖更新可能不够灵活
- 如果你要天然支持去重、排序、集合运算，那 String 就不是最优结构

### 5. 面试里怎么说 String
你可以这样说：

String 是 Redis 里最常用的数据结构，很多缓存场景最终都是基于 String 实现的，比如缓存商品详情、用户信息、验证码、token、计数器等。它的优点是简单直接、适用范围广，尤其适合缓存整个对象结果或者存简单状态值。

---

## 五、Hash：当你想把“对象字段”拆开管理时，很适合它

### 1. Hash 是什么
Hash 可以理解成：
一个大 key 下面挂多个 field-value。

你可以把它类比成一个对象：
- key = user:1001
- field = name, age, city, avatar
- value = 对应字段值

所以 Hash 很适合表示“一个对象的多个属性”。

### 2. Hash 和 String 的区别，先用一句话记住

String 更像“整个对象一起存进去”；
Hash 更像“把对象拆成多个字段来存”。

这个区分非常重要。

### 3. Hash 最典型的业务场景

#### 场景 1：用户信息缓存
比如用户资料：
- nickname
- avatar
- gender
- city

如果你希望：
- 有时只读头像
- 有时只改昵称
- 有时只查某几个字段

那 Hash 就比整个对象序列化成 String 更灵活。

#### 场景 2：配置项缓存
比如系统配置：
- maxLoginRetry
- smsSwitch
- couponExpireDays

这些东西天然就是“一个对象下的多个字段”。

#### 场景 3：商品对象 / 店铺对象的字段缓存
如果对象字段比较固定，而且经常只更新局部字段，Hash 会很顺手。

### 4. 为什么不是所有对象都一定用 Hash
这点很重要。

很多人学完 Hash 之后，会产生一种误区：
“只要是对象，就应该用 Hash。”

不一定。

如果你的场景是：
- 读的时候总是整个对象一起取
- 写的时候也是整个对象整体刷新
- 对局部字段更新需求不强

那直接把整个对象序列化后放进 String，也完全可以，而且经常更简单。

所以你不要死背成：
对象 -> 一律 Hash

而是应该这样判断：
- 整体读写多 -> String 也很合适
- 按字段读写多 -> Hash 更合适

### 5. Hash 的优点和边界

优点：
- 适合存对象型数据
- 支持按字段读写
- 局部更新更灵活
- 比起每次整体覆盖，对某些场景更自然

边界：
- 如果场景根本不需要字段级操作，Hash 的价值就没那么明显
- 如果需求是排序、判重、队列，这就不是 Hash 的强项

### 6. 面试里怎么说 Hash
你可以这样表达：

Hash 适合存对象的多个字段，比如用户信息、配置项、商品属性等。相比把整个对象序列化成一个 String，Hash 更适合按字段读取和局部更新，所以在对象型缓存场景里比较常见。

---

## 六、List：核心不是“列表”，而是“有顺序、可头尾操作”

### 1. List 是什么
List 可以先理解成一个有顺序的元素序列。

和前面几个结构相比，它最重要的特点不是“能放多个元素”，而是：
- 元素有先后顺序
- 可以从头部或尾部插入、弹出

### 2. 你要先建立一个场景感
什么时候你会想到 List？

通常是当业务要求：
- 数据有顺序
- 你关心先来后到
- 经常做头尾插入、头尾消费

这时 List 才有意义。

### 3. List 的典型业务场景

#### 场景 1：简单消息队列 / 待处理队列
比如：
- 异步处理任务
- 简单订单通知队列
- 某些低复杂度的消费场景

但这里你一定要知道表达边界：

Redis List 可以做简单队列，但如果是复杂、可靠性要求高的消息系统，通常还是会交给 Kafka、RocketMQ、RabbitMQ 这类专业 MQ。

这是很重要的面试边界。

#### 场景 2：最新动态 / 最新评论列表
比如：
- 用户最近浏览记录
- 最新 20 条评论
- 最近操作记录

这类场景特点是：
- 有先后顺序
- 经常只关心最近一段

List 很适合这种“时间顺序感很强”的数据。

#### 场景 3：按顺序处理的数据流
比如一些简单任务流，先入先出处理。

### 4. List 的优点和边界

优点：
- 适合顺序型数据
- 适合头尾插入和消费
- 比普通 String / Hash 更贴合“队列”语义

边界：
- 如果你需要判重，List 不适合
- 如果你需要按分数排序，List 不适合
- 如果你要复杂可靠消息投递，也不能只靠 List 来扛所有场景

### 5. 面试里怎么说 List
你可以这样说：

List 适合有顺序的数据场景，比如简单消息队列、最新记录列表、按先后顺序处理的数据等。它的重点不是“能存多个值”，而是支持按顺序组织数据，并且适合头尾操作。

---

## 七、Set：你一看到“去重 / 是否存在”，就应该想到它

### 1. Set 是什么
Set 是无序集合，但元素唯一。

你可以先记住两个关键词：
- 无序
- 不重复

其中真正最重要的是“不重复”。

### 2. Set 最核心的业务价值是什么

判重。

很多业务需求，本质上根本不是“存一堆值”，而是：
- 某个用户点过赞没有
- 某篇文章被哪些用户收藏过
- 某个标签集合里是否已经有这个元素
- 某批数据有没有重复

只要你看到“唯一性”或者“是否存在”，Set 的存在感就会变得很强。

### 3. Set 的典型业务场景

#### 场景 1：点赞、收藏、关注判重
比如：
- article:like:1001 -> 用户集合
- user:follow:2001 -> 已关注对象集合

你要判断：
- 某用户有没有点过赞
- 有没有关注过某账号

Set 非常适合。

#### 场景 2：标签集合 / 权限集合
比如某用户有哪些标签、某角色有哪些权限。

#### 场景 3：共同好友 / 共同关注
集合类问题天然适合 Set。

### 4. 为什么 Set 适合这类业务
因为很多业务真正想解决的是：
- 去重
- 成员判断
- 集合关系处理

而不是复杂对象缓存。

所以你要逐渐建立这种判断力：

如果问题的核心是“这个元素在不在”，那 Set 往往比 List 更合适。

### 5. Set 的优点和边界

优点：
- 天然去重
- 适合成员判断
- 很适合集合类业务语义

边界：
- 不适合表达严格顺序
- 不适合做排行榜
- 如果你还需要分值排序，就要考虑 ZSet

### 6. 面试里怎么说 Set
你可以这样表达：

Set 最大特点是元素唯一，所以非常适合做去重和成员判断。像点赞判重、关注关系、标签集合、共同好友这类场景，用 Set 会比较自然。

---

## 八、ZSet：只要你看到“排名 / 分数 / 排序”，优先想到它

### 1. ZSet 是什么
ZSet 可以理解成：

“带分数的有序集合”。

它有两个核心特征：
- 元素本身具有唯一性
- 每个元素都带一个 score，并且可以按 score 排序

这是它和 Set 的关键区别。

### 2. 为什么 ZSet 很重要
因为很多业务不是单纯去重，而是：
- 既要唯一
- 又要排序

这类场景用普通 Set 很难优雅表达，但 ZSet 天然就适合。

### 3. ZSet 的典型业务场景

#### 场景 1：排行榜
这是最经典、最必会的场景。

比如：
- 游戏积分榜
- 用户活跃榜
- 商品热度榜
- 点赞排行

为什么排行榜适合 ZSet？
因为排行榜本质就是：
- 有一个对象
- 对象有一个分数
- 你要按分数排序

这和 ZSet 的结构天然匹配。

#### 场景 2：按权重 / 分值排序的数据集合
比如按热度、积分、得分、优先级排序。

#### 场景 3：延迟队列的简单实现思路
比如用 score 表示时间戳，到时间后把应该执行的任务取出来。

这里你不用现在学太深，但要知道：
ZSet 不只用于排行榜，它还适合所有“带排序权重”的场景。

### 4. ZSet 和 Set 的区别，必须会说
这是高频追问。

你可以直接这样说：

Set 只有“元素唯一”这一层；
ZSet 不仅元素唯一，还为每个元素引入了 score，所以可以按分值排序。

### 5. ZSet 的优点和边界

优点：
- 既能保证唯一性，又能排序
- 非常适合排行榜类场景
- 很适合“分数驱动”的业务语义

边界：
- 如果业务只是单纯去重，不需要排序，用 Set 可能更直接
- 如果只是简单缓存对象，也没必要上来就用 ZSet

### 6. 面试里怎么说 ZSet
你可以这样表达：

ZSet 是带分数的有序集合，既保证元素唯一，又能按 score 排序，所以特别适合排行榜、积分榜、热度排序、按优先级处理等场景。面试里最经典的例子就是排行榜。

## 九、把五种核心结构放到一张脑图里理解

你可以先把它们压缩成下面这组映射：

- String：简单值、整个对象缓存、token、验证码、计数器
- Hash：对象字段缓存、按字段读写
- List：有顺序的数据、简单队列、最新记录
- Set：去重、成员判断、标签/关系集合
- ZSet：排序 + 分值、排行榜、热度榜、优先级集合

你会发现：

Redis 数据结构并不是杂乱的 5 个名词，而是在回答 5 类常见业务问题：
- 我只是想存个值 -> String
- 我想存对象字段 -> Hash
- 我关心顺序 -> List
- 我关心唯一性 -> Set
- 我既关心唯一性又关心排序 -> ZSet

如果你把这张映射图记住，很多面试题都会变简单。

## 十、不要机械套结构，选型时你应该怎么想

这部分非常重要。

真实业务里，很多需求不是“标准题”，而是混合型场景。

所以你不能只会背“排行榜用 ZSet”，还要学会判断。

### 一个最实用的选型顺序
当你看到一个业务需求时，可以按下面顺序问自己：

#### 第一步：我最核心要解决的问题是什么？
- 是简单缓存？
- 是对象字段管理？
- 是顺序处理？
- 是判重？
- 是排序？

#### 第二步：这个场景最重要的特征是什么？
- 要不要唯一性？
- 要不要顺序？
- 要不要按分值排序？
- 要不要局部字段更新？
- 值是不是本来就很简单？

#### 第三步：有没有更简单的结构已经够用了？
比如：
- 只是缓存一个用户对象，String 可能就够了
- 只是判断有没有点过赞，Set 比 List 自然
- 只是按字段改头像昵称，Hash 比 String 灵活

真正成熟的表达不是：
“我知道很多结构。”

而是：
“我能根据业务特点选一个最合适、最不别扭的结构。”

## 十一、面试里最容易被问的几组对比题

### 1. String 和 Hash 怎么选
你可以这样说：

如果是把整个对象结果直接缓存下来，而且通常也是整体读取，那么 String 就很合适；如果一个对象有多个字段，而且经常需要按字段读取或局部更新，那 Hash 会更灵活。

### 2. List 和 Set 怎么选
你可以这样说：

如果业务重点是顺序，比如最新记录、简单队列，那更适合 List；如果重点是去重和判断元素是否存在，那更适合 Set。

### 3. Set 和 ZSet 怎么选
你可以这样说：

如果只需要唯一性和成员判断，用 Set 就够了；如果除了唯一性，还要按照某个分值排序，比如排行榜、热度榜，那就更适合 ZSet。

### 4. 为什么排行榜通常用 ZSet
你可以这样说：

因为排行榜的本质是对象和分数的对应关系，并且要按分数排序，而 ZSet 正好支持元素唯一和 score 排序，所以是很自然的结构选择。

## 十二、把“数据结构与业务场景”整理成 30 秒面试回答

### 30 秒基础版
Redis 常见数据结构里，最常用的是 String、Hash、List、Set 和 ZSet。String 适合缓存简单值、对象结果、验证码和计数器；Hash 适合存对象字段，比如用户信息；List 适合有顺序的数据场景，比如简单队列和最新记录；Set 适合去重和成员判断；ZSet 因为支持按分数排序，所以很适合排行榜和热度排序场景。

### 1 分钟展开版
Redis 常见的数据结构我一般会从业务场景来理解。String 是最常用的，很多缓存、token、验证码、计数器本质上都可以用 String；Hash 更适合对象型数据，尤其是需要按字段读取或局部更新时；List 适合有顺序的数据，比如简单消息队列、最近记录；Set 的特点是元素唯一，所以适合点赞判重、标签集合、关注关系这类场景；ZSet 是带分数的有序集合，既能保证唯一性又能排序，所以排行榜、积分榜、热度榜通常会优先想到它。

## 十三、你当前阶段最值得记住的不是 API，而是“业务关键词 -> 结构联想”

你可以先把这组联想练熟：

- 缓存对象 -> String / Hash
- 验证码 / token -> String
- 计数器 -> String
- 对象字段更新 -> Hash
- 最新记录 / 简单队列 -> List
- 点赞判重 / 标签集合 -> Set
- 排行榜 / 热度排序 -> ZSet

如果你把这些联想练到看到题目就能脱口而出，那么 Redis 这块已经进入“可面试”状态了。

## 十四、这节课你要避免的几个误区

### 误区 1：只会背名字，不会说场景
只会背 String、Hash、List、Set、ZSet 没意义。
真正有价值的是你能说出每个结构为什么适合那个场景。

### 误区 2：把对象缓存一律说成 Hash
不对。
很多对象缓存直接放 String 也很常见。
关键看你是整体读写，还是字段级读写。

### 误区 3：把 List 说成“Redis 版消息队列万能解”
List 可以做简单队列，但别把它讲得过头。
复杂 MQ 还是专业中间件更合适。

### 误区 4：把 Set 和 ZSet 混掉
Set 是唯一集合；
ZSet 是带分数、可排序的唯一集合。
你必须分清。

### 误区 5：一上来就讲底层结构实现细节
你当前阶段重点不是源码级实现，而是先讲清业务映射关系。

## 十七、给你一个当前阶段最实用的总结

如果你现在只想先抓住这一节最重要的一句话，那就是：

Redis 常见数据结构的学习重点，不是背名字和命令，而是建立“业务问题 -> 结构特征 -> 选型理由”的映射。String 适合简单值和整体缓存，Hash 适合对象字段，List 适合顺序型数据，Set 适合去重，ZSet 适合排序和排行榜。
