## 一、这一节先回答一个问题：事务到底是为了解决什么

你已经知道，MySQL 不是单纯“存一下数据”的工具，而是业务系统里“核心事实的账本”。

一旦它承担的是账本角色，就会立刻遇到一个问题：

“很多操作并不是改一处数据就结束，而是必须一组动作一起成立，业务才算正确。”

这就是事务出现的根本背景。

所以先不要把事务理解成一个要背定义的名词。
你应该先把它理解成：

“为了保证一组相关操作在业务上不出错，数据库提供的一种整体执行机制。”

这一节的目标，不是先背 ACID，而是先把“为什么没有事务会出问题”这件事彻底想明白。

## 二、为什么单条 SQL 正确，不等于业务正确

很多初学者刚接触数据库时，会有一个隐含误解：

“只要每条 SQL 都写对了，系统应该就是对的。”

但业务系统里的问题，往往不是“单条 SQL 错了”，而是：

- 多条 SQL 之间本来应该一起成功
- 或者一起失败
- 或者执行过程中不能被别人插进来打乱

也就是说，真正要保护的，不只是某一条语句，而是“一个完整业务动作”。

比如转账场景：

- A 账户扣 100
- B 账户加 100

如果你把它拆成两条独立 SQL：

- update account set balance = balance - 100 where id = A;
- update account set balance = balance + 100 where id = B;

从语法上看，两条都没问题。
但业务上，只有“这两步一起成立”，转账才成立。

如果第一条执行成功，第二条因为异常失败了，会发生什么？

- A 的钱少了
- B 没收到钱
- 系统账错了

所以事务要解决的第一个核心问题就是：

“业务动作往往由多步组成，数据库需要保证它们作为一个整体来处理，而不是各自为战。”

## 三、事务为什么会出现：因为真实业务天然是“多步骤联动”

数据库里真正重要的写操作，通常不是孤立的。

### 1. 转账

最经典。

- 扣减付款方余额
- 增加收款方余额
- 记录转账流水

这三步缺一不可。
如果只做成一部分，账就会不一致。

### 2. 电商下单

一个下单动作常常至少涉及：

- 生成订单
- 扣减库存
- 记录支付状态或待支付状态
- 写订单明细

如果订单创建了，但库存没扣成功，会有“超卖”风险。
如果库存扣了，但订单没生成，又会出现“货少了但没人买成”的异常。

### 3. 账户余额和积分发放

例如支付成功后：

- 订单状态改为已支付
- 用户余额或积分发生变化
- 记录账户流水

这些本质上也是同一个业务事件的多个面向。

### 4. 后台审批流

例如管理员审核通过某条记录时：

- 主表状态更新
- 审批记录写入日志表
- 某些派生统计被更新

这些步骤如果只完成一半，系统对这件事的描述就会前后矛盾。

所以你要建立一个非常重要的认知：

事务不是数据库“额外附赠的高级功能”，而是因为业务世界本来就存在大量“必须联动成功”的动作，数据库不得不提供这层机制。

## 四、如果没有事务，会出什么问题

这一部分最重要。你要先对“没有事务的世界”形成直观恐惧感，后面学 ACID 才不会变成死记。

### 1. 只执行了一半

还是转账例子。

假设执行顺序如下：

1. 扣掉 A 账户 100 元
2. 程序突然报错
3. B 账户加钱没有执行

结果是：

- 钱凭空少了 100
- 数据处于“中间状态”

这就是典型的“部分成功”。

但对业务来说，部分成功往往比完全失败更可怕。
因为完全失败至少容易发现，而部分成功会把账写坏。

### 2. 外部看到中间状态

还是上面的例子。

如果 A 扣款后、B 加款前，另一个请求恰好来查余额，它可能看到：

- A 已经少钱
- B 还没加钱

这其实不是一个应该暴露给外界的“最终业务状态”，只是执行过程中的中间态。

如果系统允许别人读到这种中间态，就会引出各种异常判断。

### 3. 并发时互相打架

比如库存只有 1 件，两个用户同时下单。

两个请求都先查库存，看到都是 1。
然后都去扣减库存。
最后可能出现：

- 两个人都下单成功
- 实际库存变成负数或逻辑上超卖

这说明数据库面对的不是“一个人慢慢改数据”，而是“很多人同时改”。
如果没有事务与并发控制机制，数据会非常容易乱。

### 4. 系统异常时无法保证业务边界

现实里异常很多：

- 程序崩了
- 网络闪断
- 机器重启
- SQL 执行中间超时
- 应用代码抛异常

如果没有事务，你几乎无法清晰定义：

“这次业务操作，到底算成功了，还是算没成功？”

而事务的一个重要价值，就是给这类问题一个明确边界：

- 成功了，就整体生效
- 失败了，就整体撤回

## 五、你可以把事务理解成“给业务动作加一个打包边界”

这是现阶段最推荐你的理解方式。

事务的本质不是一句抽象定义，而是：

“把若干条本来属于同一个业务动作的数据库操作，打成一个包。”

这个包有两个核心要求：

- 要么整个包都成功
- 要么整个包都不生效

同时，在这个包执行过程中，数据库还要尽量避免外界看到不该看到的中间状态，并处理好多请求并发带来的冲突。

所以事务不是只解决“回滚”问题。
它其实同时在处理三类事：

- 多步骤操作的整体性
- 并发访问下的数据正确性
- 异常场景下的恢复边界

## 六、为什么事务会和并发问题绑在一起

很多人以为事务只是“多条 SQL 一起提交”。
这只说对了一部分。

事务真正变复杂，是因为数据库不是单用户环境，而是高并发环境。

### 1. 单用户世界里，问题还没那么严重

如果整个数据库永远只有你一个人在操作：

- 你改完再查
- 没人和你抢
- 没人看到你的中间过程

那很多问题看起来都不明显。

### 2. 多用户并发世界里，问题立刻爆炸

现实里会发生：

- 你在更新一条订单
- 别人在同时查询这条订单
- 第三个人在同时修改库存
- 第四个人又在批量统计当天销售额

这时候数据库必须回答：

- 你的中间修改，别人能不能看到？
- 两个人同时改同一条数据，谁先谁后？
- 一个事务读到的数据，能不能被另一个事务中途改掉？
- 重复读两次时，结果要不要保持一致？

于是事务不只是“打包提交”，还进一步演化出：

- 隔离级别
- 锁
- MVCC
- undo log / redo log

所以你后面会看到：

事务像一个总入口，后面牵出整套并发控制体系。

## 七、先不要急着背 ACID，先用业务语言理解它们为什么存在

你之后一定会学到 ACID，但现在先翻译成业务直觉。

### 1. A：Atomicity，原子性

业务直觉：
一组操作不能只做一半。

转账时，扣钱和加钱必须一起成功或一起失败。
这就是为什么事务要强调“不可分割”。

### 2. C：Consistency，一致性

业务直觉：
事务执行前后，系统应满足业务规则，不能把数据搞到自相矛盾。

比如：

- 总账不能平白少钱
- 库存不能莫名其妙错乱
- 订单状态不能前后冲突

### 3. I：Isolation，隔离性

业务直觉：
多个事务同时执行时，彼此不要互相干扰到把结果搞乱。

你做你的转账，我做我的下单，不能因为互相穿插执行，就把账搞错。

### 4. D：Durability，持久性

业务直觉：
一旦事务提交成功，就不能因为宕机而当作没发生。

也就是：
“系统既然认账了，就要认到底。”

你现在先不用背标准定义，但要记住：
ACID 不是背书材料，而是数据库为了解决真实业务问题抽象出来的四个保障方向。

## 八、一个最小案例：为什么“扣库存”必须进入事务视角

假设商品库存初始为 10。
一个用户下单 1 件。

如果你只看单条 SQL，可能会觉得：

- update product set stock = stock - 1 where id = 1;

很简单。

但业务上通常不止这一步，还包括：

- 判断库存是否足够
- 扣减库存
- 创建订单
- 写订单明细

如果没有事务，可能发生：

场景 A：库存扣了，订单没创建成功
- 库存少了
- 但系统里没有对应订单

场景 B：订单创建了，库存没扣成功
- 用户以为下单成功
- 实际库存没锁住，后续可能超卖

场景 C：两个请求同时读取库存后再扣减
- 都觉得库存足够
- 最后扣出错

你会发现：
“扣库存”这个事之所以复杂，不在于 update 这条 SQL 本身，而在于它必须和其他操作一起形成一个正确的业务闭环。

这正是事务视角的价值。

## 九、把事务和上一节连起来看

上一节你已经建立了一个框架：

- MySQL 是核心事实数据中心
- 它要在并发下保证数据别乱
- 后面那些复杂机制，都是为正确性和性能平衡服务

那么事务正好就是你进入内部机制的第一扇门。

因为当数据库承担“核心账本”角色时，它必须先回答：

“一个完整业务动作，如何被正确地提交或撤销？”

这就是事务最先要解决的问题。

也就是说：

- 上一节解决的是“数据库为什么重要”
- 这一节解决的是“既然数据库这么重要，那它如何保证业务动作别做坏”

这样知识链条就接上了。

## 十、这一节最核心的认知总结

先别急着背细节，先记住这几句话：

- 事务出现，不是因为数据库爱复杂，而是因为真实业务天然是多步骤联动的
- 单条 SQL 正确，不等于整个业务动作正确
- 没有事务，系统会出现部分成功、中间态暴露、并发打架、异常后边界不清等问题
- 事务本质上是在数据库里给“一个业务动作”加上整体执行边界
- 后面的隔离级别、锁、MVCC、日志，都是围绕事务继续展开的

## 十一、你现在应该能回答的 4 个问题

学完这一节，你至少要能用自己的话回答：

1. 为什么业务系统会需要事务，而不是把多条 SQL 依次执行就行？
		因为业务系统往往不是一条SQL就能解决的，事务能够保证业务的一致性，以及多条sql执行时发生异常后，能够进行回滚，保证事务的原子性
2. 没有事务时，最容易出现哪几类问题？
		事务如果没有，那么系统就会部分成功、中间态暴露、并发打架、异常后边界不清等问题
3. 事务本质上在数据库里提供了什么能力？
		事务在数据库里提供了原子性、一致性、隔离性、持久性这些能力
4. 为什么一讲事务，后面就会连到锁、MVCC、日志？
		因为当存在并发问题，就必须要用这些技术来保障事务在高并发下也能正确执行

如果这四个问题你能回答顺，后面再进入 ACID、隔离级别、脏读 / 不可重复读 / 幻读，你会轻松很多。
