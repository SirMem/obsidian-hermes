## 一、这节课为什么要学

前面几节你已经把 Redis 的几个核心角色逐步串起来了：
- Redis 和 MySQL 为什么要分工
- Redis 为什么不只是缓存，还常常承担并发协调
- 分布式锁是在补什么边界
- 缓存一致性、热点 key、异步补偿这些问题为什么会出现

继续往下，项目和面试里很容易进入一个更综合的话题：

“如果是秒杀系统，Redis、MQ、MySQL 到底各做什么？限流、库存扣减、幂等、防重这些方案又该怎么拼起来？”

这类问题之所以重要，是因为它已经不再是单个知识点，而是在考你有没有系统视角。

秒杀场景的本质，不只是“高并发”三个字，
而是：
- 流量会瞬间冲高
- 热点会极度集中
- 用户会重复点击
- 请求会超卖、重单、积压、重试
- 你既想扛住流量，又想最终数据别乱

所以这一节课的目标，是把一条可用于项目表达的主线搭出来：

“秒杀系统通常不是靠某一个神奇组件解决问题，而是 Redis 做前置削峰和协调，MQ 做异步削峰和解耦，MySQL 做最终订单与库存正确性落地，三者一起组成一条分层防线。”

## 二、先别急着看技巧，先看秒杀系统到底在和什么对抗

你可以先把秒杀想成一场“系统抗压战”，它同时在对抗四类问题：

### 1. 对抗瞬时洪峰

同一个商品在几秒内涌入大量请求。
如果这些请求都直接打到 MySQL：
- 连接数会被打满
- 行锁竞争会急剧上升
- 数据库很容易成为系统瓶颈

### 2. 对抗重复请求

一个用户可能：
- 手抖多点几次
- 网页卡顿后重复提交
- 客户端超时后自动重试

如果系统没有防重，可能出现：
- 一人多单
- 多次扣库存
- 同一条消息被重复消费

### 3. 对抗超卖

秒杀里最核心的正确性问题之一就是：

“库存只有 100 件，最后不能卖出 101 件。”

### 4. 对抗链路抖动

真实系统里，请求链路并不总是一次成功：
- 应用服务可能重试
- MQ 消费可能重复
- 网络可能超时
- 某一步可能成功了，但响应丢了

所以秒杀系统要解决的，不只是快，
还包括：
- 把流量压住
- 把重复动作识别掉
- 把库存守住
- 把最终结果落正确

这就是后面 Redis、MQ、MySQL 分工的背景。

## 三、先给一句总纲：Redis、MQ、MySQL 在秒杀里各自保护什么

你可以先记这句总纲：

“Redis 更像前置流量闸门和高并发协调层，MQ 更像削峰排队与异步传递层，MySQL 更像最终订单与库存正确性的落地层。”

把这句话再拆开一点：

### 1. Redis：先挡在前面，别让所有请求直接碰库

Redis 在秒杀里经常承担这些角色：
- 热点数据承载，比如活动信息、库存摘要、用户秒杀资格
- 限流与防刷
- 用户级防重复提交
- 预扣库存 / 库存令牌扣减
- 分布式协调，比如锁、去重标记、排队状态标记

它的特点是：
- 快
- 适合做高频判断
- 适合做前置拦截
- 适合把大量无效请求挡在数据库外面

### 2. MQ：不是立刻把所有工作做完，而是把洪峰摊平

MQ 在秒杀里常见职责是：
- 把通过前置校验的请求异步排队
- 削峰填谷
- 解耦“前台抢购”与“后台下单”
- 让订单创建、库存落库等动作改成异步处理
- 支持失败重试、重放、补偿

它的特点是：
- 让入口请求更轻
- 让后端消费速度更可控
- 让系统不必在峰值时同步完成所有重操作

### 3. MySQL：真正负责最终数据别乱

MySQL 在秒杀里通常承担：
- 订单持久化
- 库存最终扣减
- 一人一单约束
- 事务保证
- 最终一致性兜底

它的特点是：
- 速度不如 Redis，但强在最终正确性
- 它不适合直接扛所有秒杀流量
- 但它必须负责最后“到底卖了多少、订单是否真的成立”

### 4. 一句话概括三者关系

你在面试里可以先这样说：

“秒杀系统里一般不会让所有请求直接打 MySQL，而是先用 Redis 做活动信息缓存、限流、防重和前置库存判断，再通过 MQ 做异步排队削峰，最后由 MySQL 负责订单落地、库存最终扣减和唯一约束兜底。Redis 更偏入口拦截和协调，MQ 更偏削峰和异步化，MySQL 更偏最终正确性。”

## 四、秒杀系统最小主链路：把整个流程先串起来

先看一个简化版链路：

1. 用户点击秒杀
2. 请求先到应用层
3. 应用先查 Redis 中的活动状态、商品是否开抢、是否售罄
4. 通过 Redis 做限流、防刷、用户级防重
5. 通过 Redis 预扣库存或扣减库存令牌
6. 资格通过后，把下单请求投递到 MQ
7. 消费者从 MQ 拉取消息，执行真正下单逻辑
8. 在 MySQL 中创建订单、扣减真实库存、写入订单明细
9. 利用唯一索引 / 幂等表 / 消费幂等控制，避免重复落单
10. 返回用户“排队中”或“抢购结果”

这个流程的核心思想不是“每一步都最强一致”，
而是：

“先用 Redis 在前面尽量拦截和压缩请求，再让 MQ 把有效请求变成可控流量，最后由 MySQL 保证最终结果正确。”

## 五、Redis 在秒杀里具体承担哪些角色

这一部分你要重点建立：Redis 不只是“存库存”，而是在做前置调度。

### 1. 活动信息与热点数据缓存

比如：
- 活动开始时间
- 商品是否参与秒杀
- 秒杀价格
- 剩余库存摘要
- 商品售罄标记

这些数据访问频率极高，放 MySQL 会很容易把数据库打热。
Redis 适合承接这种热点读。

最小例子：
- `seckill:activity:2001`
- `seckill:stock:sku:3001`
- `seckill:soldout:sku:3001`

### 2. 限流

秒杀不是所有请求都值得继续往后走。
很多系统会在 Redis 这一层先做：
- 接口级限流
- 用户级限流
- IP 级限流
- 活动级令牌桶 / 计数器控制

它在解决什么？

它在解决：
“先把明显超量、异常、刷子流量挡掉，不要让它们进入后续库存和下单链路。”

最小例子：
- `INCR req:user:1001:1s`
- 超过阈值直接拒绝

### 3. 用户级防重复

比如同一用户对同一商品：
- 只允许一次请求进入有效链路
- 或者短时间内只能提交一次

最小 key：
- `seckill:dup:user:1001:sku:3001`

这样做的价值是：
还没到 MQ、还没到 MySQL，就先把明显重复的动作压掉。

### 4. 预扣库存 / 库存令牌控制

这是秒杀里 Redis 最经典的角色之一。

核心思路不是：
“所有请求都进数据库，再由数据库判断库存够不够。”

而是：
“先在 Redis 里把能成功的资格数量卡住，减少大量无效请求碰库。”

最小例子：
- 初始库存 100
- Redis 中 `seckill:stock:sku:3001 = 100`
- 每个通过前置校验的请求先 `DECR`
- 如果结果 >= 0，说明拿到一个库存名额
- 如果结果 < 0，说明没库存了，立刻回滚或直接失败

这一步本质上是：
先把“可能成功的人数”控制住。

### 5. 状态标记与协调

除了库存，Redis 在秒杀里还经常承担这些协调标记：
- 用户是否已抢购成功
- 用户是否正在排队
- 订单结果是否已生成
- 某个商品是否已售罄
- 某个消息是否已消费过

所以 Redis 在这里越来越像：
“高并发状态机的快速外层存储”。

## 六、MQ 在秒杀里为什么几乎总会出现

很多初学者会觉得：

“Redis 都已经很快了，为什么还要 MQ？”

因为 Redis 再快，也只是让前置判断更快，
但后面的创建订单、写数据库、发优惠、写流水这些动作，依然是重操作。

如果这些重操作都同步做：
- 接口响应会很慢
- 高峰时应用线程会被占满
- 数据库仍然可能被短时间压垮

所以 MQ 的价值，不是替代 Redis，
而是接过 Redis 筛选后的“有效请求”，把它们变成可排队、可按速率处理的工作流。

### 1. MQ 做削峰

核心思想是：
入口 1 秒来了 5 万个请求，
但后端真正稳定能处理的，可能只有每秒几千个。

MQ 就像一个缓冲带：
- 前面快速收
- 后面平稳消费

### 2. MQ 做异步化

用户请求进入后，不一定要同步等到：
- 订单写完
- 库存落库完
- 所有后续逻辑全部完成

很多时候前台可以先返回：
- 排队中
- 请求已受理
- 结果稍后查询

这样前台接口更轻，系统也更稳。

### 3. MQ 做解耦

秒杀成功后，后面还可能有：
- 发优惠券
- 发站内信
- 更新统计
- 写审计日志

这些都不该和“主链路下单”强耦合在一起。

### 4. MQ 也带来新问题

你不能把 MQ 想成只带来好处。
它也会带来：
- 消息重复消费
- 顺序问题
- 积压问题
- 消费失败重试问题

所以一旦引入 MQ，幂等就必须跟上。

## 七、MySQL 在秒杀里为什么仍然是最终兜底

哪怕前面 Redis 做了预扣库存，MQ 做了削峰，
最后仍然不能说系统就已经正确了。

因为 Redis 和 MQ 更偏“前置控制”和“流程调度”，
真正涉及：
- 订单是不是成立
- 库存最终扣了没有
- 同一个用户有没有真的只下一单
- 数据是否长期可追溯

这些还是要落回 MySQL。

### 1. 订单要持久化

Redis 适合高频状态，
但正式订单、订单明细、支付状态这些核心业务数据，仍然应该长期落库。

### 2. 真实库存要有最终落地扣减

Redis 的库存更像前置名额控制。
MySQL 的库存扣减才是最终账本。

### 3. 一人一单要有数据库级兜底

比如在订单表建立唯一约束：
- `(user_id, activity_id, sku_id)` 唯一

这样即使前面某一层防重漏了，
数据库仍然能挡住最终重复订单。

### 4. 事务要保证订单和库存关系别断裂

比如：
- 创建订单
- 扣减库存
- 写订单明细

这些如果属于同一业务事务，就应该尽量在数据库事务中保证原子性或通过可靠消息实现最终一致。

## 八、Redis 分布式锁在秒杀里常见误区有哪些

这是很容易被问的点。
你不要一提秒杀就机械说“加 Redis 锁就好了”。
很多时候问题恰恰出在这里。

### 1. 误区一：把 Redis 分布式锁当成秒杀的万能解

很多人会觉得：

“库存扣减前加一把锁，不就不会超卖了吗？”

这类想法太粗了。
因为秒杀问题不是只有“互斥”这一个问题，
它还有：
- 洪峰流量
- 重复请求
- 消息重复
- 数据落库正确性
- 可扩展性

如果你把所有请求都串行锁住：
- 吞吐会很差
- 锁竞争会很重
- 系统容易卡死在入口

所以要记住：

“秒杀系统里分布式锁通常只是局部工具，不是总方案。”

### 2. 误区二：以为有 Redis 锁，就不需要 MySQL 最终兜底

这是非常危险的想法。
Redis 锁更像入口控制，
MySQL 才是最终账本。

即使你在业务前面加了锁，也仍然需要：
- 数据库库存扣减条件控制
- 唯一索引防重单
- 事务 / 最终一致兜底

一句话：
Redis 锁不能替代数据库级正确性控制。

### 3. 误区三：锁粒度过大

比如把整个活动都锁成一把：
- `lock:seckill:activity:2001`

结果就是：
- 所有人都争同一把锁
- 本来能并发的事情被强行串行
- 系统性能急剧下降

锁粒度如果要用，通常应该围绕真正需要互斥的资源设计，
而不是一把大锁锁天下。

### 4. 误区四：只会 setnx，却不重视过期时间和锁值唯一性

如果只是简单：
- `SETNX lock:key 1`

却没有：
- 过期时间
- 唯一 requestId
- 校验后再释放

就会出现：
- 服务异常后锁不释放
- 锁误删
- 其他线程的锁被自己删掉

正确思路至少应该是：
- 加锁时带过期时间
- value 使用唯一标识
- 解锁时校验 value 再删除，通常借助 Lua 保证原子性

### 5. 误区五：业务执行时间超过锁过期时间，却没处理续期问题

比如锁只设 3 秒，
但实际业务跑了 8 秒。

那么可能出现：
- 锁提前过期
- 第二个实例拿到新锁
- 两个执行者同时进入关键区

这会让你误以为“明明加锁了，怎么还并发”。

### 6. 误区六：把锁放在过于靠后的位置

如果前面已经做了大量计算、查库、发请求，
最后才想到加锁，
那很多系统消耗已经发生了。

锁如果要做业务互斥，通常要尽量放在关键流程入口附近。

### 7. 误区七：以为拿到锁就等于实现了幂等

锁控制的是“同一时刻谁能执行”，
幂等控制的是“重复请求进来了，最终结果仍然只生效一次”。

它们不是一回事。

举个最小例子：
- 第一次请求拿锁成功，执行完下单
- 客户端超时又重试一次
- 第二次请求稍后又拿到锁成功

如果没有幂等控制，仍然可能重复下单。

所以：
锁 != 幂等

### 8. 误区八：高并发场景里到处加锁，却不优先考虑无锁化原子方案

很多秒杀库存控制，其实更适合：
- Redis 原子扣减
- Lua 脚本原子判断 + 扣减
- MySQL 条件更新
- 唯一索引兜底

而不是先想到“每次都加一个重锁”。

从工程角度说：
能用原子操作解决的，通常优先于大范围互斥锁。

## 九、限流、库存扣减、幂等，这几个方案到底怎么组合

这是这节最关键的部分。
你要形成的认知不是“选一个”，而是“分层组合”。

先给结论：

“限流管入口流量，库存扣减管资源名额，幂等管重复生效，三者分别保护不同层面，通常要组合起来用，而不是互相替代。”

### 1. 限流在保护什么

限流保护的是系统容量。

它回答的问题是：
- 这波请求量，系统接不接得住？
- 哪些明显超量请求要提前拒绝？

它通常发生在最外层。

### 2. 库存扣减在保护什么

库存扣减保护的是商品资源本身。

它回答的问题是：
- 库存还有没有
- 谁拿到了可购买资格
- 会不会超卖

它通常发生在通过限流之后。

### 3. 幂等在保护什么

幂等保护的是最终结果不要重复生效。

它回答的问题是：
- 同一个用户重试多次，最终是不是仍然只生成一笔订单
- 同一条 MQ 消息被重复消费，最终是不是仍然只扣一次库存 / 建一笔订单

它通常贯穿：
- 请求入口
- MQ 消费
- 数据库落地

### 4. 三者的顺序通常怎么摆

一个比较常见的顺序是：

1. 先限流
2. 再做用户级防重 / 幂等预检查
3. 再做 Redis 预扣库存
4. 成功后投递 MQ
5. MQ 消费端再次做幂等校验
6. MySQL 中做最终库存扣减与订单落地
7. 数据库唯一约束做最后兜底

这套思路你要注意：
前面偏“少让无效请求进来”，
后面偏“即使进来了，也别让结果重复生效”。

## 十、一个更完整的组合方案，按层拆开理解

下面给你一个适合面试表达的组合版。

### 第 1 层：入口限流

目的：
先保护系统别被打穿。

可做法：
- 网关限流
- Redis 计数器限流
- 用户/IP 维度限流

这一步主要挡：
- 明显异常流量
- 刷子流量
- 超过系统承载上限的流量

### 第 2 层：活动与资格校验

目的：
不要让无效用户继续往下走。

检查：
- 活动是否开始
- 商品是否参与秒杀
- 用户是否有资格
- 用户是否已购买过

这些通常可放 Redis 热点数据里快速判断。

### 第 3 层：用户级防重复提交

目的：
挡住用户连点、重复请求。

可做法：
- Redis 短期防重 key
- 请求 token
- 前端按钮置灰 + 后端防重双保险

### 第 4 层：Redis 预扣库存

目的：
尽量不要让明知无库存的请求进入 MQ / MySQL。

常见做法：
- `DECR` 预扣
- Lua 脚本做“判断库存 > 0 再扣减”的原子操作

如果库存不足，直接失败。
如果成功，说明先拿到一个名额。

### 第 5 层：投递 MQ 异步排队

目的：
把有效请求从同步链路切到异步链路，做削峰。

前台此时可以返回：
- 请求已接收
- 正在排队
- 请稍后查询结果

### 第 6 层：消费端幂等控制

目的：
防止消息重复消费造成重复下单。

可做法：
- 消息唯一 ID + 消费记录表
- Redis 幂等标记
- 订单业务唯一号

### 第 7 层：MySQL 最终落单 + 扣减真实库存

目的：
让结果真正成立。

可做法：
- `update stock set count = count - 1 where sku_id = ? and count > 0`
- 订单表唯一索引限制一人一单
- 事务里写订单、写明细、更新库存

### 第 8 层：失败补偿

目的：
如果后面落库失败，要修正前面状态。

比如：
- Redis 已预扣库存，但 MQ 消费落单失败
- 需要回补 Redis 库存
- 或标记该资格失效并触发补偿任务

这一步很重要，
因为你不能只看到“扣”，还要看到“失败时怎么回”。

## 十一、最小案例：100 件库存的秒杀，方案怎么跑起来

假设：
- 商品 sku:3001
- 秒杀库存 100
- 限购 1 件

可以这样理解主流程：

### 1. 活动预热

系统提前把这些数据放进 Redis：
- 活动信息
- 商品状态
- `seckill:stock:sku:3001 = 100`
- 售罄标记初始化为空

### 2. 用户请求进入

先做：
- 网关限流
- Redis 用户/IP 限流
- 校验活动是否开始
- 校验用户是否已抢购成功

### 3. 防重

给用户打一个短期防重标记：
- `seckill:req:user:1001:sku:3001`

如果短时间内重复提交，直接拒绝。

### 4. 预扣库存

用 Lua 脚本原子执行：
- 判断库存是否大于 0
- 大于 0 则扣 1
- 否则返回失败

如果失败，直接告诉用户“已售罄”。

### 5. 投递 MQ

拿到库存资格后，把下单请求投进 MQ。
前台可以返回“排队中”。

### 6. 消费端落库

消费者拿到消息后：
- 先做消息幂等
- 再检查订单是否已存在
- MySQL 条件扣减真实库存
- 创建订单
- 提交事务

### 7. 成功后更新状态

可在 Redis 里标记：
- 用户已成功
- 用户查询结果为 success

### 8. 如果失败

比如消费端落库失败，
系统需要：
- 记录失败原因
- 触发补偿
- 必要时回补 Redis 库存名额

这个案例想说明的是：
秒杀系统不是只靠某一条命令扛起来的，
而是每一层都在拦一类风险。

## 十二、面试里怎么回答“秒杀系统里 Redis、MQ、MySQL 怎么配合”

你可以直接用下面这个口语版：

“秒杀系统里我会把 Redis、MQ、MySQL 看成三层分工。Redis 主要负责前置拦截和高并发协调，比如活动信息缓存、限流、防重、预扣库存、售罄标记等，尽量不让无效请求直接打到数据库。MQ 主要负责把通过校验的请求异步排队，做削峰填谷，让后端按稳定速率处理。MySQL 则负责最终订单落地、真实库存扣减、事务和唯一约束兜底，保证结果最终正确。

如果再细一点，限流、防刷是第一层，防重和幂等贯穿入口和消费端，Redis 预扣库存控制的是前置名额，MQ 解决的是洪峰异步化，MySQL 负责最后不超卖、不重复下单。这几个方案不是互相替代，而是分层组合。”

## 十三、这节课你最该记住的几句话

### 1.
Redis 在秒杀里不只是缓存，更像高并发入口协调层。

### 2.
MQ 的核心价值不是“高级”，而是把瞬时洪峰变成后端可控流量。

### 3.
MySQL 不适合直接扛所有秒杀请求，但必须负责最终正确性。

### 4.
Redis 分布式锁只是局部工具，不是秒杀系统的万能答案。

### 5.
限流、库存扣减、幂等分别保护系统容量、资源名额、结果唯一，通常必须组合使用。

### 6.
前面越多地拦掉无效请求，后面数据库越稳；但最终正确性仍然要靠 MySQL 兜底。
