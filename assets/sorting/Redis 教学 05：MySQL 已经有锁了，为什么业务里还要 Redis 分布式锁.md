## 一、这节课为什么要学

当你刚学完 MySQL 的事务、MVCC、行锁、间隙锁这些内容时，很容易冒出一个自然问题：

“既然 InnoDB 已经会自动加锁，数据库最终也确实是数据落地的地方，那为什么业务里还要 Redis 分布式锁？”

这个问题非常好。
因为它说明你已经开始意识到：
- MySQL 里有锁
- 业务系统里也经常说锁
- 但这两类锁看起来都在解决‘并发不要乱’

如果这时候不把边界讲清楚，你就会进入一种很容易混乱的状态：
- 看到并发问题就想上 Redis 锁
- 或者反过来，觉得 MySQL 都有锁了，Redis 锁是不是多余的

所以这一节要做的，不是让你背“分布式锁定义”，
而是把一条真正能用于项目表达和面试回答的主线搭起来：

“MySQL 锁和 Redis 分布式锁，到底各自在保护什么，它们为什么经常要一起出现？”

---

## 二、先给出一句最核心的话

你可以先记住这句总纲：

“MySQL 的锁，主要保护数据库内部的数据读写一致性；Redis 分布式锁，主要保护数据库之外的业务执行过程互斥。”

如果把这句话再展开一点，就是：
- MySQL 锁更偏数据层
- Redis 分布式锁更偏业务层
- MySQL 锁更像最终正确性的兜底
- Redis 分布式锁更像前置并发协调

这就是整节课最核心的认知。

---

## 三、为什么说 MySQL 锁擅长保护“数据库里的东西”

你现在学到的 InnoDB 锁，最擅长保护的是：
- 某一行记录
- 某一段索引范围
- 某个事务里的数据库访问顺序
- 数据库内部的一致性

比如：
- 两个事务同时 update 同一行
- 一个事务 select ... for update 后，另一个事务又想改这一行
- 一个范围更新时，不希望别人往这个范围里插入新记录

这些问题有一个共同点：

“冲突直接发生在数据库记录或索引范围上。”

也就是说，MySQL 锁的强项是：

“当多个事务都在碰库里的同一批数据时，由数据库自己来控制顺序。”

这个能力非常重要，
但它主要是在数据库这个边界里发力。

---

## 四、为什么真实业务里的冲突，常常不只发生在数据库这一刻

很多初学者会下意识地认为：

“最终数据都进 MySQL，那冲突当然就交给 MySQL 锁处理。”

这句话只对一半。

因为真实业务流程通常不是：
- 一条 SQL
- 立刻结束

而更常见的是这样：

1. 请求进来
2. 校验用户状态
3. 查 Redis / 查缓存
4. 做业务判断
5. 调第三方接口
6. 生成订单号或任务号
7. 最后写 MySQL
8. 再发消息或更新别的系统

你会发现：
真正要防冲突的，很多时候不是“某一行记录被同时 update”这么简单，
而是：

“同一个业务动作，能不能只允许一个执行者进去？”

这就是 MySQL 锁和 Redis 锁开始分工的地方。

---

## 五、Redis 分布式锁到底在补什么

Redis 分布式锁，本质上是在业务入口层先做一道互斥控制。

你可以把它理解成：

- MySQL 锁：你已经进数据库里干活了，我来保证库里的并发别乱
- Redis 分布式锁：你开始做整套业务动作之前，我先决定这件事现在只能谁做

所以 Redis 锁锁住的，常常不是“某行数据库记录”，
而是一个业务资源标识，比如：
- 用户 ID
- 订单号
- 商品 ID
- 任务名
- 某个业务主键

它控制的是：
- 这次业务流程谁先执行
- 同一时间是否允许多个服务实例一起处理同一件事

所以 Redis 分布式锁并不是去替代 MySQL 行锁，
而是在更外层，先做一层“业务动作互斥”。

---

## 六、为什么“最终数据落在 MySQL”，也不代表只靠 MySQL 锁就够了

这个点你一定要想透。

“最终落库在 MySQL”
不等于
“整个业务冲突都只发生在 MySQL 内部”。

举个下单过程的最小抽象：

1. 用户点击下单
2. 服务检查用户是否重复提交
3. 服务判断库存是否足够
4. 服务计算优惠
5. 服务创建订单
6. 服务扣库存
7. 服务写数据库
8. 服务发通知

这里面很多风险，并不是“某行数据有没有锁住”这么简单。

更可能的问题是：
- 用户连点两次，两个请求同时进来
- 两台应用服务器同时处理同一个用户的下单请求
- 某个任务重复执行了两遍
- 某个消费消息被多个节点重复处理

这些问题的共性是：

“你想限制的是整个业务动作，而不是某条库记录。”

这时候 Redis 锁就比直接依赖 MySQL 行锁更贴合业务边界。

---

## 七、最小案例 1：防重复下单

### 场景

同一个用户短时间内连续点两次“提交订单”。
两个请求几乎同时到达服务端。

### 如果只想靠 MySQL 锁

你当然可以在订单表上做一些唯一约束或事务控制，
最后也可能兜住“不要生成两笔重复订单”。

但问题是：
在请求真正写库之前，两个请求可能都已经做了很多重复工作：
- 查库存
- 算价格
- 调优惠券服务
- 生成订单草稿
- 走了一大段业务流程

这时候即使最后数据库层拦住了，
前面的系统消耗也已经发生了。

### 如果加 Redis 分布式锁

你可以先按用户维度加一个业务锁：

```text
lock:order:user:1001
```

谁先拿到锁，谁进入下单流程；
后来的请求直接返回“请勿重复提交”或短暂等待。

### 这个案例想说明什么

这里 Redis 锁保护的不是“订单表某一行”，
而是：

“用户 1001 的这次下单动作，同一时刻只能有一个执行者。”

这就是典型的业务动作互斥。

而真正到数据库里：
- 订单创建
- 库存扣减
- 状态更新
仍然应该由 MySQL 事务和约束去保证最终正确性。

所以这类场景经常是：
- Redis 锁做入口防重
- MySQL 做最终一致性兜底

---

## 八、最小案例 2：定时任务防重复执行

### 场景

系统里部署了 3 台应用实例。
每天凌晨有一个“结算任务”要执行一次。

### 只靠 MySQL 锁会怎么样

理论上你也可以设计一张任务表，
让多个实例去抢某条任务记录。

这并不是完全做不到。
但工程上很多时候你真正想表达的是：

“今天这次结算任务，全系统只能有一个实例执行。”

这是一个非常典型的“跨机器、跨进程协调问题”。

### 如果用 Redis 分布式锁

比如：

```text
lock:job:daily-settlement
```

谁先拿到锁，谁执行；
其他实例发现拿不到锁，就直接退出。

### 这个案例想说明什么

这里你要锁住的对象，根本不是某条数据库业务记录，
而是一个“任务执行资格”。

这种场景天然更像 Redis 分布式锁擅长的领域。

---

## 九、最小案例 3：秒杀 / 库存场景里为什么经常两种锁一起出现

### 场景

某个商品库存很少，短时间内有大量请求涌入。
你希望：
- 不要超卖
- 不要把数据库打爆
- 不要让同一个用户疯狂重复提交

### 如果只靠 MySQL

MySQL 可以通过：
- 事务
- 行锁
- 条件更新
- 唯一约束

来尽量保证：
- 库存最终不会扣成负数
- 某条库存记录不会被并发改乱

但如果所有请求都直接打到数据库，
数据库压力可能会非常大。
而且很多重复请求在进入数据库前，其实就该先被挡住一部分。

### 如果 Redis 和 MySQL 一起配合

你可以把分工理解成：

#### Redis 侧
- 做秒杀资格校验
- 做热点削峰
- 做用户级防重
- 用分布式锁控制某些关键业务动作不要并发冲进来

#### MySQL 侧
- 对库存记录做最终扣减
- 用事务保证订单与库存的一致性
- 用数据库约束保证最终结果正确

### 这个案例想说明什么

Redis 更像在前面做“限流、削峰、资格控制、互斥协调”；
MySQL 更像在后面做“最终落账、最终正确性保证”。

所以两者不是谁替代谁，
而是经常一起组成一个完整方案。

## 十、你不要形成的误区

### 误区 1：有 Redis 锁了，就不用 MySQL 事务和约束了

不对。

Redis 锁通常只是前置协调，
MySQL 事务、唯一索引、行锁等机制仍然是最终正确性的核心保障。

如果只靠 Redis 锁，系统可能仍然会遇到：
- 锁过期
- 服务卡顿
- 网络抖动
- 客户端异常退出
- 锁误释放

所以成熟系统不会把 Redis 锁当成唯一防线。

### 误区 2：有 MySQL 锁了，Redis 锁就一定多余

也不对。

如果你的问题本质上是：
- 同一个业务动作只能有一个执行者
- 同一个任务只能由一个节点跑
- 同一个用户短时间内不要重复进入一段流程

那这已经不是单纯的“库里某行记录怎么锁”的问题了。

Redis 锁会更贴合业务边界。

### 误区 3：所有并发问题都先上分布式锁

也不对。

如果你的问题本质只是：
- 更新某条记录
- 扣减某个余额
- 修改某个订单状态

那首先应该想到的是：
- MySQL 事务
- 行锁 / 条件更新
- 唯一索引
- 乐观锁 / 悲观锁

而不是一上来就把问题全甩给 Redis 锁。

---

## 十一、什么时候优先想到 MySQL 锁，什么时候优先想到 Redis 分布式锁

你可以用一个很实用的判断法：

### 更偏 MySQL 的场景

如果问题本质是：
“同一批数据库记录不能被并发改乱。”

比如：
- 转账扣减余额
- 更新订单状态
- 库存记录扣减
- 根据状态条件做更新

这时优先想到：
- MySQL 事务
- 行锁
- 唯一索引
- 乐观锁 / 悲观锁

### 更偏 Redis 分布式锁的场景

如果问题本质是：
“某个业务动作在分布式系统里，同一时刻只能有一个执行者。”

比如：
- 防重复下单
- 防重复支付
- 定时任务防重复执行
- 某个消费逻辑不能被多节点同时处理
- 某个用户的一段关键流程需要串行化

这时更容易想到 Redis 分布式锁。

---

## 十二、把这节课压成一句项目表达

你在项目或面试里可以这样说：

“MySQL 的锁主要解决数据库内部的并发读写一致性问题，比如事务里对记录和索引范围的访问控制；Redis 分布式锁主要解决数据库之外的业务过程互斥问题，比如防重复下单、定时任务防重、分布式多实例竞争。前者更偏数据层，后者更偏业务层。成熟系统里通常不是二选一，而是 Redis 做前置协调，MySQL 做最终正确性兜底。”

这段话如果你能顺下来，说明你已经不是把锁当成单个技术点在看了，而是在按系统分层理解并发控制。

---

## 十三、这一节你先记住的最小骨架

如果你现在只记最小骨架，就记这 6 句：

1. MySQL 锁主要保护数据库内部的数据读写一致性
2. Redis 分布式锁主要保护业务执行过程的互斥
3. 最终数据落在 MySQL，不等于整个业务冲突只发生在 MySQL
4. Redis 锁常用于防重复提交、任务防重、分布式实例协调
5. MySQL 事务和约束是最终正确性的核心保障
6. Redis 锁和 MySQL 锁经常是配合关系，不是替代关系

---

## 十四、面试版简答模板

### 1）既然 MySQL 已经有锁了，为什么还要 Redis 分布式锁？

“MySQL 的锁主要作用在数据库内部，保护事务对记录或索引范围的并发访问，解决的是数据层一致性问题。但真实业务往往不是一条 SQL，而是一整段跨服务、跨缓存、跨实例的业务流程。很多时候我们希望控制的是‘同一个业务动作只能有一个执行者’，比如防重复下单、定时任务防重、分布式实例竞争等，这时候只靠 MySQL 行锁不够贴合业务边界，所以会在业务入口层再加 Redis 分布式锁做互斥协调。不过 Redis 锁通常不是替代 MySQL 锁，而是和 MySQL 事务、唯一索引等机制一起使用，Redis 更偏前置协调，MySQL 更偏最终正确性兜底。”

### 2）Redis 分布式锁和 MySQL 锁最大的区别是什么？

“MySQL 锁更偏数据库内部的数据访问控制，Redis 分布式锁更偏业务流程级别的互斥协调。一个锁的是数据访问，一个锁的是业务执行资格。”

### 3）是不是有 Redis 锁就够了？

“不够。Redis 锁通常只能减少并发冲突进入核心流程，但最终的数据正确性还是要靠 MySQL 事务、约束、状态判断等机制来兜底。”

---

## 十五、学完这一节后，你应该能回答什么

学完这一节，你至少应该能顺着说出：
- 为什么 MySQL 锁和 Redis 分布式锁不是一回事
- 为什么最终数据落在 MySQL，业务里仍然可能需要额外的锁
- 为什么 Redis 锁更像业务入口互斥，而 MySQL 锁更像数据层兜底
- 防重复下单、定时任务防重、秒杀场景里它们分别在做什么

如果这些你能顺下来，说明你对“锁”已经不再只是停留在数据库内部，而是开始进入项目级并发控制视角了。
