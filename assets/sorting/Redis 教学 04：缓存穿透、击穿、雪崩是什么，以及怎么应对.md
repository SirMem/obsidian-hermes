## 一、这节课学什么

前面几节你已经把 Redis 的基础定位先理顺了：
1. Redis 在系统里一般做什么
2. Redis 为什么快
3. 常见数据结构和场景怎么对应
4. 持久化为什么存在，为什么核心数据通常还是放 MySQL

接下来这一节，我们开始进入 Redis 面试里非常高频、也非常容易混淆的一组问题：
- 缓存穿透
- 缓存击穿
- 缓存雪崩

很多人背的时候觉得自己会，但一到面试里一紧张，就很容易把这三个概念说串。

所以这节课的目标不是让你死记定义，而是把它们放回真实系统里理解清楚：

到底是什么请求路径出了问题？
为什么会把压力打到数据库？
它们三者的区别到底在哪？
项目里一般怎么应对？

你只要把这条主线讲顺，面试官继续追问时你也不容易乱。

## 二、先建立总视角：为什么缓存问题最后都会变成“数据库扛压问题”

先别急着区分穿透、击穿、雪崩。

先想一个总问题：
为什么大家一提缓存异常，最后都在担心数据库？

因为 Redis 在很多系统里承担的是“数据库前面的缓冲层”。

理想情况是：
- 大量请求先到 Redis
- Redis 命中后直接返回
- 只有少量请求才会继续访问 MySQL

这样数据库压力会小很多。

但一旦缓存层失效、缺失，或者挡不住请求，流量就会继续往后打到 MySQL。

问题的本质就在这里：

缓存本来是替数据库挡流量的。
如果缓存没有挡住，请求就会集中压到数据库。

所以你可以先把这三个问题统称理解成：

都是“缓存没有有效挡住数据库压力”的不同表现形式。

只是三者触发原因不同。

---

## 三、先给你一个最快区分口诀

你可以先记住这组最实用的区分：

- 穿透：查一个根本不存在的数据
- 击穿：某个热点 key 失效了，大量请求一起打进来
- 雪崩：大量 key 在同一时间集体失效

这三句一定要先练熟。

但只背这三句还不够。

因为真正面试时，面试官常常会继续问：
- 为什么会这样？
- 风险在哪里？
- 怎么解决？

所以下面我们逐个拆开讲。

---

## 四、缓存穿透：请求的数据本来就不存在，缓存和数据库都挡不住

### 1. 什么是缓存穿透
缓存穿透的典型场景是：

客户端一直查询一个根本不存在的数据。

比如：
- 查询一个根本不存在的商品 ID
- 查一个非法用户 ID
- 查一个数据库里根本没有的订单号

这时会发生什么？

如果系统逻辑是：
1. 先查 Redis
2. Redis 没有
3. 再查 MySQL
4. MySQL 也没有

那么问题来了：

因为这个数据本来就不存在，所以 Redis 里永远不会命中；
而如果每次查数据库发现没有之后，也不做任何防护，那每次请求都会继续打到 MySQL。

这就叫缓存穿透。

### 2. 你要抓住它的本质
缓存穿透的关键不是“缓存失效”，而是：

请求的目标数据从一开始就不存在。

所以缓存根本无从建立，数据库也查不到，结果就是请求不断“穿过”缓存，直接打到数据库。

“穿透”这个词其实就来自这里。

### 3. 为什么缓存穿透危险
因为它看起来只是“查不到”，但在高并发下会变成：
- Redis 挡不住
- MySQL 一直被无效请求打
- 如果是恶意攻击，数据库压力会被迅速放大

也就是说：

缓存穿透的危险，不在于业务结果为空，
而在于“无效请求也持续占用数据库资源”。

### 4. 常见应对思路

#### 方案 1：缓存空值
这是最经典、最常见的方案。

做法是：
如果查数据库发现也不存在，不要直接什么都不做，而是把“空结果”也缓存起来，只不过给一个较短过期时间。

这样下次再查同样的不存在数据时，就可以直接在 Redis 命中这个“空值”，不再打到数据库。

这是你当前阶段最应该先掌握的方案。

它的优点是：
- 简单直接
- 对很多常见业务已经够用

但你也要知道边界：
- 如果恶意构造大量不同不存在 key，空值缓存也会占用内存

#### 方案 2：参数校验 / 非法请求拦截
如果一个请求连格式都不合法，比如：
- 商品 ID 明显不符合规则
- 用户 ID 范围非法
- 请求参数为空或异常

那就不应该让它进入“查缓存 -> 查数据库”流程。

也就是说：

能在业务入口挡住的无效请求，尽量在前面就挡住。

#### 方案 3：布隆过滤器
你当前阶段不用讲太深，但要知道它的定位：

布隆过滤器可以用来快速判断“某个 key 大概率存不存在”。

如果过滤器判断它一定不存在，那就直接拦掉，不再打 Redis / MySQL。

这是处理大量不存在请求时比较常见的一类增强方案。

### 5. 面试里怎么说缓存穿透
你可以这样表达：

缓存穿透指的是查询的数据本身就不存在，所以缓存中不会命中，数据库里也查不到。如果系统没有做保护，这类请求每次都会继续打到数据库，导致无效请求持续消耗数据库资源。常见解决思路包括缓存空值、参数校验以及使用布隆过滤器等。

---

## 五、缓存击穿：不是大量 key 都失效，而是“某个热点 key”突然失效

### 1. 什么是缓存击穿
缓存击穿的典型场景是：

某个非常热点的 key，在失效的那个瞬间，刚好有大量并发请求同时来访问它。

比如：
- 首页爆款商品详情
- 某个明星直播间信息
- 高频访问的热门文章
- 首页核心配置结果

这些 key 平时访问量非常高。

正常情况下，它们都由 Redis 扛着。

但如果某一刻这个热点 key 过期了，那么一瞬间的大量请求就会同时发现：
“Redis 里没有了。”

于是这些请求就会一起打到 MySQL。

这就叫缓存击穿。

### 2. 你要抓住它的本质
缓存击穿的关键不是“不存在”，也不是“大量 key 一起过期”，而是：

某个热点 key 失效，导致本来被它挡住的大量请求同时冲向数据库。

所以它的关键词是：
- 热点 key
- 瞬时高并发
- 失效点触发

### 3. 为什么缓存击穿危险
因为它打的是“热点数据”。

热点数据意味着：
这个 key 背后原本就有很高访问量。

所以一旦它失效，数据库面对的不是普通流量，而是瞬时洪峰。

这会导致：
- 数据库短时间压力暴增
- 查询耗时上升
- 更严重时可能引发级联故障

### 4. 常见应对思路

#### 方案 1：互斥锁 / 单飞机制
这是最经典的方案。

做法是：
当热点 key 失效后，不让所有请求都去查数据库，而是只允许其中一个请求去查库并回填缓存，其他请求等待或快速失败后重试。

你可以把它理解成：

“缓存重建这件事，只让一个人做，其他人别一起冲进去。”

这是应对缓存击穿最常见的一条主线。

#### 方案 2：热点数据不过期 / 逻辑过期
对特别热点、特别关键的数据，有时不会简单设置普通过期，而是：
- 不给它真正删除
- 或者用逻辑过期方式，让旧值先继续对外提供，再由后台异步刷新

这样做的核心思想是：

宁可短时间返回一个稍旧的数据，也不要让大量请求同时穿透到数据库。

这在高并发系统里是很常见的权衡。

#### 方案 3：预热热点数据
在活动开始前、系统启动后、热点已知时，提前把高频 key 放进缓存中，减少在高峰时现查现建的风险。

### 5. 面试里怎么说缓存击穿
你可以这样表达：

缓存击穿指的是某个热点 key 在失效瞬间，恰好有大量并发请求访问它，导致这些请求同时穿过缓存打到数据库。它的重点不在于 key 不存在，而在于“热点 key + 瞬时并发”。常见解决思路包括互斥锁保证只有一个线程回源重建缓存，以及对热点数据采用不过期或逻辑过期等方式。

## 六、缓存雪崩：不是一个热点 key 出问题，而是大量 key 集体失效

### 1. 什么是缓存雪崩
缓存雪崩的典型场景是：

大量缓存 key 在同一时间大面积失效，或者缓存层整体不可用，导致原本被缓存挡住的请求，突然大批量压向数据库。

这里你要注意：

雪崩比击穿范围更大。

击穿通常是一个热点 key 的问题；
雪崩通常是一大片缓存一起出问题。

### 2. 雪崩怎么发生
常见触发方式有两类：

#### 情况 1：大量 key 设置了相同过期时间
比如系统上线时，统一给一批 key 配了同样的 TTL。

结果到某个时刻，这一批 key 一起过期。

#### 情况 2：Redis 整体异常
比如：
- Redis 宕机
- Redis 集群不可用
- 网络异常导致缓存层访问失败

这时效果和“大量 key 同时失效”非常像：
缓存层整体失去拦截能力，请求直接涌向数据库。

### 3. 你要抓住它的本质
缓存雪崩的本质是：

本来承担流量拦截作用的大批缓存，在同一时间大面积失效，导致后端数据库承受集中的洪峰请求。

它的关键词是：
- 大量 key
- 同时失效 / 整体不可用
- 大范围流量后压

### 4. 为什么雪崩通常比击穿更可怕
因为它影响的是一大片。

如果击穿是“一个热点口子破了”，
那雪崩更像是“整面堤坝一下子塌了”。

所以它带来的问题通常更重：
- MySQL 压力全面升高
- 应用线程池被拖满
- 响应时间整体恶化
- 甚至可能引发整个系统连锁故障

### 5. 常见应对思路

#### 方案 1：过期时间加随机值
这是最基础也最常用的方案。

目的很简单：
不要让大量 key 在同一秒一起过期。

做法是：
基础 TTL 上再加一个随机偏移量，把失效时间打散。

这样就能显著降低“大面积同秒失效”的概率。

#### 方案 2：多级缓存
比如：
- 本地缓存 + Redis
- Redis + 数据库之间做进一步保护

这样即使 Redis 出现问题，也不至于所有流量直接毫无遮挡地打到数据库。

#### 方案 3：限流、降级、熔断
如果缓存层出了问题，必须有能力在业务层做保护：
- 限制请求量
- 对非核心接口降级
- 暂时返回兜底数据
- 对部分请求快速失败

本质上是：

数据库已经扛不住时，系统要学会主动“少接一点”。

#### 方案 4：提高 Redis 自身可用性
比如主从、哨兵、集群等。

你当前阶段不用展开太深，但要知道：
减少雪崩风险，不只是业务代码层面的事，也包括缓存基础设施本身的高可用建设。

### 6. 面试里怎么说缓存雪崩
你可以这样表达：

缓存雪崩指的是大量缓存 key 在同一时间集体失效，或者 Redis 整体不可用，导致原本由缓存承担的大量请求集中打到数据库。它比缓存击穿影响范围更大。常见应对思路包括给过期时间加随机值、做多级缓存、限流降级，以及提升 Redis 本身的高可用能力。

## 七、现在把三者并排对比，你就不容易混了

你可以把它们压缩成这张对比表来记：

### 1. 触发原因
- 穿透：查的东西本来就不存在
- 击穿：某个热点 key 失效
- 雪崩：大量 key 同时失效或缓存层整体异常

### 2. 问题规模
- 穿透：单类无效请求持续打库
- 击穿：某个热点点位瞬时打库
- 雪崩：大范围请求集体打库

### 3. 关键词
- 穿透：不存在
- 击穿：热点 key
- 雪崩：大量 / 同时

### 4. 常见方案
- 穿透：缓存空值、参数校验、布隆过滤器
- 击穿：互斥锁、热点不过期、逻辑过期、预热
- 雪崩：过期时间随机化、多级缓存、限流降级、高可用

如果你把这组对比记熟，三者基本就不容易说串了。

## 八、把它们放回项目视角，你会更容易理解

假设你做一个电商系统。

### 穿透例子
有人不断请求一个根本不存在的商品 ID：
/product/999999999

如果系统每次都：
Redis 没有 -> MySQL 查不到 -> 直接返回
那数据库就会被无效请求反复击打。

### 击穿例子
某个爆款商品详情缓存刚好过期，此时大促开始，大量用户同时访问这个商品页面。

所有请求同时发现缓存没了，于是都去查 MySQL。

### 雪崩例子
你给首页相关的几千个缓存 key 统一设成 30 分钟过期。

结果 30 分钟一到，这批 key 几乎同时失效，数据库瞬间承受大量查询流量。

你会发现：

这三者虽然都表现为“数据库压力变大”，
但问题的源头完全不同。

这就是为什么面试官会把它们拆开问。

## 九、面试里最实用的回答顺序

如果面试官问：
“你说说缓存穿透、击穿、雪崩。”

你不要一上来就乱背方案。

最稳的顺序是：

### 第一步：先定义
- 穿透：查不存在的数据
- 击穿：热点 key 过期
- 雪崩：大量 key 同时失效

### 第二步：再说风险
三者都会导致请求绕过缓存、打到数据库，造成后端压力上升。

### 第三步：最后说应对
- 穿透：缓存空值、布隆过滤器、参数校验
- 击穿：互斥锁、逻辑过期、热点不过期
- 雪崩：TTL 加随机值、限流降级、多级缓存、高可用

这个顺序最稳，也最像你真的理解了。

## 十一、把三者整理成可直接背诵的面试模板

### 30 秒基础版
缓存穿透、击穿、雪崩的核心区别在于触发原因不同。缓存穿透是查询本来就不存在的数据，导致请求每次都穿过缓存去查数据库；缓存击穿是某个热点 key 在失效瞬间遇到大量并发请求，导致这些请求同时回源；缓存雪崩则是大量 key 同时失效，或者 Redis 整体不可用，导致大量请求集中打到数据库。常见应对思路分别是：穿透用缓存空值、参数校验、布隆过滤器；击穿用互斥锁、逻辑过期、热点不过期；雪崩用过期时间随机化、限流降级、多级缓存和高可用。

### 1 分钟展开版
我理解这三个问题本质上都是缓存没能有效挡住数据库流量，但触发原因不一样。缓存穿透是请求的数据本来就不存在，所以缓存和数据库都查不到，这类无效请求会持续打到数据库，常见做法是缓存空值、做参数校验或布隆过滤器。缓存击穿是某个热点 key 在过期瞬间，恰好有大量并发访问，导致大量请求同时回源，所以通常会用互斥锁保证只有一个线程去重建缓存，或者对热点数据采用逻辑过期。缓存雪崩则是大量 key 同时失效，或者 Redis 整体不可用，影响范围更大，通常要通过随机过期时间、限流降级、多级缓存和 Redis 高可用来应对。

## 十二、你要避免的几个常见误区

### 误区 1：把穿透和击穿混成一类
不一样。
- 穿透：数据本来就不存在
- 击穿：数据原本存在，但热点 key 过期了

### 误区 2：把击穿和雪崩只理解成“缓存失效”
这样太粗。
- 击穿强调一个热点 key
- 雪崩强调大量 key / 整体缓存层

### 误区 3：只会背方案，不知道为什么
比如说“加锁就行”。
你至少要说出：
为什么要加锁？
因为要防止多个请求同时回源重建缓存。

### 误区 4：觉得这些问题只存在于 Redis
其实这些是“缓存架构问题”，只是 Redis 是最常见承载者。

### 误区 5：面试里把所有方案都讲得特别重
你当前阶段不用把所有高级方案都展开。
先把基础主线讲清，已经很够用了。

## 十五、给你一个当前阶段最实用的总结

如果你现在只想先抓住这一节最重要的一句话，那就是：

缓存穿透、击穿、雪崩本质上都是缓存没能有效挡住数据库流量，但它们的问题源头不同：穿透是查不存在的数据，击穿是热点 key 失效，雪崩是大量 key 同时失效或缓存层整体异常；因此应对思路也不同，分别侧重于拦无效请求、控制热点回源，以及防止大面积同时失效并做好系统保护。
