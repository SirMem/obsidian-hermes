## 一、这一篇为什么要学

上一份你已经知道：
- 查询慢，很多时候是因为扫描范围太大
- 索引本质是在优化查询路径
- 联合索引、最左前缀、覆盖索引、索引失效，都会影响 SQL 最终怎么查

那接下来一个非常自然的问题就是：

“我怎么知道 MySQL 最后到底打算怎么查？”

这时候 explain 就出现了。

很多人学 explain 的时候容易走两个极端：
- 要么完全不会看，只知道网上复制 explain 一贴
- 要么一上来背十几个字段，结果越背越乱

你现在最合适的方式不是把 explain 当成一个八股清单，而是先把它理解成：

“explain 是用来观察 SQL 执行计划的大致窗口，让你判断这条 SQL 有没有走索引、走了哪个索引、扫描范围大不大、有没有额外代价。”

注意这里我特意说“大致”。
因为 explain 不是把执行过程一帧一帧录像给你看，而是告诉你：

“优化器准备怎么做。”

所以你这一阶段的目标不是成为调优专家，而是先学会看几个最关键的信息：
- key
- type
- rows
- Extra

如果这四个你能看明白，explain 就已经不是黑盒了。

## 二、先别背字段，先回答：为什么要看 explain？

你可以把 explain 的用途先记成 3 句话：

1. 看有没有走索引
2. 看走的是哪个索引
3. 看扫描代价大不大

比如同样是一条“查用户”的 SQL：
- 一种可能是只扫 1 行
- 一种可能是扫 500 行
- 一种可能是扫 50 万行

你从代码层面看，可能都只是一个 select。
但数据库内部代价差别非常大。

所以 explain 的价值，不是“告诉你 SQL 能不能执行”，而是：

“帮助你理解 SQL 的访问路径和大概成本。”

这句话要建立起来。
以后你看到慢 SQL，不要只问“这条 SQL 对不对”，而要问：

“它是怎么查的？扫了多少？有没有走上我预期的索引？”

## 三、先建立一个整体感觉：explain 在看什么

你可以把 explain 想成医生看体检报告。

它不是把你身体的每一个分子都展示出来，
而是给你几个关键指标，帮你快速判断：
- 有无明显问题
- 问题可能出在哪
- 下一步应该往哪看

对于你现阶段来说，最关键的四个指标是：
- key：最后用了哪个索引
- type：访问方式大概怎么样
- rows：预计要扫描多少行
- Extra：有没有额外开销或额外信息

只要你把这四个字段和“查询路径”联系起来，就会突然顺很多。

## 四、先讲最重要的一个：key

### 1）key 是什么

key 表示：

“优化器最终决定使用哪个索引。”

这是 explain 里最直观、最容易看懂、也是最适合你当前阶段先看的字段。

因为上一份我们一直在说：
索引决定访问路径。

那 key 本质上就是在告诉你：

“这次 SQL 选了哪条路。”

### 2）怎么理解 key

如果你看到：
- key = PRIMARY

往往表示用了主键索引。

如果你看到：
- key = idx_user_name

往往表示用了名为 idx_user_name 的索引。

如果你看到：
- key = null

通常说明没有使用索引。

这时候你就应该立刻产生几个追问：
- 是不是根本没建索引？
- 这条 SQL 写法是不是导致索引失效了？
- 数据量太小，优化器觉得全表扫更划算？
- 条件选择性太差，优化器不想走索引？

注意，不要把 “key = null” 机械理解成“SQL 一定有错”。
有时全表扫描确实是优化器判断后更划算的选择。

但如果你的表已经很大，而 key 还是 null，那就通常值得警惕。

### 3）你现在该怎么用 key

最简单的思考方式就是：

“我预期这条 SQL 应该走哪个索引？实际有没有走？”

如果预期和实际一致，说明你的索引设计和 SQL 写法大体没跑偏。
如果不一致，就要顺着查：
- 是索引没建对？
- 是字段顺序不对？
- 是 where 写法破坏了索引利用？
- 是 select 列太多，导致回表代价太高？

所以 key 最像什么？
最像“路由结果”。
它告诉你数据库最后选了哪条路。

## 五、第二个关键字段：type

### 1）type 是什么

type 表示访问类型，也可以理解成：

“数据库是以什么方式去找数据的。”

它比 key 稍微抽象一点，
但你现阶段完全不用把所有枚举值背下来。

你只需要先建立一个粗粒度判断：
- const / eq_ref / ref / range：通常还不错
- index：通常表示扫索引
- all：通常表示全表扫描，要重点警惕

### 2）为什么 type 很重要

因为 key 只能告诉你“走了哪个索引”，
但并不能完全告诉你“走得好不好”。

举个直觉例子：
- 两条 SQL 都可能用了索引
- 但一条可能是精准命中
- 另一条可能只是做了一个较宽的范围扫描

它们的代价差别是很大的。

type 就是在帮助你粗略判断：

“这次访问方式是比较精准，还是比较粗放？”

### 3）现阶段你最需要认识的几个 type

#### const

通常表示通过主键或唯一索引，把查询几乎当成常量一样定位。

你可以简单理解成：
“非常精准，通常只会命中极少量数据。”

例如：
select * from user where id = 1;

如果 id 是主键，这种通常就很理想。

#### ref

通常表示使用了普通索引进行等值匹配。

比如：
select * from user where name = 'tom';

如果 name 上有普通索引，常见会看到 ref。

它通常也不错，但没有 const 那么“唯一、精准”。

#### range

表示范围扫描。

例如：
- id > 100
- create_time between ... and ...
- age >= 18

range 不代表坏，很多业务场景本来就需要范围查询。
但你要知道：
它通常比精确等值匹配扫得更多。

#### index

这个比较容易误会。

index 通常表示扫描索引树本身。
它比 all 好一些，因为至少不是扫整张表的数据页，
但它并不一定意味着“效率高”。

有时它是因为在做覆盖索引扫描；
有时它可能是“整个索引都扫了一遍”。
所以看到 index，不要立刻兴奋，也不要立刻恐慌，要结合 rows 和 Extra 一起看。

#### all

这是你现阶段最需要敏感的。

all 通常表示全表扫描。
也就是说，数据库可能要把整张表一行行看过去。

如果表很小，也许问题不大。
但如果数据量大，all 往往意味着：
- 扫描范围大
- IO 成本高
- 性能风险高

所以你以后看 explain，看到 all，脑子里要先亮红灯。
不是一定错误，而是要重点检查。

### 4）你现在不需要死记顺序，但要有方向感

很多资料会让你背 type 从好到坏的完整顺序。
你现阶段没必要。

先记这个就够了：

“const / ref / range 通常还可以；all 通常表示代价比较高；index 要结合 rows 和 Extra 一起看。”

## 六、第三个关键字段：rows

### 1）rows 是什么

rows 表示：

“优化器预估这条 SQL 大概要扫描多少行。”

注意，是预估，不是绝对精确值。

但它仍然非常重要，
因为它和我们前面一直讲的“扫描范围”直接挂钩。

### 2）为什么 rows 是 explain 里特别有价值的字段

因为很多时候，SQL 慢不慢，本质上不是语法问题，而是代价问题。
而 rows 恰好就是一个很直观的代价信号。

比如：
- rows = 1
- rows = 20
- rows = 3000
- rows = 500000

就算你不懂所有 explain 细节，
你也应该能感受到：
后面两个通常比前面两个更危险。

所以 rows 非常像什么？
像“数据库觉得自己要翻多少页、看多少人”。

它越大，通常意味着数据库要付出的代价越高。

### 3）rows 大就一定慢吗

也不是绝对。

你要避免另一种机械化理解：
- 看到 rows 小就一定快
- 看到 rows 大就一定慢

真实情况是：
- rows 是重要信号，但不是唯一信号
- 还要结合表大小、key、type、Extra、缓存命中、返回列等一起判断

但对于你当前阶段来说，
rows 绝对是最值得优先观察的字段之一。

因为它能帮你建立“扫描量意识”。

### 4）一个很实用的判断习惯

以后看 explain，问自己一句：

“这条 SQL 真有必要扫这么多行吗？”

如果你心里觉得“不应该”，
那往往就说明：
- 索引没设计好
- 索引没命中好
- SQL 条件写法有问题
- 或者查询需求本身就太粗

这就是 rows 的价值。

## 七、第四个关键字段：Extra

### 1）Extra 是什么

Extra 可以理解成：

“除了前面的基本访问路径之外，还有没有额外信息或额外动作。”

它往往是 explain 里最容易让人觉得杂乱的字段。
因为里面可能出现很多不同的提示词。

但你现在别贪多，先认识几个最高频的就够：
- Using index
- Using where
- Using filesort
- Using temporary

### 2）Using index

这通常是个比较积极的信号。
很多时候它表示查询使用了覆盖索引。

也就是：
数据库要的列都能从索引里直接拿到，
不需要再回表。

所以看到 Using index，
你可以优先联想到：
“可能是覆盖索引，少了一次额外数据回取。”

### 3）Using where

表示 MySQL 还需要根据 where 条件做过滤。

这本身并不稀奇，很多正常查询都会有。
所以不要一看到 Using where 就觉得有问题。

你应该把它理解成：
“数据库取到候选数据后，还要继续按条件筛。”

### 4）Using filesort

这是很高频的一个优化信号。
它通常表示排序操作没能直接利用索引顺序，而是需要额外排序。

这就可能带来额外开销。

它不一定一定很慢，
但如果数据量大、排序频繁，就值得关注。

### 5）Using temporary

通常表示执行过程中使用了临时表。
在 group by、复杂排序、某些子查询场景里比较常见。

它也不代表一定错误，
但通常说明执行过程不够“顺”。
如果数据量大，也可能成为性能负担。

### 6）你该怎么对待 Extra

你现在先不要试图背全。
只需要把它当成“额外提示区”：

“这里会告诉你有没有覆盖索引、额外过滤、额外排序、临时表等附加代价。”

## 八、把四个字段串起来看，才真正有用

真正看 explain，不是孤立看某一个字段，
而是把它们连起来。

你可以用这套最简单的顺序：

第一步：看 key
- 用了哪个索引？
- 有没有用到我预期的索引？

第二步：看 type
- 是精准查找、范围扫描，还是全表扫描？

第三步：看 rows
- 预计要扫描多少行？
- 这个数量合理吗？

第四步：看 Extra
- 有没有覆盖索引？
- 有没有额外排序？
- 有没有临时表？

这四步一旦形成习惯，explain 就不再是黑盒，而是一张“访问路径草图”。

## 九、通过几个场景建立直觉

下面我不用你现在就记住所有细节，
而是先带你建立“看 explain 时脑子怎么转”。

### 场景 1：主键精确查询

SQL：
select * from user where id = 1;

如果 id 是主键，你大概率会期待：
- key: PRIMARY
- type: const
- rows: 很小，通常接近 1

你的直觉应该是：
“这是比较理想的查询路径，定位很精准。”

### 场景 2：普通索引等值查询

SQL：
select * from user where name = 'tom';

如果 name 上有普通索引，你可能看到：
- key: idx_name
- type: ref
- rows: 比较小，但不一定是 1

你的直觉应该是：
“走了索引，还不错；但如果 name 不是唯一字段，可能仍会扫多行。”

### 场景 3：范围查询

SQL：
select * from order_info where create_time >= '2026-01-01';

如果 create_time 上有索引，可能看到：
- key: idx_create_time
- type: range
- rows: 视范围大小而定

你的直觉应该是：
“它确实走了索引，但因为是范围查，不可能像主键等值那样精准。”

### 场景 4：没有命中索引

SQL：
select * from user where substring(name, 1, 3) = 'tom';

如果 name 上虽然有索引，但因为做了函数处理，可能会看到：
- key: null
- type: all
- rows: 很大

你的直觉应该是：
“索引大概率失效了，所以数据库只能扫很多数据。”

### 场景 5：覆盖索引查询

SQL：
select name, age from user where name = 'tom';

如果有联合索引 (name, age)，可能看到：
- key: idx_name_age
- type: ref
- Extra: Using index

你的直觉应该是：
“这可能是覆盖索引，不需要回表，所以路径更顺。”

## 十、别把 explain 看成“用了索引就万事大吉”

这是特别容易犯的错误。

很多人只看 key，不看别的。
一看到 key 不是 null，就说：
“用了索引，应该没问题。”

这其实很危险。

因为：
- 用了索引，不等于扫描范围小
- 用了索引，不等于没有回表
- 用了索引，不等于没有额外排序
- 用了索引，不等于代价就一定比全表更低

所以 explain 最重要的不是看“有没有索引”这一个点，
而是看：

“这条 SQL 的整体访问路径到底健不健康。”

## 十一、你现在最容易犯的几个误区

### 误区 1：看到 key 不是 null 就放心

不对。
还要继续看：
- type 是什么
- rows 大不大
- Extra 有没有 filesort / temporary

### 误区 2：看到 all 就觉得一定错

也不对。
如果表很小，或者查询本来就要取大量数据，
全表扫描未必一定是坏事。

但在大表高频查询里，all 确实值得重点关注。

### 误区 3：把 rows 当真实行数

rows 是优化器估算值，不是最终精确统计值。
但它依然有很强参考价值。

### 误区 4：想一次背全 explain 所有字段

不适合你现在的阶段。
你现在先掌握：
- key
- type
- rows
- Extra

已经能解决很多基础面试和初级分析场景。

## 十二、项目里怎么用 explain 说人话

你在项目表达里，不需要说得像数据库内核工程师。
你只需要会这样表达：

“我会先用 explain 看这条 SQL 的执行计划，重点关注有没有命中预期索引、访问类型是什么、预估扫描行数有多少，以及是否存在额外排序或临时表。这样可以快速判断性能瓶颈是在索引命中、扫描范围，还是排序/过滤等额外操作上。”

这句话非常实用。
因为它体现的是“分析框架”，不是死背字段。

## 十三、一个你现在就可以记住的 explain 观察模板

以后拿到一条 SQL，你就按这个顺序问自己：

1. 这条 SQL 我预期应该走哪个索引？
2. explain 里的 key 真的是那个索引吗？
3. type 是精准查找、范围查找，还是全表扫描？
4. rows 预估扫描了多少？这个数量合理吗？
5. Extra 里有没有 Using index、Using filesort、Using temporary？
6. 如果结果不符合预期，是索引设计问题，还是 SQL 写法问题？

如果你能按这个模板思考，说明 explain 已经真正变成你的工具了。

## 十四、面试版简答模板

### 1）explain 是干什么的？

“explain 用来查看 SQL 的执行计划。它可以帮助我们理解 MySQL 大概会怎么访问数据，比如有没有走索引、用了哪个索引、预计扫描多少行、是否存在额外排序或临时表等，从而辅助判断 SQL 的性能风险。”

### 2）看 explain 你最先看什么？

“我现阶段一般优先看 key、type、rows 和 Extra。key 看用了哪个索引，type 看访问方式，rows 看预估扫描量，Extra 看有没有覆盖索引、额外排序或临时表。”

### 3）rows 大说明什么？

“rows 表示优化器预估要扫描多少行。它不是精确值，但能很好反映扫描范围。rows 越大，通常说明 SQL 代价越高，也更值得进一步检查索引命中和查询写法。”

### 4）key 为 null 一定有问题吗？

“不一定，但通常说明没有使用索引。要结合数据量和场景判断：有可能是没建索引，也可能是 SQL 写法导致索引失效，或者优化器认为全表扫描更划算。”

## 十五、这一篇你先记住的最小骨架

如果你现在只想先记最核心的，记住下面这 7 句：

1. explain 是查看 SQL 执行计划的大致窗口
2. 它最重要的价值，是帮助判断访问路径和大概代价
3. key 看用了哪个索引
4. type 看访问方式粗不粗
5. rows 看预计扫描范围大不大
6. Extra 看有没有额外动作，比如覆盖索引、排序、临时表
7. 真正分析 explain，要把 key、type、rows、Extra 连起来看

- 为什么 rows 能反映性能风险

如果这些能说顺，说明你已经从“看不懂 explain”进到“能用 explain 做初步分析”了。

### 1. `LIKE` 模糊查询

**常见 SQL：**

SQL

```
SELECT * FROM users WHERE name LIKE '张%';
```

- **可能走的索引：** 如果 `name` 字段上有普通索引（如 `idx_name`），这条 SQL 会走该索引（范围扫描 Range）。
    
- **失效情况（不走索引）：** * **左模糊或全模糊：** 当 SQL 写成 `LIKE '%张'` 或 `LIKE '%张%'` 时，索引会失效，转为全表扫描。因为 B+ 树索引是按照字符从左到右排序的，前缀不确定就无法在树中进行搜索。
    
    - **特例：** 如果查询的列恰好被索引完全覆盖（比如只 `SELECT name, id FROM users`），优化器可能会选择扫描整个索引树（Index 级别扫描），这比全表扫描稍快，但依然不是精准匹配。
        

### 2. 联合索引（最左前缀匹配）

**假设场景：** 表 `orders` 上有一个联合索引 `idx_user_status_date (user_id, status, create_date)`。

**常见 SQL：**

SQL

```
SELECT * FROM orders WHERE user_id = 1001 AND status = 'PAID';
```

- **可能走的索引：** 完美命中联合索引 `idx_user_status_date` 的前两个字段。
    
- **失效情况（不走索引/部分失效）：**
    
    - **跳过首列（完全失效）：** `WHERE status = 'PAID' AND create_date = '2023-10-01'`。因为没有包含最左边的 `user_id`，B+ 树无法定位，导致全表扫描。
        
    - **范围查询截断（部分失效）：** `WHERE user_id > 1000 AND status = 'PAID'`。此时 `user_id` 可以走索引（范围查询），但在其右侧的 `status` 字段无法继续走索引树去过滤，只能在提取数据后进行回表计算（MySQL 5.6 引入的索引下推 ICP 可以优化这一步，但树结构的搜索依然在第一列就停止了）。
        

### 3. 隐式类型转换（非常隐蔽的杀手）

**假设场景：** 表 `users` 中 `phone` 字段的类型是 `VARCHAR(20)`，且建有索引。

**常见 SQL：**

SQL

```
SELECT * FROM users WHERE phone = '13800138000';
```

- **可能走的索引：** 正常命中 `phone` 字段的字符串索引。
    
- **失效情况（不走索引）：**
    
    - **传入了数字类型：** 如果 SQL 写成 `WHERE phone = 13800138000`（没有单引号）。MySQL 在遇到字符串和数字比较时，会自动将**字符串转换为数字**。这等价于在字段上套了一个函数：`CAST(phone AS signed) = 13800138000`。由于对索引列进行了内部函数运算，导致索引失效，触发全表扫描。
        

### 4. 对索引列进行函数或数学运算

**假设场景：** `create_time`（DATETIME 类型）和 `age`（INT 类型）都有独立索引。

**常见 SQL：**

SQL

```
SELECT * FROM users WHERE create_time >= '2023-01-01 00:00:00' AND create_time < '2023-01-02 00:00:00';
```

- **可能走的索引：** 走 `create_time` 的范围索引。
    
- **失效情况（不走索引）：**
    
    - **使用函数：** 如果为了省事写成 `WHERE DATE(create_time) = '2023-01-01'`。
        
    - **数学运算：** `WHERE age + 1 = 18`。
        
    - **原因：** B+ 树中存储的是列的原始值，一旦你对列进行了加工（运算或函数），原有的树结构就无法匹配新结果了。**正确做法永远是把列“干净”地留在等号左边**（例如改成 `WHERE age = 18 - 1`）。
        

### 5. `OR` 条件查询

**常见 SQL：**

SQL

```
SELECT * FROM users WHERE id = 100 OR age = 25;
```

- **可能走的索引：** 如果 `id` 有主键索引，且 `age` 也有独立索引，MySQL 优化器通常会触发 **Index Merge（索引合并）** 机制，分别查出两部分数据然后再合并。
    
- **失效情况（不走索引）：**
    
    - **包含无索引列：** 假设 `age` 字段**没有**建立索引。此时即使 `id` 是主键，为了找到 `age = 25` 的数据，数据库依然必须进行全表扫描。既然反正都要全表扫描一遍了，优化器就会直接放弃走 `id` 索引，导致整条语句完全不走索引。
        

---

> **核心提示：** > 索引“失效”并不总是 SQL 写错了。如果查询优化器评估后认为 **“通过索引查找 + 回表” 的成本大于 “直接全表扫描”**（例如你的查询条件会返回表中 30% 以上的大量数据），它也会主动放弃使用索引。
> 
> 验证 SQL 到底走没走索引，唯一绝对可靠的标准是在 SQL 前面加上 **`EXPLAIN`** 关键字查看执行计划。
