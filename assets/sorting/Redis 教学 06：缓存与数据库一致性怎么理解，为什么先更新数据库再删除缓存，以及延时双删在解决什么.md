## 一、这节课为什么要学

前面几节你已经逐步把 Redis 在系统里的定位理顺了：
- Redis 为什么快
- Redis 常见数据结构怎么和业务场景对应
- 为什么 Redis 有持久化，但核心数据通常还是放在 MySQL
- 缓存穿透、击穿、雪崩分别是什么，以及它们为什么最后都会变成数据库扛压问题

顺着这条学习路线，下一步非常自然就会遇到一组高频追问：
- Redis 和数据库一致性问题怎么理解
- 为什么经常说“先更新数据库，再删除缓存”
- 延时双删到底是在解决什么问题

这组问题在面试里非常常见。
而且它和上一节不一样：

上一节更偏“缓存失效后，数据库会不会被打爆”；
这一节更偏“缓存参与读写后，数据会不会短暂不一致”。

也就是说：
- 上一节偏性能与抗压
- 这一节偏更新路径与一致性

如果你只背结论，不理解背后的请求时序，一到面试官追问“为什么”时就很容易乱。

所以这一节的目标不是让你死记几个套路名词，
而是把下面这条主线真正讲顺：

“只要系统里同时维护数据库真值和缓存副本，读写并发下就天然可能出现短暂不一致；工程上做的，不是追求神话式绝对一致，而是选择更稳妥的更新顺序，并尽量缩小不一致窗口。”

---

## 二、先建立总视角：为什么一有缓存，就会冒出“一致性”问题

### 1. 没有缓存时，数据关系很单纯

如果系统只有 MySQL，逻辑相对简单：
- 读，去 MySQL 读
- 写，去 MySQL 写

这时候虽然数据库内部仍然有并发问题，要靠事务、锁、MVCC 等机制处理，
但至少从“系统里有几份数据副本”这个角度看，关系是比较单纯的。

因为真正对外提供数据的核心来源，主要就是 MySQL。

### 2. 有了缓存后，系统里其实变成了“两份数据”

一旦引入 Redis 做缓存，情况就不一样了。

这时候往往变成：
- MySQL 里有一份数据
- Redis 里也可能有一份数据

你可以这样理解：
- MySQL 更像最终真实数据
- Redis 更像为了加速读取，提前放在前面的副本

只要业务发生更新，一个天然问题就出现了：

我应该先改谁？
改完另一份什么时候同步？
同步失败怎么办？
同步过程中来了读请求，会看到旧值还是新值？

这就是缓存与数据库一致性问题真正的来源。

### 3. 一致性问题的本质不是“出 bug”，而是“两份数据 + 并发”

很多初学者一听“一致性问题”，会本能地理解成：
“是不是 Redis 出故障了？”

其实不是。

一致性问题最本质的来源是：
- 系统里有两份数据
- 更新动作不是原子地同时完成
- 并且还有并发读写

只要满足这三个条件，就天然可能在某个时刻出现：
- 数据库已经是新值
- 缓存还是旧值

或者反过来，某些方案里也可能出现：
- 缓存是新值
- 数据库还没改成功

所以你要记住：
缓存一致性问题本质上不是某种偶发异常，
而是“系统设计上天然存在的时序问题”。

## 三、先用一个最小例子把一致性问题看清楚

假设商品 price 原来是 100。

此时：
- MySQL 里是 100
- Redis 缓存里也是 100

现在业务把价格改成 120。

### 情况 1：数据库先改成 120，但缓存里还是 100

这时如果有用户来读，并且刚好读的是缓存，
那他看到的还是 100。

这就说明：
- MySQL 已经新了
- Redis 还没新

这是一种典型不一致。

### 情况 2：缓存先改成 120，但数据库更新失败了

这时如果系统没有做好兜底，就会出现：
- Redis 是 120
- MySQL 还是 100

这也是不一致。

而且从系统最终真实落地的角度看，这种情况通常更危险，
因为数据库才是多数业务里的最终数据来源。

### 3. 所以“一致性”到底在问什么

所谓缓存与数据库一致性，真正想问的是：

当一条数据被修改时，
怎么安排数据库和缓存的更新顺序，
才能让系统出现错误结果的概率更低，
并把不一致持续的时间尽量缩短？

你会发现，这个问题本质上不是单纯问“谁对谁错”，
而是在问：

“面对并发与失败，哪种更新路径更稳妥？”

## 四、先明确一个工程前提：为什么很多系统不是“更新缓存”，而是“删除缓存”

在讨论“先更新数据库，再删除缓存”之前，你要先明白：

为什么大家经常说的是删缓存，
而不是更新缓存？

### 1. 典型缓存使用方式：Cache Aside

很多业务系统采用的是一种很常见的思路：

读：
1. 先查缓存
2. 缓存没有，再查数据库
3. 把数据库结果回填到缓存

写：
1. 更新数据库
2. 删除缓存

这种模式的核心思想是：
缓存主要服务读请求，
而不是每次写请求都强行精细维护一份完整缓存状态。

### 2. 为什么不总是“写数据库时顺便更新缓存”

因为这样会带来一些额外问题：
- 有些数据改完后，后面未必很快有人读，提前更新缓存可能白做了
- 多个业务入口都要负责维护缓存，逻辑容易变复杂
- 如果缓存结构不是数据库原样映射，而是聚合结果、计算结果，更新缓存会更麻烦
- 写路径本来就要保证正确性，再加缓存更新逻辑，失败场景更多

所以在很多场景里，工程上更常见的选择是：

“写操作只保证数据库正确落地；缓存不主动精确更新，而是删掉它，让下一次读请求按新数据重新加载。”

这就是“删缓存而不是更缓存”的背景。

---

## 五、为什么经常说“先更新数据库，再删除缓存”

这是这一节最核心的问题之一。

你不要把它背成教条，
而是要真正理解：

这个顺序不是因为它完美，
而是因为和其他顺序相比，它通常风险更低。

### 1. 先看一种直觉方案：先删缓存，再更新数据库

很多人第一反应会觉得：
“要改数据了，那我先把缓存删掉，不就行了吗？”

听起来很合理，
但并发场景下，这个顺序有一个非常经典的问题。

假设初始状态：
- DB = A
- Cache = A

线程 T1 要把数据改成 B。

如果它采用“先删缓存，再更新数据库”：

第 1 步：T1 删除缓存
第 2 步：此时线程 T2 恰好来读，发现缓存没了
第 3 步：T2 去查数据库，但这时数据库还没被 T1 改完，查到的还是旧值 A
第 4 步：T2 把旧值 A 回填到缓存
第 5 步：T1 再把数据库更新成 B

最后结果变成：
- DB = B
- Cache = A

也就是说，
线程 T2 在并发窗口里，把旧值重新灌回了缓存。

这就是为什么“先删缓存，再更新数据库”风险很大。

### 2. 它的问题到底出在哪

问题不是“删缓存”这个动作本身，
而是：

当你先删缓存后，
在数据库真正改完之前，会出现一个空窗期。

在这个空窗期里：
- 任何读请求看到缓存 miss
- 都会去数据库查
- 而数据库里此时可能还是旧值
- 旧值就可能被重新写回缓存

所以它最危险的点在于：

“为旧值重新进入缓存创造了机会。”

### 3. 再看常见方案：先更新数据库，再删除缓存

还是同样的初始状态：
- DB = A
- Cache = A

线程 T1 要把它改成 B。

如果采用“先更新数据库，再删除缓存”：

第 1 步：T1 先把数据库改成 B
第 2 步：T1 再删除缓存

这时候风险为什么更小？

因为只要缓存被删掉以后，再来的读请求去数据库查，查到的大概率已经是新值 B。

也就是说：
后续如果发生回填，
回填进去的通常也是新值，而不是旧值。

### 4. 这套顺序真正想避免什么

“先更新数据库，再删除缓存”真正想避免的是：

缓存 miss 之后，读请求去数据库回源时，读到的还是旧值，进而把旧值重新写回缓存。

换句话说，它要解决的是：
“降低旧值脏回填的概率。”

### 5. 为什么它不是绝对完美，但仍然最常见

因为它虽然不是 100% 无问题，
但它把最常见、最危险的一类并发时序风险压低了。

你可以这样比较：
- 先删缓存再更新数据库：更容易出现旧值回填缓存
- 先更新数据库再删缓存：旧值回填的概率通常更低

工程上很多时候不是选“绝对正确且零成本”的方案，
而是选“风险更低、成本更可控、实现更简单”的方案。

这就是它被广泛采用的原因。

## 六、但一定要知道：先更新数据库，再删除缓存，也不是银弹

这一点面试里最好主动补一句，
这样你的回答会显得更成熟。

### 1. 最典型的问题：数据库改成功了，但删缓存失败了

还是初始：
- DB = A
- Cache = A

线程 T1 执行：
1. 更新数据库为 B，成功
2. 删除缓存，失败

结果就变成：
- DB = B
- Cache = A

这时缓存里旧值还在。
后面的读请求如果继续命中这个旧缓存，
就会持续读到旧数据。

### 2. 所以它解决的是“顺序风险”，不是“所有失败风险”

你要特别分清：

“先更新数据库，再删除缓存”主要在解决并发顺序上的问题，
但它并不能自动解决：
- 删除缓存失败
- 删除请求丢失
- 网络抖动
- Redis 瞬时不可用
- 应用刚更新完数据库就宕机

所以工程上通常还会配合一些补救措施：
- 删除缓存失败重试
- 异步消息补偿
- 订阅 binlog 再做缓存失效
- 给缓存设置过期时间兜底

也就是说：
这个方案很常见，
但它从来不是“做了就万无一失”。

## 七、那延时双删到底是在解决什么问题

### 1. 先直接给结论

延时双删要解决的核心问题是：

“第一次删缓存之后，在数据库更新完成前后的并发窗口里，旧值可能又被读请求重新写回缓存。”

所以它的重点不是“再删一次看起来更保险”，
而是专门针对：

“旧值脏回填”

这个竞态条件做补丁。

### 2. 为什么会出现“第二次删除”这种想法

我们先看一个常见思路：

有些人采用的是：
1. 先删缓存
2. 再更新数据库

但前面你已经看到了，
这个顺序在并发读写下容易出现：
- 读请求发现缓存没了
- 去数据库查到旧值
- 再把旧值写回缓存

于是有人就会想：

“那我第一次先删掉缓存，写完数据库后，过一会儿再删一次缓存，不就能把刚才被误写回去的旧值再清掉了吗？”

这就是延时双删背后的直觉来源。

### 3. 用最小时序看懂它

假设初始状态：
- DB = A
- Cache = A

写线程 T1 要改成 B。
读线程 T2 在并发访问。

如果采用延时双删：

第 1 步：T1 先删除缓存
第 2 步：T2 读缓存，发现 miss
第 3 步：T2 去数据库查，但此时 T1 还没完成数据库更新，所以 T2 查到旧值 A
第 4 步：T2 把旧值 A 回填到缓存
第 5 步：T1 把数据库更新成 B
第 6 步：T1 休眠一小段时间后，再次删除缓存

这样第 6 步就能把第 4 步里被错误回填进去的旧值 A 再清掉。

这就是延时双删最想解决的问题。

### 4. 所以它到底在补哪一刀

你可以把它理解成：

第一次删缓存：
是为了让旧缓存先失效，给写操作让路。

延时后的第二次删缓存：
是为了把并发窗口里可能被错误回填进去的旧缓存，再清掉一次。

也就是说，它补的是：
“旧值在并发窗口里重返缓存”这件事。

---

## 八、延时双删为什么不是严格意义上的完美方案

### 1. 它非常依赖“延时多久”

延时双删最尴尬的一点就在这里：

第二次删除到底延时多久合适？

如果延时太短：
- 可能读请求还没完成数据库查询和旧值回填
- 第二次删除已经执行完了
- 后面旧值依然可能被写回缓存

如果延时太长：
- 旧值在缓存里存在的时间就更久
- 系统不一致窗口可能反而被拉长

也就是说，
这个“延时”并没有一个理论上通吃所有场景的固定答案。

### 2. 它更像工程补丁，不像严格证明正确的方案

延时双删可以在很多场景下起到缓解作用，
但它本质上不是一个可以严格证明“完全消除不一致”的方案。

它更像是在说：

“我知道有一个高风险并发窗口，那我在经验上再补删一次，尽量把旧值残留概率降下来。”

所以你最好把它定位成：
- 一种工程上的缓解手段
- 针对特定竞态问题的补丁
- 而不是最终万能答案

### 3. 它也不能解决所有类型的不一致

比如这些问题，延时双删就不一定能根治：
- 删除命令本身失败
- 应用执行到一半宕机
- 多服务实例下时钟和调度不稳定
- Redis 本身不可用
- 更复杂的多 key 关联更新问题

所以面试里讲到这里时，你可以顺手补一句：

“延时双删主要是缓解并发时旧值回填问题，不是解决所有缓存一致性问题的银弹。”

这样会显得你不是在背八股，而是在理解边界。

## 九、那现在更成熟的工程理解应该是什么

你现在最应该建立的是下面这层认知：

### 1. 缓存和数据库很难追求绝对实时强一致

因为只要：
- 有两份数据
- 有网络
- 有并发
- 有失败可能

就很难用极低成本做到严格强一致。

如果真的要追求极高等级的一致性，
往往就要牺牲：
- 性能
- 可用性
- 系统复杂度

而缓存本来就是为了性能服务的。

所以大多数普通业务系统追求的目标并不是：
“缓存和数据库任何时刻都绝对一致。”

而是：
“允许短暂不一致，但要控制在可接受范围内，并尽量缩短窗口。”

### 2. 工程上更常见的是“最终一致 + 风险控制”

也就是说，大家更在意的是：
- 正常路径下，大部分请求读到正确值
- 出现不一致时，持续时间不要太长
- 异常场景下有补偿机制
- 热点数据和关键数据做重点保护

从这个角度看：
- 先更新数据库，再删除缓存，是常见主线
- 重试、补偿、TTL，是兜底手段
- 延时双删，是某类问题下的补丁方案

这才是比较完整的工程视角。

## 十、用一个最小业务案例把三者串起来

假设你做的是商品详情页缓存。

缓存 key：product:1001

初始时：
- MySQL 中 product:1001 的 price = 100
- Redis 中 product:1001 的 price = 100

现在运营修改商品价格为 120。

### 场景 1：先删缓存，再更新数据库

可能出现：
1. 写线程删掉缓存
2. 读线程发现缓存 miss
3. 读线程查数据库，此时数据库还是 100
4. 读线程把 100 回填缓存
5. 写线程再把数据库改成 120

最后变成：
- DB = 120
- Cache = 100

### 场景 2：先更新数据库，再删除缓存

大概率会变成：
1. 写线程先把数据库改成 120
2. 写线程删缓存
3. 后续读线程缓存 miss，再去数据库查，查到 120
4. 回填缓存 120

最后变成：
- DB = 120
- Cache = 120

当然，如果第 2 步删缓存失败，还是可能有问题。
但至少从并发时序角度看，它更稳妥。

### 场景 3：延时双删

如果业务使用某种需要双删补偿的路径，可能会变成：
1. 先删缓存
2. 更新数据库为 120
3. 延时一小段时间
4. 再删一次缓存

这里第二次删除的目的，不是因为“删两次显得更高级”，
而是为了把并发窗口里可能被读线程重新塞回缓存的旧值再清掉。

## 十一、你在面试里最容易说错的几个点

### 1. 不要把一致性问题说成“Redis 不可靠”

真正的问题是：
两份数据副本在并发更新下天然有时序差。

### 2. 不要把“先更新数据库，再删除缓存”说成绝对无问题

更准确的说法是：
它是工程上更常用、总体风险更低的方案。

### 3. 不要把延时双删说成万能解法

更准确的说法是：
它主要针对旧值回填问题，是一种工程补丁，不是银弹。

### 4. 不要忽视“失败场景”

如果面试官追问，你最好知道：
- 删缓存可能失败
- 失败后要有重试或补偿
- TTL 也常作为最终兜底

这样回答会更完整。

## 十二、面试里怎么回答这三个问题

### 1）如何理解 Redis 和数据库一致性问题？

你可以这样说：

“缓存和数据库一致性问题，本质上是因为数据库里有真实数据，缓存里有一份用于加速读取的副本。只要发生写操作，两份数据就不可能天然同时完成更新，再叠加并发读写，就可能在某个时间点出现数据库是新值、缓存还是旧值的情况。所以工程上通常不是追求缓存和数据库绝对实时强一致，而是尽量选择更稳妥的更新路径，并缩小不一致窗口。”

### 2）为什么经常说先更新数据库，再删除缓存？

你可以这样说：

“因为如果先删缓存再更新数据库，在并发场景下，读请求可能在缓存 miss 后去数据库读到旧值，再把旧值写回缓存，导致缓存和数据库不一致。先更新数据库再删除缓存，至少能降低旧值被重新回填到缓存的概率，所以这是工程上更常见的选择。不过它也不是绝对完美，删除缓存失败时仍然需要重试、补偿或 TTL 兜底。”

### 3）延时双删在解决什么问题？

你可以这样说：

“延时双删主要是在解决并发读写下旧值可能被重新回填到缓存的问题。第一次删缓存是为了让旧缓存失效，第二次延时删除是为了清理在数据库更新前后并发窗口里，被读请求错误回填进去的旧缓存。它更像一种工程补丁，主要是缓解特定竞态问题，而不是解决所有缓存一致性问题的银弹。”

---

## 十三、把这节课压缩成 3 句最重要的话

你现在最需要记住的是这三句：

1. 一致性问题的本质：两份数据 + 并发更新，天然可能短暂不一样。
2. 先更新数据库，再删除缓存：不是因为完美，而是为了降低旧值回填缓存的风险。
3. 延时双删：是在补救并发窗口里旧值被重新写回缓存的问题。

如果这三句你能真正理解，
这组 Redis 面试题就不再只是“背过”，而是能讲顺逻辑了。

---

## 十四、给你两个最小问答练习

### 1. 为什么缓存和数据库很难做到完全强一致？
参考回答：
因为系统里存在数据库真值和缓存副本两份数据，更新动作无法天然原子完成，再加上并发、网络和失败场景，就很难低成本做到任何时刻绝对一致，所以工程上通常追求的是最终一致和不一致窗口可控。

### 2. 延时双删为什么不是万能方案？
参考回答：
因为它非常依赖第二次删除的延时时机，只能缓解某些并发窗口里的旧值回填问题，不能严格解决所有失败场景，比如删缓存失败、服务宕机、Redis 不可用等，所以它更像一种经验性补丁。
