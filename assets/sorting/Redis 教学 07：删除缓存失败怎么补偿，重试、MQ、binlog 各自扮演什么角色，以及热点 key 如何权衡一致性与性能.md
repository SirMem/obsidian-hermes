## 一、这节课为什么要学

上一节你已经把缓存一致性的基础主线先搭起来了：
- 为什么缓存和数据库天然可能短暂不一致
- 为什么工程上常说“先更新数据库，再删除缓存”
- 延时双删到底在补什么问题

但如果只停在这里，你对这个话题的理解还停留在“概念题”层面。

因为真实项目里，面试官常常会继续追问：
- 如果删除缓存失败了怎么办？
- 只靠代码里直接删一次，够不够？
- 为什么有些系统会引入 MQ？
- 为什么有些团队会提到 binlog 同步缓存？
- 热点 key 如果特别敏感，是优先保性能，还是优先保一致？

这些问题一追，话题就从“更新顺序”进入了“工程补偿机制”。

所以这一节的目标，不是让你把所有复杂方案背成八股，
而是让你建立一个更真实的工程视角：

“缓存一致性不是单靠某一个动作解决的，而是主路径 + 失败补偿 + 异步修正 + 热点治理，一起构成的。”

如果你把这层认知建立起来，后面再学更复杂的架构设计，就会顺很多。

---

## 二、先建立总视角：为什么“删除缓存失败”会成为一个独立问题

### 1. 先更新数据库，再删除缓存，只是主路径

上一节我们讲过，很多系统采用的是：
1. 先更新数据库
2. 再删除缓存

这个方案之所以常见，是因为它能降低并发场景下旧值回填缓存的风险。

但你要注意：
这里的“能降低风险”，并不等于“后面就没有问题了”。

因为整个动作里，仍然有很多失败点：
- 数据库更新成功了，但删缓存超时
- Redis 短暂不可用
- 应用服务在删缓存前宕机
- 删缓存请求发出去了，但网络抖动导致没有真正成功
- 多实例部署下，某次失败没有被后续流程补救

于是就会出现一种很典型的不一致：
- DB 已经是新值
- Cache 里还是旧值

这就是为什么“删除缓存失败”必须单独被认真对待。

### 2. 本质上，它是在问：主路径失败后，系统怎么自我修正

你可以把这个问题理解成：

主路径：
是“正常情况下怎么做”。

补偿机制：
是“正常路径某一步失败后，系统怎么把自己拉回来”。

缓存一致性在工程上，真正比拼的往往不是：
“你的主流程是不是大家都知道的先更库再删缓存”，
而是：
“主流程失手时，你有没有后手。”

这就是这一节要学的重点。

## 三、删除缓存失败时，工程上一般怎么补偿

先直接给你一个总答案：

删除缓存失败后的补偿思路，核心就是三类：
1. 立刻重试
2. 异步补偿
3. 最终兜底

你可以把它理解成三层保险。

### 1. 第一层：同步重试

最直接的想法是：

“这次删缓存没成功，那我马上再试几次。”

这是最基础、也是很多系统第一层就会做的事情。

比如：
- 第一次删除失败
- 立即重试 2~3 次
- 每次间隔几十毫秒到几百毫秒

这样做的原因很简单：
很多失败并不是永久性故障，而是瞬时抖动。
比如：
- Redis 瞬时负载高
- 网络抖动
- 连接池短暂异常

在这种情况下，重试往往就能成功。

#### 这一层的优点
- 简单
- 实现成本低
- 对瞬时失败有效

#### 这一层的边界
- 如果应用线程已经准备返回，不能无限阻塞在这里
- 如果 Redis 持续不可用，重试也无济于事
- 只靠同步重试，无法覆盖应用已经崩溃的情况

所以你要记住：
同步重试是第一层保险，但不是全部。

### 2. 第二层：异步重试 / 补偿任务

如果同步删失败了，很多系统不会就此放弃，
而是会把这次失败记录下来，交给后台异步任务继续补删。

你可以把它理解成：

“当前请求线程先尽量做，做不成的，后面有专门的人继续补。”

常见做法包括：
- 记录一条失败日志到表里
- 写入一个待补偿任务表
- 投递到延迟队列或补偿队列
- 由定时任务扫描失败记录后重试删除

#### 这一层的价值
它解决的是：

“当前线程删不掉，但系统不能因此永远放弃修正缓存。”

这时候，请求可以先结束，
但系统后台仍然保留继续恢复一致性的能力。

#### 一个最小案例
假设商品价格更新成功了，但删缓存失败。
系统此时可以：
1. 先把商品价格更新到数据库
2. 删除缓存失败后，记录一条“product:1001 删除失败”的补偿任务
3. 后台任务稍后重试删除 Redis 中的 product:1001

这样虽然中间有短暂不一致，
但系统还有机会把旧缓存清掉，恢复到正确状态。

### 3. 第三层：TTL 兜底

很多缓存本身会设置过期时间。
这件事除了控制内存和数据生命周期，
还有一个很重要的副作用：

即使缓存删除失败了，旧数据也不会永远存在。

也就是说，TTL 在一致性里常常扮演的是：
“最终兜底的自动失效机制。”

你不要把它理解成主方案，
而应该理解成：

“即使前面都失手，最差也让旧数据在有限时间后自然过期。”

#### TTL 的优点
- 简单有效
- 不依赖额外补偿链路也能最终清理旧值

#### TTL 的问题
- 旧值会在一段时间内持续对外可见
- TTL 越长，不一致窗口越长
- TTL 越短，又可能增加缓存重建压力

所以 TTL 更像底线保护，
不是高质量一致性的唯一手段。

### 4. 所以工程上一般是“多层补偿一起上”

你现在要建立的，不是“只靠哪一个技巧”，
而是这个组合认知：

- 主流程：先更新数据库，再删除缓存
- 第一层补偿：同步重试
- 第二层补偿：异步补偿 / 定时任务 / 延迟队列
- 第三层兜底：TTL 自然过期

这才是很多系统里更真实的工程样子。

## 四、缓存一致性里，重试、消息队列、binlog 分别在扮演什么角色

这一部分是面试里非常容易被问到的。

你不要把它们理解成彼此互斥的方案，
更准确地说，它们是在不同层面上承担不同职责。

### 1. 重试在扮演什么角色

重试扮演的是：

“对瞬时失败做第一时间修复。”

它通常离业务代码最近，
也是最直觉的一层处理。

#### 它适合解决什么
- Redis 瞬时抖动
- 网络短暂异常
- 某次命令超时但后续恢复正常

#### 它的特点
- 快
- 简单
- 成本低
- 适合兜住短时失败

#### 它不适合单独承担什么
- 服务进程已经挂掉
- 长时间 Redis 不可用
- 需要跨系统可靠传递“删除缓存”这个意图

所以重试更像第一反应，
不是最终闭环。

### 2. 消息队列在扮演什么角色

消息队列在缓存一致性里，常常扮演的是：

“把缓存失效这件事，从同步动作变成一个可异步、可重放、可解耦的事件。”

这句话很重要。

因为直接在业务代码里删缓存，属于：
- 当前线程自己做
- 成功失败都压在当前请求里

而引入 MQ 后，可以变成：
1. 业务先完成数据库更新
2. 发送一条“某个 key 需要失效”的消息
3. 消费者异步执行删除缓存
4. 如果消费失败，还可以重试、重投、告警

#### MQ 的价值 1：异步补偿
如果主线程里删缓存失败，
消息队列可以承接后续删除动作。

#### MQ 的价值 2：解耦
业务服务不需要自己处理所有复杂补偿逻辑，
而是把“缓存失效”作为一个事件交出去。

#### MQ 的价值 3：可重试、可堆积、可观测
如果消费者失败：
- 可以重新消费
- 可以进入死信队列
- 可以做监控告警

这让“删除缓存”从一次脆弱的即时调用，
变成一个更可治理的异步流程。

#### 但你要知道 MQ 的边界
MQ 不是魔法。
它也会带来：
- 消息发送失败
- 消息重复消费
- 顺序问题
- 业务实现复杂度上升

所以它更适合：
- 对一致性要求更高
- 写路径较复杂
- 需要更强补偿能力
的系统。

### 3. binlog 在扮演什么角色

binlog 扮演的角色，和重试、MQ 又有点不一样。

你可以把它理解成：

“让缓存失效动作，不再依赖业务代码是否顺利走完整条链路，而是直接根据数据库真实发生过的变更来驱动。”

这是一个非常重要的工程思想升级。

#### 为什么会想到 binlog
因为如果缓存失效完全依赖应用代码自己处理，
就会有很多风险：
- 代码没执行到那一步
- 应用宕机了
- 中途异常了
- 某个服务实现不统一

于是有人会想：

“既然真正的真值变化是发生在数据库里的，那我能不能以数据库最终提交的事实为准，再去触发缓存失效？”

这就是 binlog 方案的直觉来源。

#### binlog 方案在做什么
大致思路是：
1. 业务更新数据库
2. 数据库产生 binlog
3. 某个订阅程序（如 Canal、CDC 服务等）消费这条变更
4. 根据变更内容定位应该删除哪些缓存 key
5. 再异步删除对应缓存

#### 它的核心价值
binlog 本质上是在扮演：

“基于数据库真实变更事实的异步修正来源。”

这意味着：
- 只要数据库真的改成功了
- 就有机会通过 binlog 感知到这次变更
- 从而再去补删缓存

#### 它解决了什么核心痛点
它主要在解决：

“不要把缓存一致性的修复能力完全绑死在业务代码的一次调用上。”

#### 它的代价
- 系统复杂度更高
- 要维护 CDC / binlog 消费链路
- 要做好 key 映射关系
- 一致性仍然不是实时强一致，而是更稳的最终一致

所以 binlog 更常见于：
- 业务体量较大
- 多服务写库
- 对一致性修正闭环要求较高
- 团队有能力维护中间件链路
的系统。

## 五、把三者并排看，你就更不容易混

你可以用下面这组对比来记：

### 1. 重试
关键词：立刻补救

它更像：
“这次调用失败了，我马上再试一下。”

适合：
- 瞬时失败
- 简单场景
- 做第一层保险

### 2. 消息队列
关键词：异步补偿、解耦、可重放

它更像：
“把缓存失效变成一个可独立处理的事件，后面慢慢补。”

适合：
- 需要异步处理
- 需要失败重试和监控
- 系统链路较复杂

### 3. binlog
关键词：以数据库真实变更为准

它更像：
“只要数据库真的变了，就从数据库变更事实出发，再去修缓存。”

适合：
- 追求更稳的最终一致闭环
- 多服务共享数据库
- 不希望一致性修复完全依赖业务代码

一句压缩总结：
- 重试：补一次调用失败
- MQ：把补偿流程事件化、异步化
- binlog：把补偿依据建立在数据库真实变更事实之上

## 六、那热点 key 为什么会让一致性问题更敏感

前面讲的一致性方案，放到普通 key 上已经够你理解主线了。

但在真实系统里，最麻烦的往往不是普通 key，
而是热点 key。

所谓热点 key，你可以先简单理解成：
- 访问量特别高
- 被大量请求频繁读取
- 一旦失效就可能引发明显抖动
的 key。

比如：
- 首页商品详情
- 热门活动配置
- 秒杀库存摘要
- 首页推荐结果
- 高频公共配置项

### 1. 为什么热点 key 让一致性更敏感

因为热点 key 一旦被删除，
会立刻面对两类矛盾：

#### 矛盾一：你想让缓存尽快失效，确保读到新值
这是“一致性”的诉求。

#### 矛盾二：你又不想让大量请求同时打到数据库
这是“性能和抗压”的诉求。

普通 key 删除了，
后面就算有几次回源查数据库，问题也不大。

但热点 key 不一样。
一旦它失效，瞬间可能就是成百上千甚至更高并发一起回源。

所以热点 key 让你不得不面对一个更尖锐的工程权衡：

“到底是优先让缓存立刻失效保证新鲜度，还是允许短时间旧值继续存在，以换取系统稳定？”

这就是热点 key 下，一致性与性能的核心矛盾。

## 七、热点 key 在一致性和性能之间怎么权衡

### 1. 先给一个总原则

热点 key 的治理，本质上不是问：
“绝对一致和绝对高性能谁更好？”

而是问：
“这个 key 的业务含义，允许多大程度的旧数据，代价分别是什么？”

也就是说，先看业务，再定策略。

### 2. 不是所有热点 key 都值得追求极致新鲜

比如：
- 首页推荐列表延迟几秒更新，很多业务能接受
- 热门文章阅读数延迟一点，问题不大
- 活动页的一些展示字段短时间旧一点，通常也能接受

但有些数据就不一样：
- 库存
- 限购状态
- 用户资格
- 价格
- 风控标记

这些数据如果旧了，可能直接带来业务错误。

所以第一层权衡就是：

“这个热点 key 对业务正确性的影响，到底有多大？”

### 3. 常见权衡策略 1：允许短暂旧值，换系统稳定

对于很多展示型热点 key，常见做法是：
- 不追求每次更新都让缓存立刻绝对失效
- 允许短时间内返回旧值
- 通过较短 TTL 或后台刷新逐步更新

这背后的思想是：

“宁可短时间显示稍旧的数据，也不要让热点 key 大规模击穿数据库。”

这在工程上非常常见。

#### 适合哪些数据
- 展示型信息
- 排行榜摘要
- 首页聚合结果
- 热门内容列表

### 4. 常见权衡策略 2：关键热点 key 优先正确性，再配套限流和保护

对于强业务语义的热点 key，
往往不能简单接受旧值继续对外服务。

比如：
- 秒杀库存
- 是否还能下单
- 优惠资格是否生效

这类数据更看重正确性，
那就往往要配套：
- 更严格的缓存失效策略
- 互斥重建
- 限流
- 降级
- 本地锁 / 分布式锁 / 单飞机制

核心思路是：

“不能因为怕打库，就让错误结果持续对外暴露。”

也就是说：
不是简单牺牲一致性，
而是通过额外保护手段来同时守住性能底线。

### 5. 常见权衡策略 3：逻辑过期 / 异步刷新

这是热点 key 非常典型的一种折中方案。

它的思路不是：
“缓存到了时间就立刻物理删除。”

而是：
- 数据虽然逻辑上过期了
- 但旧值可以先继续提供
- 后台异步刷新新值
- 只有少数线程负责重建

这样做的好处是：
- 避免热点 key 物理删除后瞬时大量回源
- 让系统更平滑

代价是：
- 短时间内用户可能读到旧值

所以这本质上仍然是：
用可控的短暂旧值，换更高的稳定性。

### 6. 常见权衡策略 4：分层缓存与热点隔离

有些系统还会通过：
- 本地缓存 + Redis
- 热点 key 单独治理
- 热点数据预热
- 将极热 key 拆分或隔离

来降低“每次缓存变化都直接冲击数据库”的风险。

这说明热点 key 的问题，很多时候已经不只是一个“删不删缓存”的小技巧问题，
而是系统整体流量治理问题。

## 八、用三个最小案例理解热点 key 的不同取舍

### 案例 1：首页 banner 文案

特点：
- 很热
- 读很多
- 允许秒级旧值

策略：
- 可以接受短暂旧值
- 采用 TTL + 异步刷新
- 不必为绝对实时一致付出太高成本

### 案例 2：商品详情页价格

特点：
- 也可能很热
- 对用户感知敏感
- 错误价格可能引发业务问题

策略：
- 更新数据库后尽快失效缓存
- 删除失败要有补偿
- 必要时缩小 TTL
- 对高峰读流量做好防击穿保护

### 案例 3：秒杀库存 key

特点：
- 极热
- 正确性要求高
- 不只是展示问题，而是业务正确性问题

策略：
- 不能单靠普通缓存一致性思路
- 可能要引入更严格的原子扣减、异步下单、限流、队列化、最终数据库校验
- Redis 在这里往往已经不只是缓存，而是高并发控制组件的一部分

这个案例能帮你建立一个认知：

热点 key 的一致性策略，必须结合业务语义，
不能一招走天下。

## 九、所以这三部分最终应该怎么串起来理解

你现在可以把这一节收束成一条完整主线：

### 1. 主路径
先更新数据库，再删除缓存。

### 2. 失败补偿
如果删除缓存失败：
- 先做同步重试
- 再做异步补偿
- 最后靠 TTL 做兜底

### 3. 补偿工具各自角色
- 重试：修瞬时失败
- MQ：把补偿事件化、异步化、可重放
- binlog：基于数据库真实变更事实做后续修正

### 4. 热点 key 的额外治理
对于热点 key，不能只看一致性，
还要同时看：
- 删除后会不会引发大规模回源
- 是否允许短时间旧值
- 是否需要逻辑过期、限流、互斥重建等保护

这才是更完整的工程理解。

## 十、面试里怎么回答这三个问题

### 1）删除缓存失败时，工程上一般怎么补偿？

你可以这样说：

“工程上一般不会把删除缓存只当成一次性的动作，而是会做多层补偿。常见主线是先更新数据库，再删除缓存；如果删除失败，先做同步重试，兜住瞬时异常；如果还是失败，再通过补偿任务、延迟队列或定时任务异步继续删除；另外缓存本身通常还有 TTL 做最终兜底，避免旧数据永久残留。所以真实系统里通常是主流程加多层补偿，而不是只删一次就结束。”

### 2）缓存一致性里，重试、消息队列、binlog 分别在扮演什么角色？

你可以这样说：

“重试主要是在修一次即时调用的瞬时失败，是第一层快速补救；消息队列主要是把缓存失效这件事事件化、异步化，让删除动作可以解耦、重放和持续补偿；binlog 则更像以数据库真实变更事实为准，再驱动后续缓存修正，避免把一致性修复完全依赖在业务代码的一次调用上。三者不是简单替代关系，而是可以作用在不同层次上。”

### 3）热点 key 在一致性和性能之间怎么权衡？

你可以这样说：

“热点 key 的关键不只是缓存和数据库一不一致，而是缓存一旦失效，可能会导致大量并发同时回源数据库，所以必须同时权衡数据新鲜度和系统稳定性。对于展示型热点数据，通常可以接受短暂旧值，配合 TTL、逻辑过期和异步刷新；但对于库存、价格、资格这类关键数据，会更强调正确性，同时配合限流、互斥重建、队列化等手段来守住性能底线。核心不是一刀切，而是根据业务语义做取舍。”

## 十一、把这节课压缩成 4 句最重要的话

1. 删除缓存失败不是小概率角落问题，而是一致性方案必须正视的失败路径。
2. 工程上通常会用“同步重试 + 异步补偿 + TTL 兜底”来形成多层修复。
3. 重试、MQ、binlog 不是同一种东西，它们分别对应即时修复、事件化补偿、基于数据库事实修正。
4. 热点 key 的治理必须同时考虑一致性和抗压，不能脱离业务语义谈绝对新鲜。

## 十二、给你三个最小问答练习

### 1. 为什么只靠“先更新数据库，再删除缓存”还不够？
参考回答：
因为这个方案只是主路径更稳妥，但仍然可能遇到删缓存失败、服务宕机、网络异常等问题，所以真实系统还需要重试、异步补偿和 TTL 等后手来缩小不一致窗口。

### 2. MQ 在缓存一致性里最大的价值是什么？
参考回答：
它能把缓存失效从同步代码动作变成一个可异步处理、可重试、可监控的事件，从而增强补偿能力和系统解耦性。

### 3. 为什么热点 key 不能简单照搬普通 key 的一致性策略？
参考回答：
因为热点 key 一旦失效，会带来大量并发回源，问题不只是读到旧值，而是可能同时把数据库打崩，所以必须把一致性、新鲜度和抗压能力一起权衡。

## 十三、这节课学完后，你脑子里应该留下什么画面

你可以把这节课最后收成这样一张图：

- 数据写入先落数据库
- 缓存失效是主路径
- 主路径失败后，不是一句“失败了就算了”
- 而是继续靠重试、异步补偿、TTL 把系统往一致方向拉
- 如果系统更复杂，还可以借助 MQ 和 binlog 建立更稳的修正链路
- 一旦 key 很热，还要多问一句：删掉它之后，系统扛不扛得住？
- 所以热点 key 的治理，永远是“一致性 + 性能 + 业务正确性”的综合权衡

如果你能把这张图建立起来，
